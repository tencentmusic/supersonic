# 发现文档 — 数据到达完整性监控

> 基于内部访谈（执行人视角）整理
> 日期：2026-03-17

## 一句话总结

**信贷系统多合作方独立部署，每日批量推数据。跑批失败有告警，但"跑批成功、数据不全"是盲区。每月 1-2 次事后才发现，每次排查 4 小时。**

---

## 数据链路全景

```
合作方 A（钉子户）──每日跑批──→          ┌──→ 运营平台（对账/开票）
合作方 B（钉子户）──每日跑批──→  数仓  ──┤
合作方 C ──────────每日跑批──→          └──→ 报表
合作方 D ──────────每日跑批──→
   ...

各合作方跑批时间不统一
数据量级：每合作方每天几千条

     ┌──────────── 现有监控覆盖 ──────────────┐
     │ ✅ 跑批失败 → 有告警                     │
     │ ❌ 跑批成功但数据不全 → 无感知（盲区）    │
     │ ❌ 跑批成功但金额异常 → 无感知（盲区）    │
     └──────────────────────────────────────────┘
```

---

## 核心发现

### 1. 问题模式

| 维度 | 实际情况 |
|------|---------|
| **问题类型** | 上游漏推 / 修数 / 报表执行异常 |
| **发生频率** | 每月 1-2 次 |
| **发现方式** | 无主动发现机制，全靠事后倒查 |
| **发现时机** | 报表数据异常时（已滞后） |
| **排查耗时** | ~4 小时 |
| **影响范围** | 内部发现，未流转到外部（算幸运） |
| **修复方式** | 手动补数据 |
| **钉子户** | 1-2 个合作方特别容易出问题 |

### 2. 监控盲区（核心机会）

```
现有告警：跑批失败 → ✅ 能发现
真正盲区：跑批成功但数据不全 → ❌ 完全无感知
```

这就是为什么"每次都是事后才知道"——系统认为跑批成功了，但实际上数据是残缺的。

### 3. 排查瓶颈

| 排查步骤 | 耗时 | 瓶颈 |
|---------|------|------|
| 发现报表异常 | 不定 | 无主动检测，靠人眼发现 |
| 判断是上游问题还是自身问题 | 1-2 小时 | 需要查数仓表，对比数据 |
| 定位到具体哪个合作方 | 1-2 小时 | 合作方多，逐个排查 |
| 联系合作方 / 手动补数据 | 后续 | 非平台能解决 |

## 假设验证结果

| # | 假设 | 结果 | 说明 |
|---|------|------|------|
| 1 | 上游漏推是对账差异的主要来源 | ⚠️ **部分验证** | 漏推是主因之一，但还有修数、报表执行异常 |
| 2 | 定位耗时 >2 小时 | ✅ **验证** | 实际约 4 小时 |
| 3 | 财务无法自查数据链路 | ⚠️ **部分验证** | 执行人可以查数仓表，但普通财务不行 |
| 4 | 自动检测有价值 | ✅ **验证** | 明确需要，且给出了具体的 3 个检测维度 |

## 用户需求（按优先级）

### 需要的 3 个检测维度

| 优先级 | 检测项 | 逻辑 | 覆盖问题 |
|--------|--------|------|---------|
| **P0** | 各合作方今日到达行数 vs 近 7 天均值 | 偏差超 20% 标红 | 漏推、部分推送 |
| **P0** | 未到达合作方清单 | 超过约定时间仍无数据 | 跑批未执行、延迟 |
| **P1** | 金额汇总异常 | 行数对但金额偏差大 | 错推、修数未通知 |

### 告警分发

| 项目 | 要求 |
|------|------|
| 接收人 | 数据开发 + 自己（业务负责人） |
| 渠道 | 飞书群 |
| 频率 | 每天早上（各合作方跑批完成后） |

### 验收标准

| 指标 | 目标值 |
|------|--------|
| **排查时间** | 从 4 小时 → **30 分钟以内** |
| **误报率上限** | 一旦误报频繁，用户会关掉告警 |

### 关掉告警的条件（必须避免）

**误报太多**——天天告但没有真问题。所以：
- 阈值不能太敏感（20% 偏差是合理起点，可按合作方调整）
- 首版建议**只监控钉子户合作方**，不全量铺开
- 新合作方需要积累 7 天基线数据后才启用告警

## 落地到 SuperSonic 的实现路径

### 语义模板设计

```
模板名：数据到达监控
数据源：数仓中各合作方的落地表

维度（Dimension）：
- partner_code    合作方编码
- partner_name    合作方名称
- data_date       数据日期
- push_time       实际到达时间

指标（Metric）：
- row_count            今日到达行数
- avg_row_count_7d     近7天日均行数
- deviation_pct        偏差百分比 = (row_count - avg_7d) / avg_7d
- total_amount         今日金额汇总
- avg_amount_7d        近7天日均金额
- amount_deviation_pct 金额偏差百分比
```

### 告警规则

```
规则 1（行数偏差）：
  条件：deviation_pct < -20%
  消息：「⚠️ {partner_name} {data_date} 到达行数 {row_count}，
        低于近7天均值 {avg_row_count_7d} 的 20%」

规则 2（未到达）：
  条件：row_count = 0 AND 当前时间 > 该合作方约定推送时间 + 2小时
  消息：「🔴 {partner_name} {data_date} 数据未到达，
        已超过约定时间 2 小时」

规则 3（金额异常）：
  条件：amount_deviation_pct < -15% OR amount_deviation_pct > 15%
  消息：「⚠️ {partner_name} {data_date} 金额汇总 {total_amount}，
        偏离近7天均值 {avg_amount_7d} 超过 15%」
```

### 飞书推送卡片（每日汇总）

```
📊 数据到达日报 — 2026-03-17

✅ 正常到达：12 个合作方
⚠️ 行数偏差：合作方 A（-35%）、合作方 B（-22%）
🔴 未到达：合作方 X

点击查看详情 →（跳转 Web 监控页）
```

### 实现步骤（预估 1 周）

| # | 任务 | 耗时 | 依赖 |
|---|------|------|------|
| 1 | 在数仓中建到达统计表（按合作方 × 日期聚合行数和金额） | 0.5d | DBA / ETL |
| 2 | 在 SuperSonic 中创建"数据到达监控"语义模板 | 0.5d | 步骤 1 |
| 3 | 配置调度任务：每天早上 9 点执行模板查询 | 0.5d | 步骤 2 |
| 4 | 配置告警规则（偏差 >20% / 未到达 / 金额异常） | 1d | 需扩展告警订阅能力 |
| 5 | 配置飞书推送：结果发到指定飞书群 | 0.5d | 已有推送通道 |
| 6 | 先只监控 2 个钉子户合作方，跑 1 周观察误报率 | 1 周 | 步骤 5 |
