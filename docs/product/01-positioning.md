# 智能运营数据中台 — 产品定位文档

> 基于 April Dunford《Obviously Awesome》5 步定位框架生成
> 生成日期：2026-03-17

## 定位总览

本平台面向 5 类内部用户，每类用户的竞争替代品、痛点和价值主张不同，需要差异化定位。

---

## 用户群 1：一线运营人员

### 竞争替代品（不用我们，他们怎么做）
- 找数据分析师提工单，等 1-3 天
- 自己写 SQL 查 ClickHouse / Hive（会的人少）
- 从业务系统后台逐页导出 Excel，手动拼接

### 独特属性
- 飞书对话框直接用自然语言提问
- 30 秒内返回结果卡片，无需等工单
- 模板化常用查询，一键复用

### 价值
- **取数时间从 30 分钟 → 30 秒**
- 不依赖分析师，自助完成 80% 日常取数

### 目标用户画像
- 每天需要看 3-5 个数据指标的一线运营
- 不会写 SQL，习惯用飞书沟通
- 最常问："昨天 XXX 渠道的 GMV 是多少"

### 市场类别
**飞书里的数据助手**

### 一句话定位
> 对一线运营来说，它是飞书里的数据助手——在对话框用大白话问数，30 秒出结果，不用写 SQL、不用提工单。

---

## 用户群 2：数据分析师

### 竞争替代品
- Jupyter Notebook + 手写 SQL
- Metabase / Superset 自助 BI
- 业务系统自带的报表模块

### 独特属性
- 语义模板：定义一次，多场景复用（Web / API / 飞书 / 调度）
- 统一执行编排：同一个模板可以手动跑、定时跑、API 调用
- 版本管理 + 部署快照：模板改了不影响线上调度

### 价值
- **报表制作效率提升 3 倍**：定义模板 → 自动调度 → 自动分发，不再每次手动跑
- 运营自助取数后，分析师工单量减少 60%

### 目标用户画像
- 负责 3-5 个业务域数据需求的分析师
- 熟悉 SQL，但厌倦重复性取数工作
- 最常做："每周一出上周各渠道转化报表"

### 市场类别
**语义化报表工作台**

### 一句话定位
> 对数据分析师来说，它是语义化报表工作台——定义一次模板，Web / API / 飞书 / 定时调度四个出口自动跑，告别重复取数。

## 用户群 3：财务人员

### 场景修正（2026-03-17）
内部已有运营平台供财务做对账和开票。核心痛点**不是没有对账工具**，而是**上游数据漏推导致运营平台计算结果不可信**，发现晚、排查慢、修复依赖开发。

### 竞争替代品（当前应对方式）
- 月末对账时才发现数据对不上，倒查上游
- 找数据开发/ETL 开发查日志定位漏推原因（等 2 小时 ~ 1 天）
- 已有运营平台对账功能，但无法感知上游数据是否完整

### 独特属性
- **数据到达完整性检测**：对比上游源表行数 / 金额与数仓目标表，自动发现缺口
- **飞书主动告警**：上游数据少了多少条，哪个时间段缺失，每天早上推到飞书
- **差异溯源**：对账结果异常时，自动标注"可能原因：XX 系统 3 月 15 日数据少推 300 条"

### 价值
- **数据问题发现时间从月末 → 当天**
- 排查瓶颈从"找开发查日志"→ 自助看监控
- 不改变现有对账流程，在上游加一层"数据健康哨兵"

### 目标用户画像
- 使用运营平台做对账和开票的财务专员
- 反复被"上游数据不对"坑过，但自己无法排查数据链路
- 最痛的时刻："月末发现对不上，开发说上游月初就漏推了，白算了半个月"

### 市场类别
**数据健康哨兵**

### 一句话定位
> 对财务人员来说，它是数据健康哨兵——上游数据一有缺失，飞书立刻告诉你哪里少了、少了多少，不用等到月末对账才发现。

### 验证状态
**待验证** — 需通过访谈确认：漏推是否是主要原因、发现到定位耗时、自动检测的接受度。详见 [02-discovery-interviews.md](02-discovery-interviews.md)。

## 用户群 4：风控人员

### 竞争替代品
- 风控系统后台逐条查看告警（信息分散）
- 手动拉黑名单 + 人工比对交易记录
- 找开发写临时查询脚本排查风险事件

### 独特属性
- 风控域语义模板：风险指标 / 黑名单 / 异常交易统一建模
- 阈值告警 + 飞书实时推送：指标超阈值自动通知
- NL2SQL 临时排查：自然语言问"最近 7 天交易金额 >10 万的用户有哪些"

### 价值
- **风险发现时间从小时级 → 分钟级**
- 日常监控自动化，人力聚焦高价值判断

### 目标用户画像
- 日常盯风控指标、处理告警的风控专员
- 需要快速排查异常交易的场景
- 最痛："告警太多，真正有风险的淹没在噪音里"

### 市场类别
**智能风控监控台**

### 一句话定位
> 对风控人员来说，它是智能风控监控台——关键指标自动盯、异常自动报警推飞书、排查用自然语言直接问数据。

### 验证状态
**待验证** — 风控域语义模板尚未建模，需先完成领域发现。

## 用户群 5：管理层

### 竞争替代品
- 让下属每周做 PPT 汇报（信息滞后 3-7 天）
- 登录多个系统分别看数据（BI / ERP / CRM）
- 在飞书群里 @分析师 问数据（等半天）

### 独特属性
- 经营看板：核心指标一页总览，每日自动更新
- 飞书定时推送：每天早上 9 点推昨日经营数据到飞书
- 下钻能力：看到异常指标可以直接追问明细

### 价值
- **决策信息从滞后 3 天 → 实时可得**
- 不再依赖下属汇报，自己随时看、随时问

### 目标用户画像
- 关注 5-10 个核心经营指标的部门负责人 / VP
- 不写 SQL，但需要快速了解业务状况
- 最常问："昨天整体 GMV 怎么样？哪个渠道掉了？"

### 市场类别
**经营数据驾驶舱**

### 一句话定位
> 对管理层来说，它是经营数据驾驶舱——核心指标每天推到飞书，看到异常直接追问，不用等汇报、不用登多个系统。

## 平台统一定位

综合 5 类用户的共性需求：

> **智能运营数据中台**是一站式数据自助平台，让运营问数不求人、分析师告别重复工作、财务对账不加班、风控监控不漏报、管理层决策不等人。

### 定时报表与飞书触达（能力口径）

文中各用户群出现的「推到飞书」「飞书定时推送」等表述，指通过**租户内已配置的推送渠道**触达（例如飞书群自定义机器人 Webhook、邮件等），**并非**默认向当前操作者本人飞书私聊。具体语义与验收口径见 [运营数据工作台产品设计说明书 §7.1](./运营数据工作台产品设计说明书.md#71-结果投递语义当前阶段)。

### 核心差异化（vs 竞争替代品汇总）

| vs 什么 | 我们的优势 |
|---------|-----------|
| vs 提工单找分析师 | 自助即时，30 秒出结果 |
| vs 手写 SQL | 自然语言提问，零门槛 |
| vs Excel 手动拼接 | 模板化 + 自动调度 + 自动分发 |
| vs 分散的业务系统后台 | 一个入口统一查询所有数据源 |
| vs 传统 BI 工具（Metabase/Superset） | 飞书原生集成 + NL2SQL + 语义模板 |
