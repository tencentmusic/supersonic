# 战略优先级排序 — LNO + Pre-mortem

> 基于 Shreyas Doshi LNO Framework
> 前置依赖：完成用户访谈 + Event Storming 后执行

## LNO 分类法

将所有待办项分为三类，**只在 L（高杠杆）上投入最大精力**：

- **L（Leverage）**：做好了 10x 回报，做砸了严重后果
- **N（Neutral）**：做了有帮助，但不做也不致命
- **O（Overhead）**：必须做但不产生差异化价值

## MVP-3 待办项 LNO 分类

| 待办项 | L/N/O | 理由 | 建议 |
|--------|-------|------|------|
| **对账域语义模板** | **L** | 财务人员月末痛点明确（访谈验证后），ROI 清晰 | 优先做 |
| **自动差异检测** | **L** | 对账的核心价值，没有这个"自动对账"就是空话 | 与模板一起做 |

| **风控操作工具** | **O** | 写操作治理复杂度极高，需幂等/确认/审计/补偿 | MVP-3 尾声或 Phase 4 |

| **动作型工具治理** | **L** | 一旦开放写操作就必须有，但可以和具体工具同步建设 | 按需跟进 |

### MVP-3 精简路线

聚焦对账域，风控延后：
1. ✅ 对账域语义模板建模（基于 Event Storming 产出）
2. ✅ 自动差异检测逻辑
3. ✅ 对账报告自动生成 + 飞书推送

## Pre-mortem：假设项目 6 个月后失败

**引导问题**："假设今天是 2026 年 9 月，这个项目被认为失败了。最可能的原因是什么？"

| # | 失败场景 | 概率 | 预防措施 |
|---|---------|------|---------|
| 1 | "做了 MCP/Dify 集成，花了 2 个月，但没人用" | 高 | 砍掉 MVP-2 中 MCP/Dify，等有真实需求再做 |
| 2 | "对账模板建错了，财务说流程不是这样的" | 中 | Event Storming 必须有财务人员参加，不能闭门造车 |
| 3 | "做了很多功能但没人知道，平台没推起来" | 高 | 第 1 周就做定位 + 内部推介（已完成） |
| 4 | "告警太多，管理层关掉了飞书通知" | 中 | 阈值默认保守，先手动订阅再自动推送 |
| 5 | "风控写操作出了事故（误拉黑/误拦截）" | 低但致命 | MVP-3 禁止自动执行写操作，强制人工确认 |

### Pre-mortem 行动项

- [ ] MCP/Dify 从 MVP-2 移出，降为"按需引入"
- [ ] Event Storming 必须邀请实际操作人员（非主管代劳）
- [ ] 每 2 周做一次内部演示（运营/财务/风控各一场）
- [ ] 告警订阅上线时默认关闭，用户手动开启
