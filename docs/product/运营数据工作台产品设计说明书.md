## 2. 现状问题

当前系统从工程能力上已经覆盖查询、模板报表、调度、导出、飞书触达等多个方向，但从产品设计角度仍存在以下问题：

### 2.1 主叙事过大，主场景不够聚焦

当前总叙事同时覆盖：

- 数仓查询
- 模板报表
- 飞书协作
- 对账
- 风控

这会导致产品边界看起来很大，但当前真正接近可交付的核心能力主要仍是：

- 临时查询
- 固定报表
- 定时报表分发
- 飞书结果触达

### 2.2 用户心智仍然被工程对象主导

当前系统里的主要对象仍偏实现导向：

- Agent
- Plugin
- Template
- DataSet
- ReportSchedule

但业务用户真正理解和消费的是任务对象，而不是实现对象。

### 2.3 查询型能力与动作型能力混杂

当前很多能力仍然依附在查询语义下扩张，例如：

- 从查询结果创建定时报表
- 立即执行任务
- 后续扩展告警订阅

如果不在产品层明确区分查询和动作，后续能力会越来越难理解和治理。

### 2.4 入口职责边界不够稳定

Web、飞书、API 都在承载不同程度的查询、配置和任务能力，飞书尤其容易从“轻入口”不断膨胀成“半配置入口”。

### 2.5 任务管理缺少统一业务视图

执行记录、调度、导出、推送在能力上已存在，但从用户视角还缺少一个统一的“任务中心”来管理这些结果分发行为。

## 3. 改善目标

当前阶段的产品改善目标是：

**把现有工程能力收敛为一个面向运营团队的运营数据工作台，让用户在 Web 和飞书中完成高频查数、固定报表订阅、结果分发和异常提醒。**

该目标强调两点：

- 面向当前主用户收敛，而不是继续扩大产品叙事
- 先把已有能力整理成稳定闭环，再扩展更复杂场景

## 4. 当前阶段主用户

### 4.1 主验收角色

当前阶段唯一主视角是**运营数据负责人**。

他关心的是：

- 运营团队是否能稳定拿到高频经营数据
- 固定报表是否能替代人工报表生产
- 飞书是否能成为轻量结果入口
- 关键结果和提醒是否能稳定送达

### 4.2 服务对象

- 运营专员：查数、导出、接收固定报表
- 运营主管：订阅定时报表、查看任务状态、接收异常提醒
- 数据分析师：维护固定报表、参数和口径

### 4.3 后续视角

以下视角进入后续规划，不纳入当前主范围：

- 清结算对账负责人
- 风险监控负责人

## 5. 一级产品对象

当前阶段统一定义以下一级产品对象：

### 5.1 临时查询

用于即时获取结果，是所有高频分析和后续任务创建的起点。

### 5.2 固定报表

用于承载稳定口径和可复用的业务报表定义，是运营团队长期消费的核心内容对象。

### 5.3 定时报表任务

用于围绕固定报表或可复用查询结果建立周期性分发任务，是结果分发对象。

### 5.4 异常提醒任务

用于围绕核心指标建立规则触发的提醒，是监控对象。

### 5.5 任务中心

用于统一查看和管理定时报表任务、异常提醒任务及其执行状态，是统一管理对象。

## 6. 产品结构设计

### 6.1 查询与动作分层

#### 查询型能力

- 临时查询
- 查看固定报表
- 查看明细
- 查看历史结果

#### 动作型能力

- 创建定时报表任务
- 暂停/恢复/删除任务
- 立即执行任务
- 创建异常提醒任务

动作型能力必须统一满足以下要求：

- 显式确认
- 权限校验
- 留痕
- 状态可回查
- 失败原因可见

### 6.2 当前操作对象显式化

任何从查询结果转成动作的流程，必须显式展示当前操作对象。

例如创建定时报表任务前，界面或卡片中必须明确展示：

- 当前报表名称
- 来源
- 频率
- 接收对象（具体含义见 [§7.1](#71-结果投递语义当前阶段)）
- 是否立即执行一次

不能仅依赖隐式对话上下文。

### 6.3 任务中心一级化

任务中心应作为一级产品入口，而不是散落在调度、导出、推送的各自实现里。

至少应统一展示：

- 定时报表任务列表
- 异常提醒任务列表
- 最近执行记录
- 失败原因
- 立即执行
- 暂停/恢复
- 下载结果

## 7. 入口职责

| 入口 | 当前定位 | 负责内容 | 不负责内容 |
|------|---------|---------|-----------|
| Web | 配置与治理入口 | 固定报表维护、参数配置、任务管理、执行历史查看 | 轻量即时协作 |
| 飞书 | 消费与轻交互入口 | 临时查询、查看结果、轻量导出、确认创建任务、接收结果 | 复杂报表配置、复杂任务编辑 |
| API/MCP | 外部接入入口 | 对外暴露稳定能力接口 | 承载主用户产品心智 |

这三类入口的边界必须长期稳定，避免飞书入口继续承担过重配置职责。

### 7.1 结果投递语义（当前阶段）

定时报表等结果的**投递目标**与**对话入口**（Web 聊天、飞书、API）相互独立，须在产品和验收中写清，避免用户将「在 Web 里说发给我」理解为「飞书私聊发给我本人」。

1. **投递与入口解耦**：无论从 Web 还是飞书发起「创建定时报表」，实际投递以**租户内已配置且启用的推送配置**为准，不因「当前在哪个端操作」自动切换为飞书私聊或自动绑定当前操作者的飞书账号。
2. **「接收对象」的含义（现状）**：创建任务时需展示的「接收对象」指**将使用的推送渠道配置**（名称、类型、摘要），例如某条飞书群机器人 Webhook、某一组收件邮箱；**不是**从当前登录用户身份解析出的「我的飞书」或默认私聊。
3. **飞书（当前实现）**：通过自定义机器人 **Webhook** 发送通知，典型落点为**群**；报表文件通常以**下载链接**形式呈现（需在推送配置中配置可访问的下载基址），与邮件通道「附件直发」不同。
4. **更接近「只发给我」的通道（现状）**：**邮件**推送可将生成文件作为附件送达配置的收件人；若产品强调个人收件，应优先在说明与验收中体现邮件场景，并明确飞书群的共享可见性。
5. **自然语言话术口径**：对话中的「发给我」「推送给我」在现状下应理解为「按租户默认或选定的推送通道投递」。助手或界面文案宜可验证，例如提示「将发往已配置的飞书群 / 邮箱」，避免承诺未实现的「飞书私聊直达」。

### 7.2 接收对象选择规则（当前阶段）

为避免同一租户下存在多个推送配置时产生歧义，当前阶段统一采用以下规则：

1. **无可用配置不可创建**：若租户下不存在已启用的推送配置，系统应阻止创建定时报表任务，并明确提示「当前租户未配置可用推送渠道，请先在 Web 管理端完成推送配置」。
2. **仅一个可用配置时可默认带出**：若仅有一个已启用配置，可默认选中，但仍需在确认区显式展示配置名称、渠道类型和摘要。
3. **存在多个可用配置时必须显式选择**：不得静默选中某个渠道，也不得根据当前入口、当前用户或最近一次使用历史自动推断为「发给我」。
4. **创建后可在任务中心调整**：接收对象属于任务配置的一部分，后续编辑需保留审计信息，并在任务中心可回查变更记录。
5. **配置失效时应阻断执行或投递**：若创建后渠道被禁用、删除或校验失败，任务中心必须展示失败原因，不得以其他渠道静默兜底。

### 7.3 对话入口的渠道确认策略（当前阶段）

在 AI 助手、飞书机器人等对话入口中，渠道选择不应继续依赖“租户默认第一条配置”这类隐式规则，而应采用**轻确认**模式：

1. **自然语言只解析投递意图，不直接猜具体配置**：例如用户说「群和个人邮箱都要」，系统可识别为“飞书 + 邮件”的多渠道意图，但不能直接猜测是哪个飞书群、哪个邮箱组。
2. **确认卡片中显式展示具体配置实例**：对话确认页应展示本次将使用的渠道配置名称、类型和目标摘要，例如「华北运营日报群」「ops-report@example.com」。
3. **对话入口支持多选已有配置，但不承担配置治理**：聊天里可以勾选/取消已存在的推送配置；新增群机器人、维护邮箱组、测试渠道可用性等仍由 Web 管理端负责。
4. **不把 Web 重配置表单搬入聊天**：对话入口只处理“选择已有配置 + 确认创建”，不在聊天里承载复杂渠道创建、编辑和治理逻辑。
5. **最终确认结果写回任务对象**：一旦用户确认，所选渠道列表即成为任务配置的一部分，后续在任务中心和编辑流程中可完整还原。

### 7.4 默认投递策略与覆盖规则（当前阶段）

为兼顾 AI 助手的便捷性与可控性，当前阶段建议支持**默认投递策略 + 本次可覆盖**：

1. **允许配置默认投递组合**：默认值可挂在租户级或报表级，例如「运营日报默认发送到飞书群 A + 邮箱组 B」。
2. **聊天创建时默认带出策略，不等于强制锁死**：对话入口可先带出默认组合，但用户在确认页可取消或补充已有渠道。
3. **区分“渠道类型”与“具体渠道配置”**：产品表达上可说“飞书 + 邮件”，但落到执行时必须具体到配置实例，而不是只停留在类型层。
4. **禁止自动全选所有启用渠道**：默认策略必须是显式定义过的组合，而不是“把租户下所有启用渠道都发一遍”。
5. **本次确认优先于默认策略**：若用户在确认页调整了渠道组合，应以本次确认结果写入任务；任务后续执行不再回退到旧默认值。

### 7.5 渠道能力矩阵（当前阶段）

| 渠道 | 典型落点 | 内容形态 | 是否适合“只发给我” | 关键前置条件 |
|------|---------|---------|-------------------|-------------|
| 飞书 Webhook | 飞书群 | 卡片/文本 + 下载链接 | 否，默认是群共享可见 | 机器人 Webhook 可用、群可达、下载基址可访问 |
| 邮件 | 收件人/收件组邮箱 | 邮件正文 + 附件 | 是，更接近个人收件 | SMTP 或邮件服务可用、收件人有效、附件大小在可控范围内 |

### 7.6 推荐文案口径（当前阶段）

为减少自然语言误解，Web 与飞书确认页建议统一采用可验证表述：

- 「将发送到已配置的飞书群：`华北运营日报群`」
- 「将发送到已配置的邮箱组：`ops-report@example.com`」
- 「默认发送到：`华北运营日报群` + `ops-report@example.com`，本次可调整」
- 「当前为群通知链路，非飞书私聊直达」
- 「当前租户无可用推送渠道，暂不能创建定时报表任务」

### 7.7 权限与治理口径（当前阶段）

定时报表任务属于动作型能力，除数据权限外还需额外满足以下治理要求：

1. **双层权限校验**：创建、编辑、立即执行、恢复任务时，同时校验报表执行权限与外部分发权限，而不是仅校验是否能查到数据。
2. **owner 责任明确**：定时报表以任务 owner 的权限作为执行基线；接收对象不是权限来源，不能通过“发到别人邮箱/群里”放大数据可见范围。
3. **高风险动作显式确认**：涉及外部群、多人邮箱组或批量修改接收对象时，需二次确认并留痕。
4. **审计字段完整**：至少记录操作人、入口、报表对象、选定渠道、立即执行选项、执行结果和失败原因。

## 8. 核心流程

### 8.1 临时查询

用户在 Web 或飞书发起查询，系统返回结果卡片或表格，并支持导出或保存为任务。

### 8.2 固定报表消费

用户浏览固定报表列表，查看说明、参数和最近执行结果，并可从报表进入任务创建流程。

### 8.3 定时报表任务创建

用户可基于固定报表或当前可复用查询结果创建定时报表任务。

创建流程必须包含：

1. 选择或确认当前报表对象
2. 配置频率和接收对象
3. 若存在多个可用推送配置，显式选择具体渠道
4. 显式确认
5. 选择是否立即执行一次
6. 写入任务中心并可查看执行状态

创建流程中的校验要求：

- 无可用推送配置时不允许完成创建
- 同时校验报表执行权限和外部分发权限
- 确认区必须展示渠道名称、类型和目标摘要
- 不得把「发给我」解释为飞书私聊本人
- 若已配置默认投递组合，可先带出默认值，但用户需可见、可改
- 在聊天入口中支持对已有渠道做轻量多选，不在聊天里承载新增/治理渠道配置

### 8.4 异常提醒任务

用户为核心指标配置规则、频率、静默期和接收对象，并在任务中心查看提醒历史和状态。

### 8.5 多渠道执行与结果追踪

当一个任务绑定多个投递渠道时，产品对象上仍应视为**一个任务，多个渠道结果**，并满足：

1. 任务中心可按渠道查看执行结果，例如飞书成功、邮件失败。
2. 部分渠道失败时需明确标记为“部分成功”或等价业务状态，而不是笼统展示为成功。
3. 失败原因应定位到具体渠道配置，便于后续重试和排障。
4. 后续若支持重试，应优先支持按失败渠道重试，而不是默认整任务全量重发。

## 9. 当前阶段验收标准

### 9.1 验收前置条件

以下前置条件需满足后，相关能力才可按“可联调 / 可试点”口径验收：

- Web 端已具备固定报表、任务中心和推送配置管理入口
- 至少存在一个已启用且经测试可用的推送配置
- 飞书链路已完成 Webhook 可用性验证；若走链接下载，还需验证下载基址外部可访问
- 邮件链路若纳入验收，需完成邮件服务配置并验证附件送达
- 飞书机器人相关能力如用于对外宣称“可用”，需完成端到端联调，而不只停留在代码已存在状态
- 若验收包含“群 + 个人邮箱同时收”，需至少准备一条飞书群配置和一条邮件配置，并验证多渠道执行结果可回查

- 运营专员可在飞书查询高频经营数据
- 固定报表可在 Web 中清晰浏览和执行
- 用户可从固定报表创建定时报表任务
- 用户可从当前查询结果创建定时报表任务
- 任务中心可查看最近执行情况和失败原因
- 通过已配置的飞书推送渠道可稳定送达定时报表结果（典型为群机器人 Webhook 通知；非飞书私聊）
- 通过已配置的邮件推送渠道可将定时报表以附件等形式送达指定收件人（若启用邮件通道）
- 定时报表任务与异常提醒任务在产品对象上明确区分

### 9.2 关键验收用例

- 租户仅有一个启用渠道时，创建任务页面默认带出该渠道，并在确认区明确展示其名称和类型
- 租户存在多个启用渠道时，系统要求用户显式选择其一；未选择时不能提交
- 租户无可用渠道时，系统阻止创建并给出清晰提示
- 用户仅具备报表查看权限、不具备外部分发权限时，不能创建或修改带外部分发的任务
- 已配置默认投递组合时，对话确认页默认带出该组合，且用户可取消或补充已有渠道
- 用户表达「群和邮箱都要」时，系统进入多渠道确认，而不是静默使用单一默认渠道
- 飞书渠道验收时，飞书群实际收到消息，内容包含报表标识、执行时间和下载链接；不得表述为飞书私聊直发
- 邮件渠道验收时，指定收件人实际收到邮件，附件可正常打开
- 同一任务同时绑定飞书和邮件时，任务中心可分别看到两条渠道结果及对应失败原因
- 渠道失效或被禁用后，任务中心展示失败原因，且系统不自动切换到其他渠道

## 10. 状态口径

为避免工程状态与业务可交付状态混淆，当前阶段统一采用以下口径：

- **已实现**：代码已存在
- **可联调**：端到端链路可在测试环境打通
- **可试点**：可给小范围业务试用，具备基础治理
- **可推广**：具备稳定性、可观测性和运维条件，可面向更大范围推广
- **已有设计**：已有明确方案、类设计或原型，尚未完成可运行交付
- **待开发**：当前工程中不存在对应实现
