## 2. 现状问题

当前系统从工程能力上已经覆盖查询、模板报表、调度、导出、飞书触达等多个方向，但从产品设计角度仍存在以下问题：

### 2.1 主叙事过大，主场景不够聚焦

当前总叙事同时覆盖：

- 数仓查询
- 模板报表
- 飞书协作
- 对账
- 风控

这会导致产品边界看起来很大，但当前真正接近可交付的核心能力主要仍是：

- 临时查询
- 固定报表
- 定时报表分发
- 飞书结果触达

### 2.2 用户心智仍然被工程对象主导

当前系统里的主要对象仍偏实现导向：

- Agent
- Plugin
- Template
- DataSet
- ReportSchedule

但业务用户真正理解和消费的是任务对象，而不是实现对象。

### 2.3 查询型能力与动作型能力混杂

当前很多能力仍然依附在查询语义下扩张，例如：

- 从查询结果创建定时报表
- 立即执行任务
- 后续扩展告警订阅

如果不在产品层明确区分查询和动作，后续能力会越来越难理解和治理。

### 2.4 入口职责边界不够稳定

Web、飞书、API 都在承载不同程度的查询、配置和任务能力，飞书尤其容易从“轻入口”不断膨胀成“半配置入口”。

### 2.5 任务管理缺少统一业务视图

执行记录、调度、导出、推送在能力上已存在，但从用户视角还缺少一个统一的“任务中心”来管理这些结果分发行为。

## 3. 改善目标

当前阶段的产品改善目标是：

**把现有工程能力收敛为一个面向运营团队的运营数据工作台，让用户在 Web 和飞书中完成高频查数、固定报表订阅、结果分发和异常提醒。**

该目标强调两点：

- 面向当前主用户收敛，而不是继续扩大产品叙事
- 先把已有能力整理成稳定闭环，再扩展更复杂场景

## 4. 当前阶段主用户

### 4.1 主验收角色

当前阶段唯一主视角是**运营数据负责人**。

他关心的是：

- 运营团队是否能稳定拿到高频经营数据
- 固定报表是否能替代人工报表生产
- 飞书是否能成为轻量结果入口
- 关键结果和提醒是否能稳定送达

### 4.2 服务对象

- 运营专员：查数、导出、接收固定报表
- 运营主管：订阅定时报表、查看任务状态、接收异常提醒
- 数据分析师：维护固定报表、参数和口径

### 4.3 后续视角

以下视角进入后续规划，不纳入当前主范围：

- 清结算对账负责人
- 风险监控负责人

## 5. 一级产品对象

当前阶段统一定义以下一级产品对象：

### 5.1 临时查询

用于即时获取结果，是所有高频分析和后续任务创建的起点。

### 5.2 固定报表

用于承载稳定口径和可复用的业务报表定义，是运营团队长期消费的核心内容对象。

### 5.3 定时报表任务

用于围绕固定报表或可复用查询结果建立周期性分发任务，是结果分发对象。

### 5.4 异常提醒任务

用于围绕核心指标建立规则触发的提醒，是监控对象。

### 5.5 任务中心

用于统一查看和管理定时报表任务、异常提醒任务及其执行状态，是统一管理对象。

## 6. 产品结构设计

### 6.1 查询与动作分层

#### 查询型能力

- 临时查询
- 查看固定报表
- 查看明细
- 查看历史结果

#### 动作型能力

- 创建定时报表任务
- 暂停/恢复/删除任务
- 立即执行任务
- 创建异常提醒任务

动作型能力必须统一满足以下要求：

- 显式确认
- 权限校验
- 留痕
- 状态可回查
- 失败原因可见

### 6.2 当前操作对象显式化

任何从查询结果转成动作的流程，必须显式展示当前操作对象。

例如创建定时报表任务前，界面或卡片中必须明确展示：

- 当前报表名称
- 来源
- 频率
- 接收对象
- 是否立即执行一次

不能仅依赖隐式对话上下文。

### 6.3 任务中心一级化

任务中心应作为一级产品入口，而不是散落在调度、导出、推送的各自实现里。

至少应统一展示：

- 定时报表任务列表
- 异常提醒任务列表
- 最近执行记录
- 失败原因
- 立即执行
- 暂停/恢复
- 下载结果

## 7. 入口职责

| 入口 | 当前定位 | 负责内容 | 不负责内容 |
|------|---------|---------|-----------|
| Web | 配置与治理入口 | 固定报表维护、参数配置、任务管理、执行历史查看 | 轻量即时协作 |
| 飞书 | 消费与轻交互入口 | 临时查询、查看结果、轻量导出、确认创建任务、接收结果 | 复杂报表配置、复杂任务编辑 |
| API/MCP | 外部接入入口 | 对外暴露稳定能力接口 | 承载主用户产品心智 |

这三类入口的边界必须长期稳定，避免飞书入口继续承担过重配置职责。

## 8. 核心流程

### 8.1 临时查询

用户在 Web 或飞书发起查询，系统返回结果卡片或表格，并支持导出或保存为任务。

### 8.2 固定报表消费

用户浏览固定报表列表，查看说明、参数和最近执行结果，并可从报表进入任务创建流程。

### 8.3 定时报表任务创建

用户可基于固定报表或当前可复用查询结果创建定时报表任务。

创建流程必须包含：

1. 选择或确认当前报表对象
2. 配置频率和接收对象
3. 显式确认
4. 选择是否立即执行一次
5. 写入任务中心并可查看执行状态

### 8.4 异常提醒任务

用户为核心指标配置规则、频率、静默期和接收对象，并在任务中心查看提醒历史和状态。

## 9. 当前阶段验收标准

- 运营专员可在飞书查询高频经营数据
- 固定报表可在 Web 中清晰浏览和执行
- 用户可从固定报表创建定时报表任务
- 用户可从当前查询结果创建定时报表任务
- 任务中心可查看最近执行情况和失败原因
- 飞书可稳定接收定时报表结果
- 定时报表任务与异常提醒任务在产品对象上明确区分

## 10. 状态口径

为避免工程状态与业务可交付状态混淆，当前阶段统一采用以下口径：

- **已实现**：代码已存在
- **可联调**：端到端链路可在测试环境打通
- **可试点**：可给小范围业务试用，具备基础治理
- **可推广**：具备稳定性、可观测性和运维条件，可面向更大范围推广
- **已有设计**：已有明确方案、类设计或原型，尚未完成可运行交付
- **待开发**：当前工程中不存在对应实现
