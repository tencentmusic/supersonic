## 2. 现状问题

当前工程能力已覆盖查询、模板报表、调度、导出、飞书触达，但产品层仍有 5 个核心问题：

- 主叙事过大：查询、模板、飞书、对账、风控被并列讨论，导致当前最接近交付的主链路不够突出
- 用户心智偏工程对象：业务用户真正理解的是经营主题、报表、异常、任务和责任，而不是 Agent、Plugin、Template、DataSet、ReportSchedule
- 查询与动作混杂：从查数直接滑向创建任务、立即执行、后续订阅，缺少明确分层
- 入口边界不稳：Web、飞书、API 容易互相侵蚀职责，飞书尤其容易膨胀为半配置入口
- 缺少统一业务视图：调度、导出、推送、执行记录已存在，但仍未被组织成一个可治理的产品闭环

## 3. 改善目标

当前阶段的产品改善目标是：

**把现有工程能力升级为一个面向运营团队的运营 Operating System 一期，让团队围绕每日经营驾驶舱、经营主题、固定报表、异常处置流和责任账本完成日常消费、分发、确认和追责。**

该目标只强调两点：

- 面向当前主用户的真实重复劳动与责任链问题，而不是继续扩大分析平台叙事
- 先把“结果稳定产出、稳定送达、异常稳定接手”的操作闭环做成，再扩展更复杂场景

### 3.1 当前阶段的战略升级

当前文档不再把“运营数据工作台”理解为功能收纳页，而是理解为**运营团队的轻量操作系统一期**：首页是驾驶舱、一级认知是经营主题、异常是处置流、任务中心是治理入口、飞书是轻消费工作面。

### 3.2 一期的真正价值

一期不是为了证明“功能很多”，而是为了证明以下事情可以稳定发生：

- 同一类高频经营结果不再依赖技术做人肉中转
- 团队开始围绕少量经营主题形成稳定消费习惯
- 异常会被提醒、接手、确认并留痕
- 结果、分发、异常和责任第一次进入同一产品对象体系

## 4. 当前阶段主用户

### 4.1 主验收角色

当前阶段唯一主视角是**运营数据负责人**。

他关心的是：

- 运营团队是否能稳定拿到高频经营数据
- 固定报表是否能替代人工报表生产
- 飞书是否能成为轻量结果入口
- 关键结果和提醒是否能稳定送达

### 4.2 服务对象

- 运营专员：查数、导出、接收固定报表
- 运营主管：订阅定时报表、查看任务状态、接收异常提醒
- 数据分析师：维护固定报表、参数和口径

### 4.3 后续视角

以下视角进入后续规划，不纳入当前主范围：

- 清结算对账负责人
- 风险监控负责人

## 5. 一级产品对象

当前阶段统一定义以下一级产品对象。它们不再是平铺的工具模块，而是围绕“驾驶舱 -> 经营主题 -> 消费/处置/治理对象”的层次组织。

### 5.1 每日经营驾驶舱

用于承载团队每天默认打开的经营总览，是当前阶段最重要的首页对象。至少聚合：

- 今日必看经营主题
- 关键固定报表的最新状态
- 待确认或待接手的异常
- 分发失败或结果过期等可靠性风险
- 团队当日需要处理的事项摘要

它不替代详情页，只回答三件事：今天先看什么、哪些结果可直接使用、哪些问题需要立即处理。

### 5.2 经营主题

用于把固定报表、异常处置流、定时报表任务和责任账本组织成业务用户可理解的对象层。每个主题至少挂载：

- 固定报表集合
- 异常处置流集合
- 相关定时报表任务
- 默认投递与责任信息
- 在驾驶舱中的优先级

### 5.3 固定报表

用于承载稳定口径和可复用的业务报表定义，是运营团队长期消费的核心内容对象。

### 5.3A 固定报表补充原则

当前“固定报表”不能继续等同于“成功部署记录展示”，而应升级为面向长期消费的固定报表工作台。主文档只锁定以下原则：

- 固定报表是消费对象，定时报表任务是动作对象
- 固定报表页回答“今天哪些结果可消费、是否可信、是否已进入稳定分发闭环”
- 固定报表页采用“筛选区 + 主列表区 + 详情区”的工作台结构，不采用均质卡片墙
- 主操作始终是“查看结果”，不是“创建任务”
- 结果过期不是错误；最近失败也不等于完全不可消费，若存在上一版结果应明确提示
- 固定报表页负责消费对象管理，任务中心负责动作对象治理

固定报表对象定义、状态原则、信息架构、响应式和边界细节，统一下沉到：

- [运营 Operating System 一期详细设计](../details/report/08-operations-operating-system.md)

### 5.4 定时报表任务

用于围绕固定报表或可复用查询结果建立周期性分发任务，是结果分发对象。

### 5.5 异常处置流

当前阶段不再把“异常提醒”只定义为提醒任务，而升级为**异常处置流对象**。

至少包含：

- 异常类型
- 影响主题与相关报表
- 当前状态：待确认 / 已确认 / 已接手 / 已恢复 / 已关闭
- 最近正常值与异常值
- 责任人 / 接手人 / 确认人
- 处理记录与时间线

它的目标不是“发出提醒”，而是确保异常被看到、被接住、被记录、可复盘。

### 5.6 可信度与责任账本

用于承载结果可靠性、分发可靠性、异常处置可靠性和责任链留痕，是当前阶段建立组织信任的核心治理对象。至少回答：

- 这份结果由谁定义、谁维护、谁消费
- 最近一次成功和失败发生在什么时候
- 失败属于生成失败、投递失败还是处置未完成
- 谁确认过异常、谁接手过问题
- 这套自动化相比人工链路替代了多少重复劳动

### 5.7 任务中心

用于统一查看和管理定时报表任务、异常处置流及其执行状态，是统一管理对象。

### 5.8 临时查询

用于即时获取结果，是所有高频分析和后续任务创建的起点。

在当前阶段，临时查询仍然重要，但它不再是首页主角，而是进入固定报表、任务创建和主题沉淀的辅助入口。

### 5.9 飞书轻消费工作面

飞书在当前阶段不承担复杂配置治理，但不再只是通知口，而是**轻消费工作面**。

飞书至少承载以下轻量场景：

- 查看今日必看结果
- 查看异常摘要并完成确认 / 接手
- 查看上一版可参考结果
- 对现有任务执行极轻量动作

## 6. 产品结构设计

### 6.1 查询与动作分层

当前阶段的主结构不再是“查询模块 + 任务模块”的并列拼装，而是以下骨架：

```text
每日经营驾驶舱
    |
    v
经营主题
    |
    +--> 固定报表
    +--> 异常处置流
    +--> 定时报表任务
    +--> 可信度与责任账本
    |
    +--> 飞书轻消费工作面
```

查询型能力：

- 驾驶舱中查看今日经营状态
- 按经营主题查看固定报表
- 查看固定报表
- 查看明细
- 查看历史结果
- 在飞书中轻量查看结果与异常摘要
- 临时查询

动作型能力：

- 创建定时报表任务
- 暂停/恢复/删除任务
- 立即执行任务
- 确认异常 / 接手异常 / 标记恢复
- 调整默认投递或本次投递覆盖
- 在飞书中完成轻量确认动作

动作型能力必须统一满足：

- 显式确认
- 权限校验
- 留痕
- 状态可回查
- 失败原因可见

### 6.2 当前操作对象显式化

任何从查询结果转成动作的流程，必须显式展示当前操作对象。创建定时报表任务前，界面或卡片中至少展示：

- 当前报表名称
- 来源
- 频率
- 接收对象（具体含义见 [§7.1](#71-结果投递语义当前阶段)）
- 是否立即执行一次

不能仅依赖隐式对话上下文。

### 6.3 任务中心一级化

任务中心应作为一级治理入口，而不是散落在调度、导出、推送和处置记录的各自实现里。主文档层只要求它统一承载任务、处置、执行、失败与结果下载的治理视图。

### 6.4 驾驶舱与主题的关系

驾驶舱是首页对象，经营主题是一级业务对象。二者关系如下：

- 驾驶舱负责当日优先级排序
- 经营主题负责承载长期消费和治理上下文
- 固定报表、异常处置流、任务和责任账本都挂在经营主题之下
- 临时查询可以进入这些对象，但不替代这些对象

### 6.5 当前阶段不做什么

为保证一期仍然可交付，以下内容明确不纳入当前主范围：

- 通用 BI 平台能力
- 通用监控平台能力
- 飞书复杂配置治理
- 自由度极高的可视化搭建器
- 对账、风控专项产品化
- 所有报表自动进入驾驶舱的全量覆盖

## 7. 入口职责

| 入口 | 当前定位 | 负责内容 | 不负责内容 |
|------|---------|---------|-----------|
| Web | 配置与治理入口 | 驾驶舱查看、经营主题维护、固定报表维护、任务管理、责任账本与执行历史查看 | 高噪音即时沟通 |
| 飞书 | 轻消费工作面 | 查看今日必看结果、查看异常摘要、确认/接手异常、轻量任务动作、接收结果 | 复杂报表配置、复杂任务编辑、复杂渠道治理 |
| API/MCP | 外部接入入口 | 对外暴露稳定能力接口 | 承载主用户产品心智 |

这三类入口的边界必须长期稳定，避免飞书工作面继续承担过重配置职责。

### 7.1 结果投递语义（当前阶段）

定时报表等结果的**投递目标**与**对话入口**（Web 聊天、飞书、API）相互独立。

主文档层只锁定四条原则：

- 接收对象指具体渠道配置，不指当前操作者本人
- 飞书当前实现以群 Webhook + 下载链接为主，不承诺私聊直达
- 邮件更接近“只发给我”的心智
- 所有自然语言文案都必须可验证

### 7.2 接收对象选择规则（当前阶段）

为避免歧义，统一要求：

- 无可用配置时阻断创建
- 单配置可默认带出，但必须显式展示
- 多配置必须显式选择，不得静默推断
- 配置失效必须暴露失败原因

### 7.3 对话入口的渠道确认策略（当前阶段）

在 AI 助手、飞书机器人等对话入口中，渠道选择采用**轻确认**模式：只解析投递意图、展示具体渠道实例、支持已有配置选择，不在聊天中承载复杂治理。

### 7.4 默认投递策略与覆盖规则（当前阶段）

为兼顾便捷性与可控性，当前阶段支持**默认投递策略 + 本次可覆盖**：默认值可先带出但不得锁死，执行时必须具体到渠道实例，本次确认优先于旧默认值。

### 7.5 推荐文案口径（当前阶段）

为减少自然语言误解，Web 与飞书确认页统一采用可验证口径：明确写出将发送到哪个已配置渠道、明确区分默认投递与本次覆盖、明确说明飞书当前为群通知链路而非私聊直达。

### 7.6 权限与治理口径（当前阶段）

定时报表任务属于动作型能力，除数据权限外还需额外满足以下治理要求：

- 双层权限校验：同时校验报表执行权限与外部分发权限
- owner 责任明确：接收对象不是权限来源，不能放大数据可见范围
- 高风险动作显式确认并留痕
- 审计字段至少覆盖操作人、入口、报表对象、渠道、执行结果和失败原因

## 8. 核心流程

### 8.0 每日经营驾驶舱打开流程

用户进入系统后，默认先到每日经营驾驶舱，而不是先进入模块导航。驾驶舱需要先回答：

1. 今天最该看哪些经营主题
2. 哪些结果已经更新且可直接使用
3. 哪些异常待确认、待接手或待恢复
4. 哪些任务或渠道存在可靠性风险

### 8.1 经营主题消费

用户从驾驶舱进入经营主题，在同一主题下完成结果消费、异常查看、任务跳转与责任追踪。一个经营主题下至少能看到：

1. 当前主题最重要的固定报表
2. 相关异常处置流
3. 相关定时报表任务
4. 默认投递、责任与可信度摘要

### 8.2 临时查询

用户在 Web 或飞书发起查询，系统返回结果卡片或表格，并支持导出或保存为任务。

### 8.3 固定报表消费

用户浏览固定报表列表，查看说明、参数和最近执行结果，并可从报表进入任务创建流程。

### 8.4 定时报表任务创建

用户可基于固定报表或当前可复用查询结果创建定时报表任务。主链路为：确认报表对象、配置频率与接收对象、显式确认渠道与是否立即执行、写入任务中心并追踪执行状态。

### 8.5 异常处置流

用户或系统发现异常后，不以“提醒已发出”作为流程终点，而以“异常已确认并进入处置状态”作为流程起点。主链路为：进入待确认、确认已知、接手处理、补充记录、标记恢复或关闭，并在责任账本中留痕。

### 8.6 飞书轻消费工作面

飞书侧当前阶段支持轻量但高频的消费动作：查看今日必看结果、查看异常摘要、确认异常已知或接手异常、查看上一版可参考结果、对现有任务执行轻量动作。

### 8.7 多渠道执行与结果追踪

当一个任务绑定多个投递渠道时，产品对象上仍应视为**一个任务，多个渠道结果**，并满足：

- 任务中心可按渠道查看执行结果
- 部分渠道失败时需明确暴露，而不是笼统展示为成功
- 失败原因应定位到具体渠道配置
- 后续若支持重试，应优先支持按失败渠道重试

## 9. 当前阶段验收标准

当前阶段的验收不再以“模块是否都存在”为主，而以“运营操作系统一期是否开始替代人工链路”为主。

### 9.1 验收前置条件

以下前置条件满足后，相关能力才可按“可联调 / 可试点”口径验收：

- Web 端已具备固定报表、任务中心和推送配置管理入口
- 至少存在一个已启用且经测试可用的推送配置
- 飞书链路已完成 Webhook 可用性验证；若走链接下载，还需验证下载基址外部可访问
- 邮件链路若纳入验收，需完成邮件服务配置并验证附件送达
- 若验收包含“群 + 个人邮箱同时收”，需至少准备一条飞书群配置和一条邮件配置，并验证多渠道执行结果可回查

### 9.2 核心验收结果

达到以下结果时，当前阶段方可按“可试点”口径验收：

- 运营负责人每天可先从驾驶舱判断今日经营重点，而不是先进入工具菜单
- 团队能围绕至少 2 到 3 个经营主题稳定消费固定报表
- 固定报表可替代至少 2 到 3 类重复人工查数/整理/分发链路
- 定时报表任务可稳定创建、执行、追踪失败原因
- 异常不只会被提醒，还能被确认、接手和留痕
- 飞书可承担结果消费和异常确认等轻量高频动作，而不只是通知
- 责任账本能回看结果、投递、异常和责任链的关键记录

### 9.3 关键验收用例

关键验收覆盖以下场景即可：

- 渠道选择规则正确：单渠道默认、多渠道显式选择、无渠道阻断创建
- 默认投递与本次覆盖规则正确：不静默推断、不自动全选
- 飞书与邮件多渠道执行结果可分渠道回查
- 驾驶舱能聚合今日必看主题、关键报表状态和待确认异常
- 飞书中的异常确认或接手可正确回写到 Web
- 异常关闭后，责任账本可回看确认人、接手人、处理记录和关闭时间

## 10. 状态口径

为避免工程状态与业务可交付状态混淆，当前阶段统一采用以下口径：

- **已实现**：代码已存在
- **可联调**：端到端链路可在测试环境打通
- **可试点**：可给小范围业务试用，具备基础治理
- **可推广**：具备稳定性、可观测性和运维条件，可面向更大范围推广
- **已有设计**：已有明确方案、类设计或原型，尚未完成可运行交付
- **待开发**：当前工程中不存在对应实现

## 11. 后续规划

后续规划主要分三类：

- 清结算对账：补对账作业对象、账期与口径冻结、审计回放、差异复核
- 风险监控：补告警对象、规则中心、值守群与更重的告警处置闭环
- 投递增强：若需对齐“发给我本人”心智，再专项规划飞书按用户维度投递与账号绑定
