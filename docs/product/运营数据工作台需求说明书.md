# 运营数据工作台需求说明书

## 1. 文档目标

本文定义当前阶段面向业务的主产品范围。主视角是**运营数据负责人**，目标是让运营团队在 Web 和飞书中完成高频查数、固定报表订阅、结果分发和异常提醒，减少人工取数与人肉催报表。

本文是以下设计文档的业务主入口：

- [智能运营数据中台设计方案](../智能运营数据中台设计方案.md)
- [模板报表子系统技术设计说明书](../模板报表子系统技术设计说明书.md)
- [飞书机器人用户交互层设计方案](../飞书机器人用户交互层设计方案.md)

## 2. 当前阶段主用户

### 2.1 运营数据负责人

这是当前阶段唯一主验收角色，负责：

- 让运营团队稳定拿到高频经营数据
- 让固定报表脱离人工制作
- 让飞书成为轻量结果入口
- 让关键结果和提醒稳定送达

### 2.2 服务对象

- 运营专员：查数、导出、接收固定报表
- 运营主管：订阅报表、查看任务状态、接收异常提醒
- 数据分析师：维护固定报表、参数和口径

### 2.3 暂不作为主验收对象

- 清结算对账负责人
- 风险监控负责人

这两个视角进入后续规划，不进入当前阶段主范围。

## 3. 一级产品对象

本阶段只定义以下一级产品对象：

- 临时查询
- 固定报表
- 定时报表任务
- 异常提醒任务
- 任务中心

以下属于实现概念，不作为业务主叙事：

- Agent
- Plugin
- DataSet
- QueryMode
- ParseInfo

## 4. 入口职责

| 入口 | 定位 | 负责内容 | 不负责内容 |
|------|------|---------|-----------|
| Web | 配置与治理入口 | 固定报表维护、参数配置、任务管理、执行历史查看 | 轻量即时协作 |
| 飞书 | 消费与轻交互入口 | 临时查询、查看结果、轻量导出、确认创建任务、接收结果 | 复杂报表配置、复杂任务编辑 |
| API/MCP | 外部接入入口 | 暴露稳定能力接口 | 承载主用户产品心智 |

### 4.1 定时报表投递与「接收对象」

定时报表的实际投递目标以**租户内已配置且启用的推送配置**为准，与对话入口（Web / 飞书）无关；创建任务时展示的「接收对象」指**所选推送渠道**（如飞书群机器人 Webhook、邮件收件人），**不是**从当前操作者自动解析的「飞书私聊本人」。飞书通道当前典型为群通知 + 链接；邮件通道可将报表作为附件送达。完整产品与话术口径见 [运营数据工作台产品设计说明书 §7.1](./运营数据工作台产品设计说明书.md#71-结果投递语义当前阶段)。

业务规则补充如下：

- 租户下无已启用推送配置时，不允许创建定时报表任务
- 仅有一个可用推送配置时，可默认带出，但必须显式展示名称、渠道类型和目标摘要
- 存在多个可用推送配置时，用户必须显式选择，系统不得按当前入口或当前用户自动推断
- 接收对象变更属于任务配置变更，需可审计、可回查
- 渠道失效、被禁用或测试失败时，任务执行失败原因需在任务中心可见，不得静默切换到其他渠道

对话入口补充规则如下：

- AI 助手或飞书聊天中，自然语言只解析“飞书 / 邮件 / 多渠道”等投递意图，不直接猜具体群或邮箱组
- 聊天确认环节应展示具体渠道配置实例，并支持对已有配置做轻量多选
- 聊天入口负责“选择已有渠道 + 确认创建”，不负责新增、编辑、测试推送配置
- 若已配置默认投递组合，可在聊天确认页默认带出，但用户必须可见、可改
- 禁止自动将租户下所有启用渠道全部选中并发送

### 4.2 权限约束

定时报表相关动作需同时满足以下权限要求：

- 具备报表执行权限，才可创建、编辑、立即执行该任务
- 具备外部分发权限，才可将结果发送到飞书群、邮件收件人等外部渠道
- 任务执行以 owner 权限为基线；接收对象不会扩大数据权限范围
- 高风险外部分发动作需显式确认并留痕

## 5. 核心流程

### 5.1 临时查询

用户在 Web 或飞书发起查询，系统返回结果卡片或表格，并支持导出或保存为任务。

### 5.2 固定报表消费

用户浏览固定报表列表，查看说明、参数和最近执行结果，并可从报表进入任务创建流程。

### 5.3 定时报表任务创建

用户可基于固定报表或当前可复用查询结果创建定时报表任务。创建时必须显式展示：

- 当前操作对象
- 频率
- 接收对象（具体为将使用的推送渠道配置，见 [§4.1](#41-定时报表投递与接收对象)）
- 是否立即执行一次

创建时还必须满足：

- 若存在多个可用推送配置，必须显式选择其一
- 若无可用推送配置，阻止提交并提示先完成推送配置
- 提交前同时校验报表执行权限和外部分发权限
- 不得将「发给我」默认解释为飞书私聊本人
- 若存在默认投递组合，可默认带出，但用户可取消或补充已有渠道
- 若用户表达“群和邮箱都要”，系统进入多渠道确认，而不是静默套用单一默认渠道

### 5.4 异常提醒任务

用户为核心指标配置提醒规则，在任务中心查看提醒历史和状态。

### 5.5 多渠道结果追踪

当一个定时报表任务绑定多个渠道时，需求上应满足：

- 任务中心可按渠道查看每次投递结果
- 部分渠道失败时需有清晰状态，不得整体笼统展示为成功
- 失败原因需定位到具体渠道配置

## 6. 当前阶段验收标准

验收前置条件：

- 至少存在一个已启用且经测试可用的推送配置
- 飞书渠道验收前，需验证 Webhook 可用；若依赖下载链接，需验证下载基址可访问
- 邮件渠道验收前，需验证邮件服务与收件人配置可用
- 飞书入口如对外宣称可用，需完成端到端联调，而非仅代码存在
- 若验收包含“群 + 个人邮箱同时收”，需同时准备飞书群配置和邮件配置，并验证多渠道结果可回查

- 运营专员可在飞书查询高频经营数据
- 固定报表可在 Web 中清晰浏览和执行
- 用户可从固定报表创建定时报表任务
- 用户可从当前查询结果创建定时报表任务
- 任务中心可查看最近执行情况和失败原因
- 通过已配置的飞书推送渠道可稳定送达定时报表结果（典型为群机器人 Webhook；非飞书私聊）
- 若启用邮件推送配置，可将定时报表以附件等形式送达指定收件人

关键验收用例：

- 单一可用渠道时，系统默认带出该渠道且确认区展示完整渠道信息
- 多个可用渠道时，未显式选择渠道不能提交
- 无可用渠道时，不能创建任务且有明确提示
- 无外部分发权限的用户不能创建或修改外部分发任务
- 已配置默认投递组合时，聊天确认页默认带出该组合且允许本次覆盖
- 用户表达“群和邮箱都要”时，系统进入多渠道确认流程
- 飞书渠道验收时，飞书群可收到消息并包含下载链接等关键信息
- 邮件渠道验收时，指定收件人可收到邮件且附件可打开
- 同一任务同时绑定飞书和邮件时，任务中心可分别看到渠道结果与失败原因
- 渠道失效后，任务中心能看到失败原因，系统不会自动切换其他渠道

## 7. 后续规划

### 7.1 清结算对账视角

后续补：

- 对账作业对象
- 账期和口径冻结模型
- 审计回放产品化
- 差异分析与复核流程

### 7.2 风险监控视角

后续补：

- 告警对象独立建模
- 监控规则中心
- 值守群和静默策略
- 告警处置闭环

### 7.3 投递与触达增强

- **飞书按用户维度 / 私聊投递**（开放平台 IM + 账号绑定等）：与当前群 Webhook 投递区分，待专项产品与方案；可与问数机器人会话能力统筹考虑。
