# Design Sprint 计划 — 对账域验证

> 基于 Jake Knapp《Sprint》5 天冲刺方法
> 执行时机：Month 3 Week 12（对账功能开发完成后）
> 或提前到 Month 1 Week 4（如需在开发前快速验证方向）

## 冲刺目标

**在 5 天内验证**：财务人员是否愿意用"模板化自动对账 + 飞书推送"替代当前的 Excel 手动对账流程。

---

## 参与人员

| 角色 | 人员 | 职责 |
|------|------|------|
| **决策者** | 财务负责人 or 产品负责人 | 最终拍板选方案 |
| **引导者** | 你 | 控制流程节奏 |
| **技术专家** | 1 名开发 | 判断方案可行性 |
| **业务专家** | 1 名财务人员 | 提供业务真实性校验 |
| **测试用户** | 3 名财务专员（未参与设计的） | 周五测试原型 |

---

## 5 天日程

### 周一：Map — 定义问题

**上午**：
- 画出对账流程全景图（基于 Event Storming 产出）
- 标出用户旅程中的关键时刻：
  1. 拿到源数据
  2. 执行匹配
  3. 处理差异
  4. 生成报告
  5. 确认签字

**下午**：
- 每人写出"我们最大的风险是什么"（便签）
- 投票选出冲刺聚焦点（预计是"执行匹配 + 处理差异"环节）
- 定义冲刺问题：**"财务人员能否在 30 分钟内完成以前需要 2 小时的对账匹配？"**

### 周二：Sketch — 独立画方案

**规则**：每人独立画方案，不讨论。

**上午**：
- 每人收集"闪电演示"素材（5 分钟/人）：
  - 参考 1：SuperSonic 现有的查询结果卡片
  - 参考 2：Excel 中对账差异标红的体验
  - 参考 3：飞书审批流中的"待处理"交互

**下午**：
- 每人画 3 个方案草图（Crazy 8s 快速草图法，每个 8 分钟）
- 然后选最佳方案画一个完整的 3 屏方案草图（Story Board）

**预期方案方向**：
- 方案 A：Web 端全功能对账工作台（重，但完整）
- 方案 B：飞书每日推送对账摘要 + Web 处理差异（轻入口 + 重处理）
- 方案 C：纯飞书交互（飞书卡片展示差异 + 按钮确认）

### 周三：Decide — 投票选方案

**上午**：
- 所有方案贴墙，沉默投票（每人 3 票贴圆点）
- 决策者用"超级票"做最终选择
- 预计选中方案 B（轻入口 + 重处理，与平台已有能力最匹配）

**下午**：
- 画出选中方案的 Storyboard（6 格漫画）：
  1. 财务人员早上打开飞书
  2. 收到昨日对账摘要卡片（匹配 980 条，差异 20 条）
  3. 点击"查看差异"按钮 → 跳转 Web 对账页
  4. Web 页面展示差异明细（差异类型、金额、来源）
  5. 逐条处理（确认 / 标记为待调整 / 忽略）
  6. 全部处理完 → 生成对账报告 → 发送给主管

### 周四：Prototype — 做原型

**用已有能力拼接，不写新代码**：

| Storyboard 帧 | 原型实现方式 |
|---------------|-------------|
| 飞书对账摘要卡片 | 用 FeishuCardTemplate 手动构造一张卡片 JSON |
| Web 对账页 | 用 Ant Design Pro Table 搭一个静态页面（硬编码数据） |
| 差异处理交互 | Pro Table 行内编辑 + 按钮（前端 Mock） |
| 对账报告 | 用已有的 CSV 导出功能导出一份模拟数据 |

**目标**：看起来像真的，但后端全是假数据。重点验证**流程是否通顺**，不验证数据准确性。

### 周五：Test — 用户测试

**邀请 3 名未参与设计的财务专员**，每人 45 分钟：

**测试脚本**：
1. "假设你今天早上打开飞书，看到这条消息"（展示对账摘要卡片）
2. 观察：他能看懂吗？第一反应是什么？
3. "你现在想处理这些差异，你会怎么做？"（观察是否点击"查看差异"）
4. 观察：Web 页面上他会先看什么？操作顺序是什么？
5. "处理完了，你觉得这个流程比你现在的对账方式好在哪里？差在哪里？"
6. 最后问："满分 10 分，你给这个流程打几分？"

**记录方式**：
- 每个用户一张记录表：看懂了吗？卡在哪？说了什么原话？打几分？
- 3 人测完后汇总：共同的卡点是什么？

---

## 冲刺决策标准

| 结果 | 判定 | 下一步 |
|------|------|--------|
| 3/3 用户打分 ≥ 7 分 | **全速推进** | 正式开发对账域功能 |
| 2/3 用户打分 ≥ 7 分 | **调整后推进** | 修改方案后再做 1 轮小测试 |
| 1/3 或 0/3 ≥ 7 分 | **转向** | 回到访谈阶段，重新理解痛点 |
