# 运营 Operating System 一期详细设计

本文承接 [运营数据工作台产品设计说明书](../../product/运营数据工作台产品设计说明书.md)，补充实现级产品规则，覆盖固定报表产品化、入口投递规则、驾驶舱主链路、异常处置流和一期验收细节。

## 1. 固定报表对象产品化

### 1.1 对象定义

固定报表是运营团队长期消费的核心对象，至少具备以下属性：

- 报表名称
- 业务摘要
- 最近结果时间
- 当前消费状态
- 历史结果入口
- 默认投递关系
- 我的订阅状态
- 发起定时报表任务入口

固定报表对象与定时报表任务对象必须明确区分：

- 固定报表：消费对象，回答“看什么”
- 定时报表任务：动作对象，回答“怎么持续送达”

### 1.2 页面目标

固定报表页需要支持以下高频任务：

- 快速找到目标报表
- 判断最新结果是否可直接使用
- 查看历史结果
- 判断是否已订阅、默认发往哪里
- 从固定报表发起定时报表任务

### 1.3 信息架构

固定报表页采用“筛选区 + 主列表区 + 详情区”的工作台结构，不采用均质卡片墙。

```text
固定报表
├── 顶部筛选区
│   ├── 搜索：按报表名称/业务主题搜索
│   ├── 业务域筛选
│   ├── 状态筛选：可查看 / 已订阅 / 暂无结果 / 结果过期 / 最近失败
│   └── 我的视图：全部 / 我订阅的 / 最近查看
├── 主列表区
│   ├── 报表名称
│   ├── 业务摘要
│   ├── 最近结果时间
│   ├── 当前状态
│   ├── 默认投递摘要
│   ├── 我的订阅状态
│   └── 操作：查看结果 / 查看历史 / 创建定时报表任务
└── 详情区
    ├── 报表基本信息
    ├── 最新结果摘要
    ├── 历史结果入口
    ├── 默认投递与订阅状态
    └── 发起定时报表任务
```

### 1.4 信息优先级与字段

固定报表页的信息优先级固定如下：

1. 当前是否可消费
2. 最近结果是否新鲜、是否成功
3. 是否已订阅、默认发往哪里
4. 报表的业务含义与口径
5. 业务域、维护人等归档信息

不得把部署时间、模板快照等实现视角字段放在主列表主层级中。

每条固定报表至少展示：

- 报表名称
- 口径摘要
- 最近结果时间
- 状态：可查看 / 暂无结果 / 结果过期 / 最近失败
- 默认投递摘要
- 我的订阅状态
- 主操作：查看结果
- 次操作：查看历史、创建定时报表任务

### 1.5 状态原则

固定报表必须完整覆盖以下状态：

- 暂无固定报表
- 筛选后无匹配结果
- 已发布但暂无结果
- 结果过期
- 最近失败
- 未配置默认投递
- 部分渠道异常

所有状态都必须明确回答三件事：

- 当前发生了什么
- 为什么会这样
- 用户下一步可以做什么

结果过期不是错误，最近失败也不等于完全不可消费。若存在上一版结果，应明确提示是否仍可参考，避免把“最新失败”错误表达成“系统不可用”。

### 1.6 用户旅程

| Step | 用户动作 | 页面要解决的问题 |
|------|----------|------------------|
| 1 | 进入页面 | 今天先看哪些报表 |
| 2 | 扫描列表 | 每个报表是干什么的 |
| 3 | 判断状态 | 当前结果还能不能信 |
| 4 | 打开详情 | 是否可直接使用最新结果 |
| 5 | 查看历史 | 没有最新结果时还能否参考上一版 |
| 6 | 判断订阅 | 是否已进入稳定分发闭环 |
| 7 | 创建任务 | 本次会创建什么任务、发往哪里 |

页面需要让用户逐步形成稳定心智：

- 固定报表页负责消费对象管理
- 任务中心负责动作对象治理

### 1.7 视觉、响应式与可访问性

- 固定报表页属于高频任务型 App UI，不采用内容市场式卡片墙
- 主界面采用高密度、可扫读的列表工作区
- 状态识别优先于装饰性视觉
- Desktop 使用完整列表工作区，详情通过 Drawer 展开
- Tablet 保留列表模式，收敛次级字段
- Mobile 退化为紧凑信息卡片列表，详情使用独立详情页或全屏详情层
- 状态不可仅依赖颜色表达，需有文字与可读语义
- 主操作触控区域最小为 44px
- 详情层打开后需正确管理键盘焦点与返回位置

### 1.8 关键决策与边界

实施前需锁定以下决策：

- Desktop 详情承载方式：Drawer
- Mobile 详情承载方式：独立详情页或全屏详情层
- 默认主视图：优先“我订阅的”，无订阅时回退“全部固定报表”
- 结果过期时允许查看，但必须显式标记“已过期”
- 默认投递在主列表展示简摘要，完整信息放详情区
- 历史结果入口放入详情区，不占据主列表主操作位

固定报表页负责：

- 展示固定报表对象
- 提供最新结果与历史结果入口
- 提供订阅状态与默认投递信息
- 作为创建定时报表任务的起点

固定报表页不负责：

- 复杂任务治理
- 复杂推送渠道配置
- 复杂失败修复流程

## 2. 入口与投递规则

### 2.1 入口边界

| 入口 | 当前定位 | 负责内容 | 不负责内容 |
|------|---------|---------|-----------|
| Web | 配置与治理入口 | 驾驶舱查看、经营主题维护、固定报表维护、任务管理、责任账本与执行历史查看 | 高噪音即时沟通 |
| 飞书 | 轻消费工作面 | 查看今日必看结果、查看异常摘要、确认/接手异常、轻量任务动作、接收结果 | 复杂报表配置、复杂任务编辑、复杂渠道治理 |
| API/MCP | 外部接入入口 | 对外暴露稳定能力接口 | 承载主用户产品心智 |

### 2.2 投递语义

- 投递目标与对话入口解耦
- 接收对象指具体渠道配置，不指“当前操作者本人”
- 飞书当前实现以群 Webhook + 下载链接为主
- 邮件更接近“只发给我”的心智
- 所有自然语言文案都必须可验证，避免承诺未实现的飞书私聊直达

### 2.3 渠道选择与确认

- 无可用配置时，不允许创建定时报表任务
- 仅有一个可用配置时，可默认带出，但必须显式展示
- 存在多个可用配置时，必须显式选择，不得静默推断
- 接收对象属于任务配置的一部分，后续调整需可回查
- 配置失效时必须暴露失败原因，不得静默兜底

对话入口采用轻确认模式：

- 自然语言只解析投递意图，不直接猜具体配置实例
- 确认卡片必须展示本次使用的具体渠道配置
- 对话入口只处理“选择已有配置 + 确认创建”
- 用户确认后的渠道选择必须写回任务对象

默认投递策略支持“默认策略 + 本次覆盖”：

- 默认值可挂在租户级或报表级
- 对话创建时可先带出默认组合，但不得锁死
- 执行时必须具体到渠道配置实例
- 禁止自动全选所有启用渠道
- 本次确认优先于旧默认值

### 2.4 权限与治理

- 双层权限校验：同时校验报表执行权限与外部分发权限
- owner 责任明确：接收对象不是权限来源，不能放大数据可见范围
- 高风险动作显式确认并留痕
- 审计字段至少覆盖操作人、入口、报表对象、渠道、执行结果和失败原因

## 3. 主链路流程

### 3.1 每日经营驾驶舱

用户进入系统后，默认先到每日经营驾驶舱，而不是先进入模块导航。

驾驶舱需要先回答：

1. 今天最该看哪些经营主题
2. 哪些结果已经更新且可直接使用
3. 哪些异常待确认、待接手或待恢复
4. 哪些任务或渠道存在可靠性风险

### 3.2 经营主题消费

用户从驾驶舱进入经营主题，在同一主题下完成结果消费、异常查看、任务跳转与责任追踪。

一个经营主题下至少需要能看到：

1. 当前主题最重要的固定报表
2. 相关异常处置流
3. 相关定时报表任务
4. 默认投递与责任信息
5. 可信度摘要

### 3.3 临时查询

用户在 Web 或飞书发起查询，系统返回结果卡片或表格，并支持导出或保存为任务。

### 3.4 固定报表消费

用户浏览固定报表列表，查看说明、参数和最近执行结果，并可从报表进入任务创建流程。

### 3.5 定时报表任务创建

创建流程必须完成以下动作：

1. 确认当前报表对象
2. 配置频率与接收对象
3. 显式确认渠道与是否立即执行
4. 写入任务中心并可追踪执行状态

创建流程必须满足：

- 无可用推送配置时不允许完成创建
- 同时校验报表执行权限和外部分发权限
- 确认区展示渠道名称、类型和目标摘要
- 不得把「发给我」解释为飞书私聊本人
- 已配置默认投递组合时可先带出，但用户需可见、可改

### 3.6 异常处置流

用户或系统发现异常后，不以“提醒已发出”作为流程终点，而以“异常已确认并进入处置状态”作为流程起点。

异常处置流至少包含：进入待确认、确认已知、接手处理、补充记录、标记恢复或关闭，并在责任账本中留痕。

### 3.7 飞书轻消费工作面

飞书侧当前阶段支持以下轻量但高频的消费动作：

- 查看今日必看结果
- 查看异常摘要
- 确认异常已知或接手异常
- 查看上一版可参考结果
- 对现有任务执行轻量动作

### 3.8 多渠道执行与结果追踪

当一个任务绑定多个投递渠道时，产品对象上仍应视为**一个任务，多个渠道结果**，并满足：

- 任务中心可按渠道查看执行结果
- 部分渠道失败时需明确暴露，而不是笼统展示为成功
- 失败原因应定位到具体渠道配置
- 后续若支持重试，应优先支持按失败渠道重试

## 4. 一期验收

### 4.1 验收前置条件

- Web 端已具备固定报表、任务中心和推送配置管理入口
- 至少存在一个已启用且经测试可用的推送配置
- 飞书链路已完成 Webhook 可用性验证；若走链接下载，还需验证下载基址外部可访问
- 邮件链路若纳入验收，需完成邮件服务配置并验证附件送达
- 若验收包含“群 + 个人邮箱同时收”，需至少准备一条飞书群配置和一条邮件配置，并验证多渠道执行结果可回查

### 4.2 核心验收结果

达到以下结果时，当前阶段方可按“可试点”口径验收：

- 运营负责人每天可先从驾驶舱判断今日经营重点，而不是先进入工具菜单
- 团队能围绕至少 2 到 3 个经营主题稳定消费固定报表
- 固定报表可替代至少 2 到 3 类重复人工查数/整理/分发链路
- 定时报表任务可稳定创建、执行、追踪失败原因
- 异常不只会被提醒，还能被确认、接手和留痕
- 飞书可承担结果消费和异常确认等轻量高频动作，而不只是通知
- 责任账本能回看结果、投递、异常和责任链的关键记录

### 4.3 关键验收场景

- 渠道选择规则正确：单渠道默认、多渠道显式选择、无渠道阻断创建
- 默认投递与本次覆盖规则正确：不静默推断、不自动全选
- 飞书与邮件多渠道执行结果可分渠道回查
- 驾驶舱能聚合今日必看主题、关键报表状态和待确认异常
- 飞书中的异常确认或接手可正确回写到 Web
- 异常关闭后，责任账本可回看确认人、接手人、处理记录和关闭时间
