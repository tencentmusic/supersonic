# P0 上线收口实施方案

模板报表子系统 Phase 1-3 代码已全部在主干。本方案解决"代码存在但未形成可验证闭环"的问题，按优先级排列 5 个收口任务。

---

## 1. 全链路联调（优先级最高）

### 1.1 联调范围

单模板 × MySQL × 结构化查询 × 飞书推送，覆盖以下完整链路（SqlTemplateConfig 路径一期前端无入口，不纳入联调范围）：

```
创建模板 → 部署 → 手动执行 → 查看结果
                → 创建调度 → 等待触发 → 查看执行记录
                → 导出（同步 + 异步）→ 下载文件
                → 飞书推送 → 验证卡片内容
                → 模拟失败 → 验证重试 → 验证自动禁用
```

**具体执行步骤**

前置准备：
- 在飞书测试群添加自定义机器人，记录 Webhook 地址和签名 Secret；Webhook 必须是 `open.feishu.cn` 或 `open.larksuite.com` 域名。
- 准备一个已通过模板执行/调度链路验证的调度任务 ID：`{scheduleId}`。
- 准备管理员登录态 Cookie 或 Token；创建、更新、测试推送配置接口仅管理员可操作。

1. 创建飞书推送配置。

```bash
curl -X POST "$BASE_URL/api/semantic/delivery/configs" \
  -H "Content-Type: application/json" \
  -H "Cookie: $COOKIE" \
  -d '{
    "name": "P0联调-飞书测试群",
    "deliveryType": "FEISHU",
    "deliveryConfig": "{\"webhookUrl\":\"https://open.feishu.cn/open-apis/bot/v2/hook/xxx\",\"secret\":\"xxx\",\"msgType\":\"interactive\",\"title\":\"P0联调报表推送\",\"content\":\"模板：${reportName}\\n调度：${scheduleName}\\n执行时间：${executionTime}\\n数据行数：${rowCount}\\n下载链接：${downloadUrl}\"}",
    "enabled": true,
    "maxConsecutiveFailures": 3,
    "description": "P0上线收口飞书推送联调"
  }'
```

记录响应中的 `id` 为 `{deliveryConfigId}`。创建后先查库确认配置落库且启用：

```sql
select id, name, delivery_type, enabled, consecutive_failures, max_consecutive_failures
from s2_report_delivery_config
where id = {deliveryConfigId};
```

2. 发送一次配置测试消息，先排除飞书机器人配置问题。

```bash
curl -X POST "$BASE_URL/api/semantic/delivery/configs/{deliveryConfigId}:test" \
  -H "Cookie: $COOKIE"
```

验收：接口无异常返回，飞书测试群收到 `Test Report` / `Test Schedule` 测试消息。若失败，优先检查 Webhook、Secret、机器人群权限、应用到飞书域名的出网策略。

3. 绑定推送配置到调度任务。

先读取当前调度，避免覆盖已有字段：

```bash
curl "$BASE_URL/api/v1/reportSchedules/{scheduleId}" \
  -H "Cookie: $COOKIE"
```

再用 PATCH 只更新推送配置 ID；若原来已有多个配置，`deliveryConfigIds` 使用英文逗号拼接，例如 `"12,34"`。

```bash
curl -X PATCH "$BASE_URL/api/v1/reportSchedules/{scheduleId}" \
  -H "Content-Type: application/json" \
  -H "Cookie: $COOKIE" \
  -d '{
    "deliveryConfigIds": "{deliveryConfigId}"
  }'
```

查库确认绑定关系：

```sql
select id, name, enabled, delivery_config_ids, next_execution_time
from s2_report_schedule
where id = {scheduleId};
```

4. 触发一次调度并验证成功推送。

优先手动触发，避免等 5 分钟 Cron 窗口：

```bash
curl -X POST "$BASE_URL/api/v1/reportSchedules/{scheduleId}:trigger" \
  -H "Cookie: $COOKIE"
```

随后查询最近一次执行和推送记录：

```sql
select id, schedule_id, status, start_time, end_time, result_location, error_message
from s2_report_execution
where schedule_id = {scheduleId}
order by id desc
limit 3;

select id, delivery_key, schedule_id, execution_id, config_id, delivery_type, status,
       file_location, error_message, retry_count, started_at, completed_at, delivery_time_ms
from s2_report_delivery_record
where schedule_id = {scheduleId}
order by id desc
limit 5;
```

验收：
- `s2_report_execution.status = SUCCESS`。
- `s2_report_delivery_record.status = SUCCESS`，`delivery_type = FEISHU`，`config_id = {deliveryConfigId}`。
- 飞书群收到卡片消息，内容包含模板名/报表名、调度名或执行时间、数据行数；若本次执行产生 `result_location` 或下载地址，卡片中应包含下载链接。
- 如需审计回放联动，打开 `GET /api/v1/report-executions/{executionId}/snapshot`，确认 `deliveryRecords` 中出现本次飞书推送记录。

5. 验证 Webhook 错误的失败记录。

复制一份临时失败配置，不修改已验证成功的配置：

```bash
curl -X POST "$BASE_URL/api/semantic/delivery/configs" \
  -H "Content-Type: application/json" \
  -H "Cookie: $COOKIE" \
  -d '{
    "name": "P0联调-飞书失败验证",
    "deliveryType": "FEISHU",
    "deliveryConfig": "{\"webhookUrl\":\"https://open.feishu.cn/open-apis/bot/v2/hook/invalid-p0-test\",\"secret\":\"xxx\",\"msgType\":\"interactive\",\"title\":\"失败验证\",\"content\":\"${reportName}\"}",
    "enabled": true,
    "maxConsecutiveFailures": 2,
    "description": "P0上线收口飞书失败路径验证"
  }'
```

将调度临时绑定到失败配置 `{badDeliveryConfigId}` 后触发：

```bash
curl -X PATCH "$BASE_URL/api/v1/reportSchedules/{scheduleId}" \
  -H "Content-Type: application/json" \
  -H "Cookie: $COOKIE" \
  -d '{"deliveryConfigIds":"{badDeliveryConfigId}"}'

curl -X POST "$BASE_URL/api/v1/reportSchedules/{scheduleId}:trigger" \
  -H "Cookie: $COOKIE"
```

验收失败记录：

```sql
select id, status, error_message, retry_count, next_retry_at
from s2_report_delivery_record
where config_id = {badDeliveryConfigId}
order by id desc
limit 5;
```

预期：生成 `FAILED` 记录，`error_message` 可读，`next_retry_at` 非空（可重试错误）。

6. 验证连续失败自动禁用。

由于失败配置 `maxConsecutiveFailures = 2`，连续触发两次后检查配置状态：

```bash
curl -X POST "$BASE_URL/api/v1/reportSchedules/{scheduleId}:trigger" \
  -H "Cookie: $COOKIE"
```

```sql
select id, enabled, consecutive_failures, max_consecutive_failures, disabled_reason
from s2_report_delivery_config
where id = {badDeliveryConfigId};
```

验收：`enabled = 0/false`，`consecutive_failures >= max_consecutive_failures`，`disabled_reason` 含 `Auto-disabled after` 和最后一次失败原因。

7. 恢复现场。

将调度恢复到成功配置，并删除临时失败配置：

```bash
curl -X PATCH "$BASE_URL/api/v1/reportSchedules/{scheduleId}" \
  -H "Content-Type: application/json" \
  -H "Cookie: $COOKIE" \
  -d '{"deliveryConfigIds":"{deliveryConfigId}"}'

curl -X DELETE "$BASE_URL/api/semantic/delivery/configs/{badDeliveryConfigId}" \
  -H "Cookie: $COOKIE"
```

最终验收口径：D-1~D-4 以成功配置的推送记录和飞书群消息截图/录屏为准；D-5 以失败配置的 `s2_report_delivery_record.FAILED + error_message` 为准；D-6 以 `s2_report_delivery_config.enabled=false + disabled_reason` 为准。

#### 异常场景

- [x] 模板下线后触发调度 → 执行应跳过并记录失败原因 — **单测**：`ReportExecutionOrchestratorTest#executeShouldRecordFailedWhenTemplateOffline`
- [x] MySQL 连接超时 → 执行记录状态 FAILED，错误信息含超时原因 — **单测**：`ReportExecutionOrchestratorTest#executeShouldRecordFailedWhenQuerySocketTimeout`（`SocketTimeoutException` 模拟查询层读超时；真实 JDBC/MySQL 超时可选集成测或手工抽检）
- ~~[ ] SqlTemplateConfig 变量未传值 → SqlTemplateEngine 应报错而非生成错误 SQL（B-03）~~ — **N/A**：同 A-6，一期前端无此路径入口

**验收口径（与 §1.2「传入非法参数…」条相同，以自动化测试通过为准）**：上述 **未标 N/A** 的条目均可优先用 **自动化测试** 收口——对 `ReportExecutionOrchestrator`、`ReportScheduleJob` / 调度触发链路等，使用 **Mockito 模拟** 模板状态为下线、**模拟** `SemanticLayerService` / 数据源抛出超时或 `SQLException`（或包装后的运行时异常），断言 **不落成功路径**、**执行记录为 FAILED/跳过** 且 **`errorMessage`（或等价字段）含可读原因**。单测全部通过即可在联调报告中将该项标为通过；若需验真实 MySQL 超时行为，再补 **集成测** 或 **可选手工** 抽检。

### 2.2 告警规则

文件：`docker/prometheus/rules/report-slo-alert-rules.yml`

```yaml
groups:
  - name: report-slo
    rules:
      - alert: ReportExecutionSuccessRateLow
        expr: |
          (
            sum(rate(supersonic_report_execution_total{status="SUCCESS"}[10m]))
            /
            sum(rate(supersonic_report_execution_total[10m]))
          ) < 0.95
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "报表执行成功率低于 95%"

      - alert: ReportExecutionP95High
        expr: |
          histogram_quantile(0.95,
            rate(supersonic_report_execution_duration_seconds_bucket[10m])
          ) > 10
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "报表执行 P95 时延超过 10 秒"

      - alert: ExportTaskPendingHigh
        expr: supersonic_report_export_pending > 10
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "导出任务积压超过 10 个（持续 10 分钟）"

      - alert: DeliveryFailRateHigh
        expr: |
          (
            sum(rate(supersonic_report_delivery_total{status="FAILED"}[30m]))
            /
            sum(rate(supersonic_report_delivery_total[30m]))
          ) > 0.05
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "推送失败率超过 5%"

      - alert: DeliveryConfigAutoDisabled
        expr: supersonic_report_delivery_config_disabled > 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "存在被自动禁用的推送配置"
```

注意：所有规则都带 `for` 持续条件（B-09），避免瞬时抖动触发告警风暴。

### 4.2 一期覆盖的 4 个场景

| 场景 | 现象 | 关键排查命令 |
|------|------|-------------|
| 调度失败 | 执行记录状态 FAILED | 查 `s2_report_execution` → 看 error_message → 检查连接池/超时配置 |
| 导出异常 | 导出任务长时间 PENDING 或 FAILED | 查 `s2_export_task` 状态 → 检查 EXPORT 线程池使用率 → 检查磁盘空间 |
| 推送失败 | 飞书群没收到消息 | 查 `s2_report_delivery_record` → 看 error_message → 检查 Webhook URL 有效性 → 检查是否被自动禁用 |
| 推送被自动禁用 | 告警触发 `DeliveryConfigAutoDisabled` | 查 `s2_report_delivery_config` disabled 原因 → 修复根因 → 手动恢复启用 |

## 5. 压测与阈值校准（优先级第五）

### 5.1 一期范围：功能级基准测试

在 CI 中用 JUnit 验证以下行为正确性：

| 测试项 | 验证内容 | 实现方式 |
|--------|---------|---------|
| 连接池隔离 | EXPORT 池满时 INTERACTIVE 池不受影响 | 并发提交导出任务占满 EXPORT 池，同时发起交互查询 |
| 导出阈值分流 | < 阈值走同步，≥ 阈值走异步 | 构造不同行数的结果集，验证分流行为 |
| 调度并发限制 | 租户并发 > 5 时排队（AG-14） | CountDownLatch 模拟 6 个并发调度任务 |
| Quartz Misfire | 应用重启后错过的任务按策略补跑或跳过 | 注册任务 → 停止 Scheduler → 等待错过 → 重启 → 验证行为 |
| MySQL EXPLAIN 估算偏差 | RowCountEstimator 对 MySQL 的估算偏差在 5 倍以内 | 准备已知行数的表，对比估算值和实际值 |
