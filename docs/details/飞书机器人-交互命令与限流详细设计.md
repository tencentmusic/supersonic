## 1. 模板列表命令

### 1.1 目标

展示当前用户有权访问的 Agent 列表，每个 Agent 代表一个可查询的业务领域。用户可通过 `/template` 或“模板”触发，列表按用户历史使用频次降序排列，常用 Agent 排在前面。

### 1.2 实际流程

```text
用户发送 "/template" 或 "模板"
  │
  ▼
SuperSonicApiClient.getAgentList(user)
  → GET /api/chat/agent/getAgentList
  │
  ├─ 过滤 status=1（在线）
  │
  ├─ 结果为空 → 回复 "暂无可用的查询模板，请联系管理员配置"
  │
  └─ 有 Agent
      │
      ▼
    统计用户 agent 使用频次（s2_feishu_query_session.agent_id，SUCCESS 记录）
    按频次降序排列
      │
      ▼
    renderAgentListCard() 渲染卡片
```

### 1.3 渲染要点

- Header：`可用查询模板`
- Body：每个 Agent 一段，展示名称、描述和示例问题
- 当前 Agent 需要高亮标识
- 新用户无历史记录时按 API 原始返回顺序展示

### 1.4 实现映射

- `TemplateListHandler`：查询 Agent 列表并按使用频次重排
- `FeishuQuerySessionMapper.countAgentUsage(openId)`：按 `agent_id` 统计 SUCCESS 次数
- `FeishuCardRenderer.renderAgentListCard()`：渲染富文本卡片
- `FeishuQuerySessionDO.agentId`：持久化当前会话路由的 Agent

## 2. SQL 预览命令

### 2.1 目标

用户发送 `/sql <查询文本>` 时，仅执行 NL2SQL 解析生成 SQL，不实际执行查询，用于确认 SQL 正确性后再执行。

### 2.2 流程

```text
用户发送 "/sql 查昨天北京的GMV"
  │
  ▼
提取查询文本（去除 "/sql " 前缀）
  │
  ▼
SuperSonicApiClient.parse(queryText, agentId, user)
  │
  ├─ 解析失败 → 回复错误卡片
  │
  └─ 解析成功
      │
      ▼
    取 selectedParses[0]
      │
      ▼
    renderSqlPreviewCard() 渲染 SQL 预览卡片
```

### 2.3 关键实现约束

- 使用当前会话 `msg.getAgentId()` 作为路由依据，而不是固定默认 Agent
- 仅调用解析接口，不直接执行查询
- SQL 展示优先级：`correctedS2SQL > parsedS2SQL > querySQL`
- 预览卡片应明确提示“直接发送查询文本可执行此 SQL”

### 2.4 路由规则

`FeishuMessageRouter` 需识别 `/sql` 命令并路由到 `PreviewMessageHandler`。

### 2.5 主要实现点

- `PreviewMessageHandler`：解析文本、调用 API、处理错误并渲染卡片
- `SuperSonicApiClient.parse()`：调用 `POST /api/chat/query/parse`
- `FeishuCardRenderer.renderSqlPreviewCard()`：展示数据集、置信度和 SQL 代码块

## 3. 频率限制

### 3.1 目标

限制单用户的飞书查询频率，默认每分钟最多 20 次，防止刷屏和入口滥用。

### 3.2 设计结论

- 限流放在 `FeishuBotService.handleMessage()` 中，身份映射成功后、路由分发前执行
- 限流属于入口层能力，独立于底层查询与权限
- 当前采用固定窗口计数器，足够覆盖飞书机器人场景

### 3.3 实现方式

`FeishuCacheService` 对外暴露 `isRateLimited(openId, maxPerMinute)`。

两种实现：

- Caffeine：本地计数器 + TTL
- Redis：`INCR` + 条件 `EXPIRE`

### 3.4 调用点

```java
User user = userMappingService.resolveUser(openId);
if (user == null) {
    ...
}

if (cacheService.isRateLimited(openId, 20)) {
    messageSender.replyText(messageId, "查询过于频繁，请稍后再试（每分钟最多 20 次）");
    return;
}

MessageHandler handler = messageRouter.route(content);
handler.handle(msg, user);
```

### 3.5 配置项

```yaml
s2:
  feishu:
    rate-limit:
      max-per-minute: 20
```

### 3.6 取舍

- 固定窗口实现简单，便于维护
- 窗口边界存在突发放大问题，但对当前 Bot 场景可接受
- 若未来需要更精确控制，可升级为 Redis Sorted Set 滑动窗口
