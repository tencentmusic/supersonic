# 智能运营数据中台 — 整体架构设计

## 0. 一页摘要

### 0.1 平台目标

**一站式智能运营平台**，长期整合数仓查询、提数对账、风控监控、飞书协作四大能力；当前阶段先收敛为运营数据交付平台。

- 运营人员 — 日常取数、业绩查看、对账核查
- 数据分析师 — 深度分析、自定义查询、报表制作
- 财务人员 — 财务报表、收支对账、清算核对
- 风控人员 — 风险监控、预警处理、黑名单管理
- 管理层 — 经营看板、关键指标、决策支持

### 0.1.1 当前阶段主用户

当前阶段唯一主视角是**运营数据负责人**。

当前阶段主要服务三类人：

- 运营专员：查数、导出、接收固定报表
- 运营主管：订阅定时报表、查看任务状态、接收异常提醒
- 数据分析师：维护固定报表、口径和参数

以下角色暂不作为本阶段主验收对象：

- 清结算对账负责人
- 风险监控负责人

### 0.2 当前结论

- **主链路**：`Web / API / 飞书入口 → SuperSonic Parser / Agent / Plugin → 权限 / 审计 → 数据源或外部系统`
- **Dify 定位**：继续沿用，但仅作为**外部可选编排层**，不下沉为系统内核
- **MCP 定位**：对外暴露工具协议，不替代内部 Plugin 体系
- **当前主目标**：让运营团队在 Web 和飞书中完成高频查数、固定报表订阅、结果分发和异常提醒，减少人工取数与人肉催报表
- **MVP 顺序**：先完成运营查询闭环（MVP-1），再完成分发闭环（MVP-2），最后进入对账/风控等业务编排闭环（MVP-3）

### 0.4 MVP 边界

| MVP 层级 | 覆盖范围 | 非目标 |
|---------|---------|-------|
| **MVP-1 查询闭环** | 日常查数、模板报表、CSV/Excel 导出、用户调度 | 对账、风控、公开分享 |
| **MVP-2 分发闭环** | 报表/结果分发到飞书、告警订阅 | 审批流、复杂跨系统编排 |
| **MVP-3 业务编排** | 对账、风控、审批等多步骤流程 | Dashboard、嵌入、连接器市场 |

### 0.5 当前阶段一级产品对象

本阶段面向业务侧只定义以下一级对象：

- 临时查询
- 固定报表
- 定时报表任务
- 异常提醒任务
- 任务中心

`Agent`、`Plugin`、`DataSet`、`QueryMode` 属于实现概念，不作为本阶段业务主叙事。

## 一、与现有工程的关系

SuperSonic 已实现**数仓查询**的核心能力（NL2SQL、语义模板、Agent/Plugin、数据权限）。当前阶段以运营数据交付为主线，对账、风控监控属于后续垂直业务扩展：

| 能力域 | SuperSonic 现状 | 实现路径 |
|--------|----------------|---------|
| 数仓查询 | **已实现** — NL2SQL + 模板 + Agent + 导出 | 直接复用，补充 CSV 导出和用户定时调度 |
| 提数对账 | **未实现** | 新建语义模板（对账域），或 MCP 工具对接业务系统 |
| 风控监控 | **未实现** | 新建语义模板（风控域），或 MCP 工具对接风控系统 |
| 飞书协作 | **已有设计** | 新建 Plugin (WEB_SERVICE) 或 MCP 工具对接飞书 API |
| 综合流程编排 | **未实现** | Dify Workflow 外部编排，或扩展 Agent 多工具链式调用 |

## 二、架构原则

- **SuperSonic 内核优先**：查询解析、语义建模、权限控制、导出、调度、审计均以内置 Agent/Plugin/API 体系为主，不引入第二套内部运行时。
- **Dify 外置使用**：Dify 作为**外部可选编排层/渠道入口**，用于机器人接入、轻量工作流编排，不作为系统内部核心依赖。
- **MCP 对外优先**：MCP 用于向 Dify 等外部编排系统暴露稳定工具能力，不反向替代 SuperSonic 内部 Plugin 体系。
- **写操作强治理**：所有外部系统写入、通知、审批、事件创建类动作必须具备幂等、审计、确认和失败补偿设计。
- **先 MVP 再扩展**：优先交付查询、导出、调度、通知四类最小闭环能力，再扩展对账和风控场景。
- **入口边界稳定**：Web 负责配置与治理，飞书负责消费与轻交互，API/MCP 负责外部接入。
- **查询与动作分层**：查询用于看结果，动作用于发结果、定时发、建提醒；动作必须具备确认、留痕与状态可回查能力。

## 三、平台主链路

```
用户交互层
  Web 查询平台 [已上线]  |  飞书机器人 [待开发]  |  API 调用 [已上线]  |  定时触发 [已有设计]
                                       │
                         SuperSonic 内部执行链路（主路径）
                                       │
              Parser / Agent / Plugin 路由 [已实现]
              数据查询 Agent | 模板报表 Agent | 综合流程 Agent [待开发]
                                       │
              Dify / MCP 外部编排与工具暴露（可选层）[已有设计]
                                       │
              数据服务层：NL2SQL 引擎 | 模板引擎 | 权限引擎 | 导出服务 [均已实现]
                                       │
              数据存储层：ClickHouse | Hive | MySQL | PG | H2 | DuckDB [已支持]
```

### 3.1 当前阶段入口职责

| 入口 | 当前定位 | 负责内容 | 不负责内容 |
|------|---------|---------|-----------|
| Web | 配置与治理入口 | 固定报表维护、参数配置、任务管理、执行历史查看 | 轻量即时协作 |
| 飞书 | 消费与轻交互入口 | 临时查询、查看结果、轻量导出、确认创建任务、接收结果 | 复杂报表配置、复杂任务编辑 |
| API/MCP | 外部接入入口 | 对外暴露稳定能力接口 | 承载主用户产品心智 |

### 3.2 关键架构决策

| 决策项 | 设计方案 | 建议 |
|--------|---------|------|
| LLM 框架 | Spring AI | **保持 LangChain4j** — 已深度集成，迁移成本高 |
| 工具协议 | MCP | **内外分层** — 内部继续用 Plugin，外部对接 Dify 再暴露 MCP |
| 工作流引擎 | Dify Workflow | **继续沿用 Dify，但外置** — 不下沉为内核 |
| 缓存 | Redis | **按需升级** — 单机用 Caffeine，集群部署引入 Redis |
| 消息队列 | RabbitMQ | **按需引入** — 仅定时报表推送等异步场景才需要 |

## 四、子系统边界

### 4.1 模板报表子系统

**负责**：语义模板 CRUD、报表执行编排（Web/Agent/Schedule 统一入口）、CSV/Excel 导出、Quartz 调度、异步导出任务中心、查询资源隔离。

**核心组件**：`SemanticTemplateServiceImpl`, `ReportExecutionOrchestrator`, `DownloadServiceImpl`, `QuartzJobService`

详见：[模板报表子系统技术设计说明书](模板报表子系统技术设计说明书.md)

### 4.2 飞书机器人子系统

**负责**：飞书消息事件接收与验签、用户身份映射（open_id → 平台账号）、消息路由（查询/导出/模板列表等 Handler）、协议适配（不直接依赖 chat-server 重量级 service）。

**核心组件**：`FeishuEventController`, `FeishuMessageDispatcher`, `FeishuUserMappingService`, `SuperSonicApiClient`

详见：[飞书机器人用户交互层设计方案](飞书机器人用户交互层设计方案.md)

### 4.3 Dify-MCP 层

**负责**：向外部编排系统（Dify）暴露稳定的工具协议（MCP），汇聚 DataWarehouse / Feishu / 对账 / 风控等工具。飞书 MCP 工具层复用 `feishu/server` 已有的 `FeishuMessageSender`、`FeishuTokenManager` 基础能力。

**当前状态**：已有设计，Phase 2 开始实施。

## 六、MVP 路线图

### Phase 1：MVP-1 查询闭环

目标：不引入 Dify 依赖，先完成运营查数、固定报表、导出、调度三类能力闭环。

- [ ] **统一执行编排** — `ReportExecutionOrchestrator`，Web / Agent / Schedule 三入口统一委托
- [ ] **CSV 导出** — 扩展 `DownloadServiceImpl`，新增 CSV 输出格式
- [ ] **异步导出** — Export Worker + 任务中心 + JDBC 流式读取 + EasyExcel 流式写入
- [ ] **用户定时调度** — Quartz Scheduler 内嵌，用户自定义 Cron + Web 管理界面
- [ ] **查询资源隔离** — SQL 超时注入 + 报表专用连接池 + 扫描行数估算 + 租户级执行配额
- [ ] **数据同步** — `s2_data_sync_config` 增量同步，业务库 → 分析库，Web 管理界面

验收标准：运营团队可在 Web/API 完成查询、固定报表执行、导出、调度；执行链路统一收口到 `ReportExecutionOrchestrator`。

### Phase 2：MVP-2 分发闭环

目标：Dify 继续沿用但仅作为外部入口；建立统一工具层，完成飞书分发。

- [ ] **MCP Server 基座** — 新建 `mcp/` 模块，实现 MCP 协议 Tool 注册/发现
- [ ] **DataWarehouse MCP Tools** — NL2SQL / 模板查询 / 导出能力封装为 MCP Tool
- [ ] **Feishu MCP Tools** — 飞书开放平台 API 封装，复用 `feishu/server` 基础能力
- [ ] **Dify Workflow 对接** — 配置 MCP Server 地址，创建数据查询 + 飞书协作 Workflow
- [ ] **外部分发权限** — 增加飞书群、公开分享、审批发起等外部分发权限控制
- [ ] **告警订阅** — 扩展调度，支持阈值条件告警 + 飞书推送通道

验收标准：定时报表推送可跑通；飞书通知、文档上传具备最小审计与失败补偿能力。

### Phase 3：MVP-3 业务编排闭环

目标：引入对账/风控等动作型工具，并纳入统一治理。

- [ ] **对账域语义模板** — 交易流水/账户余额/清算数据建模
- [ ] **对账逻辑工具** — `reconcileData` / `analyzeDifferences` / `generateReconciliationReport`
- [ ] **风控域语义模板** — 风险指标/黑名单查询建模
- [ ] **风控操作工具** — `matchAlertRules` / `calculateRiskScore` / `createRiskEvent`
- [ ] **综合流程 Agent** — Dify Workflow 编排多工具链式调用
- [ ] **动作型工具治理** — 幂等、确认、审计、失败补偿全链路

验收标准：对账/风控工具区分查询类与动作类；动作类工具具备幂等、确认、审计、补偿；外部编排失败不破坏内部核心链路。

## 七、风险与开放决策

| 风险 / 决策项 | 说明 | 当前立场 |
|-------------|------|---------|
| Dify 外置边界 | 如果 Dify 承载太多逻辑，迁移成本高 | 明确约定：Dify 只做入口路由和 Prompt 编排，执行逻辑留在 SuperSonic |
| MCP 协议稳定性 | MCP 仍在演进，锁定风险存在 | Phase 2 再决定是否引入；内部 Plugin 不依赖 MCP |
| Redis 引入时机 | 过早引入增加运维复杂度 | 单机用 Caffeine；集群部署 + 跨节点任务状态才引入 |
| 动作型工具安全 | 写操作一旦开放，拉黑/审批/通知等副作用不可逆 | MVP-3 之前禁止自动执行写操作；强制人工确认 |
| 数据同步 vs 外部 ETL | 内置轻量同步 vs 引入 Airbyte/DataX | 轻量场景内置；重量级数据集成需外接，不作为强依赖 |
