# 智能运营数据中台 - 整体架构设计

## 一、系统定位

**一站式智能运营平台**，整合数仓查询、提数对账、风控监控、飞书协作四大能力，面向：

- 👨‍💼 **运营人员** — 日常取数、业绩查看、对账核查
- 📊 **数据分析师** — 深度分析、自定义查询、报表制作
- 💰 **财务人员** — 财务报表、收支对账、清算核对
- 🛡️ **风控人员** — 风险监控、预警处理、黑名单管理
- 👔 **管理层** — 经营看板、关键指标、决策支持

### 1.1 与现有工程的关系

SuperSonic 已实现**数仓查询**的核心能力（NL2SQL、语义模板、Agent/Plugin、数据权限），是本方案的基座平台。提数对账、风控监控、飞书协作属于垂直业务扩展，需在现有架构之上新建模块或通过 MCP 工具层对接外部系统。

| 能力域 | SuperSonic 现状 | 实现路径 |
|--------|----------------|----------|
| 数仓查询 | **已实现** — NL2SQL + 模板 + Agent + 导出 | 直接复用，补充 CSV 导出和用户定时调度 |
| 提数对账 | **未实现** | 新建语义模板（对账域），或通过 MCP 工具对接业务系统 |
| 风控监控 | **未实现** | 新建语义模板（风控域），或通过 MCP 工具对接风控系统 |
| 飞书协作 | **未实现** | 新建 Plugin (WEB_SERVICE 类型) 或 MCP 工具对接飞书 API |
| 综合流程编排 | **未实现** | Dify Workflow 外部编排，或扩展 Agent 多工具链式调用 |

### 3.2 架构决策点

当前工程与设计方案存在几个关键技术路线差异，需在实施前做出决策：

| 决策项 | 设计方案 | 工程现状 | 建议 |
|--------|---------|---------|------|
| LLM 框架 | Spring AI | LangChain4j 0.36.2 | **保持 LangChain4j** — 已深度集成，迁移成本高且无功能收益 |
| 工具协议 | MCP (Model Context Protocol) | Agent + Plugin 自有体系 | **两者并行** — 内部用 Plugin，外部 Dify 对接用 MCP |
| 工作流引擎 | Dify Workflow | 无 (内置 Parser 链) | **Dify 为外部编排层** — 不替换内置 Parser，而是在外层套 Dify |
| 缓存 | Redis | Caffeine | **按需升级** — 单机用 Caffeine，集群部署时引入 Redis |
| 消息队列 | RabbitMQ | 无 | **按需引入** — 仅定时报表推送等异步场景才需要 |

### 4.1 数仓查询工具 (DataWarehouseTools) - 8个

| 工具 | 功能 | 使用场景 | 工程映射 | 状态 |
|------|------|---------|---------|------|
| `naturalLanguageQuery` | 自然语言转SQL查询 | 日常取数，最常用 | `NL2SQLParser` → `OnePassSCSqlGenStrategy` → `JdbcExecutor` | **已实现** |
| `executeQueryTemplate` | 执行预设模板 | 固定报表 | `SemanticTemplateServiceImpl.deployTemplate()` 部署后通过 Agent 查询 | **已实现** |
| `listQueryTemplates` | 查看可用模板 | 了解有哪些模板 | `SemanticTemplateController.listTemplates()` | **已实现** |
| `previewQuery` | 预览SQL不执行 | 确认SQL正确性 | `S2ChatLayerService` 的 explain 模式 | **已实现** |
| `explainTable` | 查看表结构字典 | 了解数据结构 | `SchemaService.fetchSemanticSchema()` + `DimensionService` + `MetricService` | **已实现** |
| `dataQA` | 数据知识问答 | 了解指标含义 | `TermController` (术语管理) + Agent 智能问答 | **已实现** |
| `exportQueryResult` | 导出Excel/CSV | 数据导出 | `DownloadServiceImpl.downloadByStruct()` (EasyExcel) | **部分** — 仅 Excel 同步导出，无 CSV，无异步导出机制 |
| `scheduleQuery` | 创建定时任务 | 定期报表 | Spring `@Scheduled` 框架已有 | **Gap** — 无用户自定义调度 |

**实现说明**：

`naturalLanguageQuery` 的完整调用链：
```
用户问题 → ChatQueryController.chat()
  → ChatQueryServiceImpl.performParsing()
    → NL2SQLParser（规则解析 STRICT→MODERATE→LOOSE）
    → LLMSqlParser（LLM 解析 + 自一致性投票）
      → OnePassSCSqlGenStrategy.generate()
      → LLMSqlCorrector.correct()
  → ChatQueryServiceImpl.performExecution()
    → JdbcExecutor.execute() → 返回 QueryResult
```

`executeQueryTemplate` 通过语义模板实现：
```
SemanticTemplateConfig 定义 → deployTemplate() 创建 Domain/Model/Metric/DataSet
  → TemplateDeployedEvent 事件 → TemplateDeployedEventListener 自动创建 Agent + Plugin
  → 用户通过 Agent 查询已部署模板的数据
```

### 4.2 提数对账工具 (DataReconciliationTools) - 6个

| 工具 | 功能 | 使用场景 | 工程映射 | 状态 |
|------|------|---------|---------|------|
| `queryTransactionFlow` | 查询交易流水 | 交易明细查询 | — | **Gap** |
| `queryAccountBalance` | 查询账户余额 | 余额核对 | — | **Gap** |
| `querySettlementData` | 查询清算数据 | 清算核对 | — | **Gap** |
| `reconcileData` | 执行对账比对 | 日终对账 | — | **Gap** |
| `analyzeDifferences` | 分析对账差异 | 差异排查 | — | **Gap** |
| `generateReconciliationReport` | 生成对账报表 | 报表归档 | — | **Gap** |

**实现路径分析**：

提数对账的 6 个工具本质上是**对交易/清算数据源的结构化查询 + 比对逻辑**。有两种实现路径：

- **路径 A — 语义模板**：将交易流水、账户余额、清算数据建模为语义模板（Domain → Model → Metric → DataSet），通过 NL2SQL 引擎查询。对账比对和差异分析则作为 Plugin (WEB_SERVICE 类型) 调用后端对账 API。
  - 优势：复用已有 NL2SQL + 权限引擎，无需开发新查询层
  - 前提：交易/清算数据在可 JDBC 访问的数据源中

- **路径 B — MCP 工具**：每个工具实现为独立的 MCP Tool，直接调用业务系统 API（如交易中台、清算系统）。
  - 优势：不依赖数据源 JDBC 接入，适合调用已有微服务 API
  - 前提：业务系统已提供 REST API

**实现路径分析**：

飞书 7 个工具全部是对[飞书开放平台 API](https://open.feishu.cn/document/) 的封装。实现方式：

- **Plugin (WEB_SERVICE)**：将每个飞书 API 封装为 `ChatPlugin`（类型 `WEB_SERVICE`），通过现有 `PluginManager` + `WebServiceQuery` 调用。这是最轻量的接入方式，不需要新框架。
- **MCP Tool**：如需对接 Dify Workflow，则封装为 MCP Tool 供 Dify Agent 调用。

需要配置飞书应用凭证（App ID / App Secret），建议通过 `SystemConfig` 统一管理。

### 4.5 通用连接器架构（Plugin 基础设施）

为支持上述工具快速对接各类外部 HTTP 服务，设计**通用连接器架构**，作为 Plugin (WEB_SERVICE) 的增强实现。

#### 4.5.1 业界方案借鉴

| 产品 | 核心能力 | 借鉴点 |
|------|---------|--------|
| [n8n](https://n8n.io/) | 开源工作流自动化 | HTTP Request 节点配置模型、重试机制、数据转换 |

| [Zapier](https://zapier.com/) | 商业工作流自动化 | 预置连接器市场、OAuth 管理 |

#### 4.5.2 连接器配置模型

```
ConnectorConfig
├── 基础配置
│   ├── name              # 连接器名称
│   ├── type              # HTTP_REQUEST / GRAPHQL / WEBHOOK / INTERNAL
│   └── description       # 描述
├── 请求配置
│   ├── url               # URL（支持 {{placeholder}} 占位符）
│   ├── method            # GET / POST / PUT / DELETE
│   ├── contentType       # application/json / x-www-form-urlencoded
│   ├── headers           # 自定义请求头
│   ├── queryParams       # URL 查询参数
│   ├── bodyParams        # 请求体参数
│   └── bodyTemplate      # 请求体模板（JSON）
├── 认证配置 (AuthConfig)
│   ├── NONE              # 无认证
│   ├── API_KEY           # API Key（Header 或 Query）
│   ├── BASIC             # HTTP Basic Auth
│   ├── BEARER            # Bearer Token
│   └── OAUTH2            # OAuth2 客户端凭证
├── 响应处理 (ResponseConfig)
│   ├── dataPath          # JSONPath 提取数据
│   ├── errorPath         # JSONPath 提取错误信息
│   ├── successCodes      # 成功状态码
│   └── fieldMappings     # 字段映射
└── 高级配置
    ├── timeoutMs         # 超时（默认 30s）
    ├── retryCount        # 重试次数（默认 2）
    └── retryDelayMs      # 重试间隔（默认 1s）
```

**关键实现说明**：

行级权限注入流程（对应设计方案 `row_filters` 配置）：
```
S2DataPermissionAspect (@Around)
  → 获取当前用户维度权限（DimensionFilter 列表）
  → QueryStructReq: 注入 Filter(FilterOperatorEnum.SQL_PART)
  → QuerySqlReq: SqlAddHelper.addWhere() 修改 SQL AST
  → 多个维度过滤条件以 OR 连接
```

设计方案中的 `row_filters` 配置可映射为 SuperSonic 的**维度权限配置**（DataSet 级别为每个角色设置维度过滤条件）。`sensitive_columns` 映射为**敏感字段标记**（Dimension/Metric 的 `sensitiveLevel` 属性）。

| **Collection 组织** | 文件夹式问题/仪表盘管理 | SemanticTemplate + Domain 层级管理 | 差异化：SuperSonic 按语义域组织，Metabase 按用户目录组织 |
| **公开分享** | 一键生成公开链接 | 无公开分享机制 | **Gap** — 需实现公开链接 + Token 鉴权 |
| **管理后台** | 用户/组/权限/审计日志/使用统计 | 多租户 RBAC + 审计日志（部分） | 基本对齐，统计分析模块待完善 |
| **语义层** | 基础 Model 定义（字段描述/类型/关系） | 完整语义建模（Dimension/Metric/DataSet） + LLM 微调 | **SuperSonic 领先**：语义层是核心竞争力 |
| **NL2SQL** | 无（仅有简单问答） | 完整 NL2SQL 引擎 + 自一致性 + 纠错 | **SuperSonic 领先** |
| **多租户** | 无原生多租户 | 完整多租户隔离（TenantInterceptor + TenantSqlInterceptor） | **SuperSonic 领先** |

### 10.3 差异化优势保持

在借鉴 Metabase 功能的同时，需保持 SuperSonic 的核心差异化优势：

| 优势领域 | 说明 | 保持策略 |
|---------|------|---------|
| **NL2SQL 引擎** | 自然语言 → SQL 的核心能力，Metabase 不具备 | 持续优化 Parser 链、Exemplar 检索、自一致性策略 |
| **语义建模** | 完整的 Domain → Model → Dimension/Metric → DataSet 层级 | 作为所有功能的基础，Dashboard/告警/分享都基于语义层 |
| **多租户隔离** | 原生 SaaS 架构，数据物理隔离 | Metabase 开源版无此能力，是企业级部署的关键 |
| **LLM 集成** | 与 Dify / LangChain4j 深度集成，可编排复杂流程 | 作为智能运营平台的核心入口，不可替代 |

### 10.4 借鉴功能实现优先级

综合业务价值和实现成本，建议按以下顺序推进借鉴功能：

| 优先级 | 功能 | 业务价值 | 实现成本 | 备注 |
|--------|------|---------|---------|------|
| P0 | 告警订阅 | 高（主动推送，减少人工查询） | 中 | 复用现有调度框架 |
| P1 | Dashboard 仪表盘 | 高（管理层看板需求强烈） | 高 | 需新建前后端模块 |
| P1 | 公开分享 | 中（报表分发场景） | 低 | 仅需 Token + 路由 |
| P2 | 嵌入集成 | 中（第三方系统对接） | 中 | 视外部系统需求 |
| P3 | 使用统计增强 | 低（内部运营分析） | 低 | 可后期完善 |
