## 一、设计目标与定位

### 1.1 功能定位

飞书机器人作为智能运营数据中台的**轻量级移动查询入口**，与 Web 查询平台形成互补：

| 维度 | Web 查询平台 | 飞书机器人 |
|------|-------------|-----------|
| 使用场景 | 深度分析、报表制作、复杂查询 | 快捷取数、移动查询、即时通知 |
| 交互方式 | 表单/图表/拖拽 | 自然语言对话 |
| 结果展示 | 完整图表 + 数据表格 | 消息卡片 + 文件导出 |
| 用户群体 | 数据分析师、运营专员（桌面端） | 全部角色（移动 + 桌面） |
| 入口位置 | 独立 Web 应用 | 飞书工作台 / 群聊 / 单聊 |

### 1.2 设计原则

- **复用优先**：不重建查询、导出、调度内核，优先复用 SuperSonic 与模板报表子系统已有能力
- **入口层定位**：飞书机器人是协议适配层，不承载平台级执行编排
- **Dify 外置**：Dify 仅作为外部可选编排层，简单查询不经过 Dify
- **权限一致**：飞书用户通过共享字段自动匹配或审核绑定为 SuperSonic 用户，复用已有的行级权限、字段脱敏、DataSet 访问控制
- **渐进扩展**：阶段 1 数仓查询 → 阶段 2 入口能力增强 → 阶段 3 外部复杂编排对接

## 三、核心消息流

### 3.1 自然语言查询主流程（简化）

```
① 飞书用户发消息 "查昨天北京GMV"
      ↓ Webhook 回调 / WebSocket
② FeishuEventController 验签 + 事件去重
      ↓ 提交到 feishuExecutor 线程池
③ FeishuBotService:
      → FeishuUserMappingService.resolveUser(open_id)   [身份映射]
      → 限流检查（20次/分）
      → FeishuMessageRouter.route(text)                  [路由]
④ QueryMessageHandler:
      → SuperSonicApiClient.getOrCreateChat(agentId)    [飞书专属会话]
      → SuperSonicApiClient.query(text, agentId, user)  [NL2SQL + 执行]
        内部: NL2SQLParser → S2DataPermissionAspect → JdbcExecutor
      → FollowUpHintGenerator.generate(parseInfo)       [追问提示]
      → FeishuCardRenderer.renderQueryResult(result)    [格式化]
      → FeishuMessageSender.replyCard(messageId, card)  [回复卡片]
```

详见 [消息处理链](details/feishu/03-message-handlers.md) 和 [内部 API 客户端](details/feishu/04-api-client.md)。

### 3.2 消息路由规则

| 消息模式 | 示例 | 处理器 |
|---------|------|--------|
| `/sql <文本>` | `/sql 查昨天GMV` | `PreviewMessageHandler`（仅生成 SQL，不执行） |
| `/use <id>` | `/use 3` | `UseAgentHandler`（切换默认 Agent） |
| `/help` / `帮助` | `/help` | `HelpMessageHandler`（动态帮助卡片） |
| `导出` / `/export` / `下载` | `导出` | `ExportMessageHandler`（SQL → CSV → 文件） |
| `/template` / `模板` | `/template` | `TemplateListHandler`（Agent 列表） |
| `/history` | `/history` | `HistoryMessageHandler`（最近 10 条） |
| 其他 | `查昨天北京GMV` | `QueryMessageHandler`（默认，NL2SQL） |

## 四、身份映射策略

详见 [身份映射详细设计](details/feishu/02-identity-mapping.md)。

### 4.1 方案对比

| 方案 | 适用场景 | 实施阶段 |
|------|---------|---------|
| **A. 管理员手动映射** | 用户少，试点期 | 阶段 1（兜底） |
| **B. 共享字段自动匹配** | 已有统一身份标识（工号/邮箱） | 阶段 1（主方案） |
| **C. 飞书 OAuth 自助绑定** | 自动匹配失败的可选兜底 | 阶段 2（可选，不建议默认） |
| **D. 企业通讯录同步** | 全员覆盖，统一管控 | 阶段 2（可选） |

**推荐策略**：B（共享字段自动匹配）+ A（管理员手动兜底）。

### 4.2 自动匹配流程（简化）

```
收到 open_id
  → 查 s2_feishu_user_mapping
    → 已存在 + 启用 → 返回 SuperSonic 用户
    → 不存在 → 调飞书通讯录 API 获取 employee_id / email / phone
              → 按优先级匹配 s2_user 表
                → 成功 → 创建映射（AUTO_*），继续查询
                → 失败 → 创建 PENDING 记录，提示联系管理员
```

## 六、非功能需求

### 6.1 性能要求

| 指标 | 目标值 |
|------|-------|
| 消息接收 → 回复延迟（简单查询） | ≤ 5s |
| 消息接收 → 回复延迟（复杂查询） | ≤ 15s |
| 飞书 API 调用成功率 | ≥ 99.5%（含重试机制） |
| 并发查询支持 | ≥ 50 QPS（受数仓连接池限制） |

### 6.2 安全要求

| 安全项 | 措施 |
|-------|------|
| Webhook 验签 | HMAC-SHA256（`encryptKey` 配置后强制校验，timing-safe 比较） |
| Token 安全 | App Secret / Encrypt Key 通过环境变量注入，禁止明文配置 |
| 数据权限 | 严格复用 SuperSonic 行级权限 + 字段脱敏，不可绕过 |
| 频率限制 | 单用户每分钟最多 20 次查询 |
| 审计日志 | 所有查询记录写入 `s2_feishu_query_session` 表 |

详见 [事件接收与验签](details/feishu/01-event-receiving.md) 和 [基础设施详细设计](details/feishu/06-infra.md)。

### 6.3 监控指标

| 指标名 | 告警阈值 |
|--------|---------|
| `feishu.query.duration{status=error}` 错误率 | > 5% |
| `feishu.query.duration` p95 延迟 | > 10s |
| `feishu.mapping.pending`（PENDING 积压数） | > 10 |
| `feishu.api.call{api=token, status=error}` 失败 | 连续 2 次 |

### 阶段 1 验收标准

- 飞书单聊发送自然语言查询，5 秒内收到结果卡片
- 飞书群聊 @机器人 发送查询，结果回复到群聊并 @提问者
- 查询结果以表格卡片展示，支持 ≤20 行数据展示
- 发送"导出"，收到最近一次查询的 CSV 文件
- 未绑定用户首次发消息，系统自动通过工号/邮箱匹配
- 自动匹配失败的用户收到引导绑定的提示卡片
- 管理员可手动创建/修改/禁用用户映射
- Webhook `verification-token` 验证正确，伪造请求被拒绝
- 单用户每分钟超过 20 次查询时被限流
- 查询记录正确写入 `s2_feishu_query_session` 表
