# 飞书机器人用户交互层设计方案

> 本文档是《智能运营数据中台 - 整体架构设计》的补充设计，聚焦用户交互层中**飞书机器人**的技术实现。阶段 1 优先实现数仓查询交互，后续阶段逐步扩展至提数对账、风控监控、定时推送等场景。

## 一、设计目标与定位

### 1.1 功能定位

飞书机器人作为智能运营数据中台的**轻量级移动查询入口**，与 Web 查询平台形成互补：

| 维度     | Web 查询平台                   | 飞书机器人                   |
| -------- | ------------------------------ | ---------------------------- |
| 使用场景 | 深度分析、报表制作、复杂查询   | 快捷取数、移动查询、即时通知 |
| 交互方式 | 表单/图表/拖拽                 | 自然语言对话                 |
| 结果展示 | 完整图表 + 数据表格            | 消息卡片 + 文件导出          |
| 用户群体 | 数据分析师、运营专员（桌面端） | 全部角色（移动 + 桌面）      |
| 入口位置 | 独立 Web 应用                  | 飞书工作台 / 群聊 / 单聊     |

### 1.2 架构位置

### 1.3 设计原则

- **复用优先** — 不重建查询引擎，直接调用 SuperSonic 已有的 `ChatQueryServiceImpl` 完整链路
- **最小依赖** — 阶段 1 不引入 MCP / Dify / 消息队列，飞书机器人直接调用 Java Service 层
- **权限一致** — 飞书用户通过共享字段（工号/邮箱/手机号）自动匹配为 SuperSonic 用户，复用已有的行级权限、字段脱敏、DataSet 访问控制
- **渐进扩展** — 阶段 1 数仓查询 → 阶段 2 入口能力增强（多 Agent/追问/导出/OAuth）→ 阶段 3 Dify 编排 + 高级交互

#### 3.4.3 身份映射 — FeishuUserMappingService

飞书用户到 SuperSonic 用户的映射是权限体系正确运作的前提。账号打通方案需要兼顾实现成本、运维成本和用户体验。

##### 3.4.3.1 账号打通方案对比

| 方案                       | 适用场景                      | 运维成本               | 用户体验               | 实现复杂度 |
| -------------------------- | ----------------------------- | ---------------------- | ---------------------- | ---------- |
| **A. 管理员手动映射**      | 用户 < 50，试点期             | 高（每次新增都要操作） | 差（需联系管理员）     | 低         |
| **B. 共享字段自动匹配**    | 已有统一身份标识（工号/邮箱） | 低                     | 好（首次自动匹配）     | 低         |
| **C. 飞书 OAuth 自助绑定** | 正式推广，自动匹配失败的兜底  | 低                     | 好（一次授权永久绑定） | 中         |
| **D. 企业通讯录同步**      | 全员覆盖，统一管控            | 最低                   | 最好（零操作）         | 中高       |

**推荐策略**：阶段 1 采用 **B（共享字段自动匹配）+ A（管理员手动兜底）**，阶段 2 升级到 **C（OAuth 自助绑定）+ D（通讯录同步）**。

##### 3.4.3.2 阶段 1 — 共享字段自动匹配（主方案）

核心思路：飞书和 SuperSonic 的用户大概率共享**工号、手机号或企业邮箱**中的至少一个。用户首次给机器人发消息时，系统自动从飞书通讯录获取用户信息，与 SuperSonic 用户表进行匹配。

**自动匹配流程**：

```
用户首次给机器人发消息
  → 飞书回调携带 open_id
  → 查询 s2_feishu_user_mapping 表
    → 已存在映射：直接返回 SuperSonic 用户身份
    → 不存在映射：
        → 调用飞书通讯录 API (GET /contact/v3/users/{user_id})
        → 获取用户信息：employee_id / phone / email
        → 按优先级匹配 s2_user 表：
            ① employee_id（工号）  — 最可靠，企业内唯一（s2_user 需新增此字段）
            ② email（企业邮箱）    — 常见，格式统一（s2_user.email 已存在）
            ③ phone（手机号）      — 常见，但需注意隐私（s2_user.phone 已存在）
        → 匹配成功：自动创建映射记录，返回用户身份
        → 全部匹配失败：发卡片引导联系管理员手动绑定
```

**前提条件**：SuperSonic 的 `s2_user` 表中需包含上述匹配字段中的至少一个。当前字段现状：

| 匹配字段 | s2_user 列名 | 状态 |
|---------|-------------|------|
| 工号 | `employee_id` | ✅ 已新增（V20 迁移 + `UserDO.employeeId` 字段） |
| 企业邮箱 | `email` | ✅ 已存在 |
| 手机号 | `phone` | ✅ 已存在 |

`employee_id` 字段已通过 V20 迁移脚本添加：

```sql
-- MySQL (V20__feishu_bot.sql)
ALTER TABLE s2_user ADD COLUMN IF NOT EXISTS employee_id VARCHAR(64) DEFAULT NULL;
-- PostgreSQL (V20__feishu_bot.sql)
DO $$ BEGIN
    IF NOT EXISTS (SELECT 1 FROM information_schema.columns WHERE table_name = 's2_user' AND column_name = 'employee_id') THEN
        ALTER TABLE s2_user ADD COLUMN employee_id VARCHAR(64) DEFAULT NULL;
    END IF;
END $$;
```

**匹配配置**（通过 `application.yaml` 控制匹配策略）：

```yaml
feishu:
  user-mapping:
    auto-match-enabled: true                 # 是否启用自动匹配
    match-fields:                            # 匹配字段优先级（按顺序尝试，对应 s2_user 列名）
      - employee_id                          # s2_user.employee_id（需新增）
      - email                                # s2_user.email（已存在）
      - phone                                # s2_user.phone（已存在）
    auto-create-user: false                  # 匹配全部失败时是否自动创建 SuperSonic 用户
    default-role: VIEWER                     # 自动创建用户的默认角色
```

##### 3.4.3.3 阶段 2 — 飞书 OAuth 自助绑定（兜底方案）

针对自动匹配失败的用户，提供 OAuth 自助绑定能力，无需管理员介入：

```
自动匹配失败
  → 发飞书卡片："未能自动识别您的数据平台账号，请点击绑定"
  → 用户点击卡片按钮
  → 跳转飞书内置浏览器，打开绑定页（H5）
     ┌─────────────────────────────────┐
     │       绑定数据平台账号            │
     │                                 │
     │  账号: [___________________]    │
     │  密码: [___________________]    │
     │                                 │
     │       [ 确认绑定 ]              │
     └─────────────────────────────────┘
  → 用户输入 SuperSonic 账号密码
  → 后端验证通过 → 写入 s2_feishu_user_mapping
  → 返回飞书："✅ 绑定成功，现在可以开始查询了"
```

绑定页面通过飞书卡片的 URL 按钮触发，技术实现为一个轻量 H5 页面，无需额外前端框架。

> **详细设计**见 [9.7 节 — OAuth 自助绑定](#97-oauth-自助绑定--飞书用户自主关联-supersonic-账号)，包含 bindToken 安全设计、API 接口、H5 页面规格、FeishuBotService 改造方案、安全性分析和边界处理。

##### 3.4.3.6 完整映射流程

```
收到飞书消息 (open_id)
  │
  ▼
查询 s2_feishu_user_mapping
  │
  ├─ 映射存在 + status=1 ──────────► 返回 SuperSonic 用户身份 ──► 继续查询
  │
  ├─ 映射存在 + status=0 ──────────► 返回提示卡片："您的查询权限已被暂停"
  │
  └─ 映射不存在
      │
      ▼
    auto-match-enabled = true ?
      │
      ├─ 是：调用飞书通讯录 API 获取用户信息
      │   │
      │   ▼
      │  按 match-fields 优先级依次匹配 s2_user 表
      │   │
      │   ├─ 匹配成功 ──► 创建映射（match_type=AUTO_*） ──► 继续查询
      │   │
      │   └─ 全部失败
      │       │
      │       ▼
      │     创建 PENDING 映射记录（status=0, matchType=PENDING）
      │     自动填充飞书联系人信息（姓名/邮箱/手机/工号）
      │       │
      │       └─ 返回提示："已自动提交映射申请，请联系管理员审核"
      │          管理员在 Web 管理页编辑关联 s2_user → 启用
      │
      └─ 否 ──► 创建 PENDING 映射记录 + 返回提示卡片
```

### 4.2 与 Dify 编排的关系

飞书机器人作为用户入口，与 Dify 工作流编排是**两个独立层面**：

```
┌────────────────────────────────────────────────────────────────────────┐
│  飞书机器人（本文档）— 用户入口，协议适配层                              │
│                                                                        │
│  飞书用户 → 飞书机器人 → SuperSonic Service（直联，<5s 响应）           │
│                       ↘ Dify Workflow（仅复杂多步骤流程，可选）          │
└────────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────────┐
│  MCP Server（整体架构文档）— 平台级工具注册中心                         │
│                                                                        │
│  Dify Workflow → MCP Feishu Tools（发消息/发审批/上传文档 — 输出端）    │
│               → MCP DataWarehouse Tools（NL2SQL/模板查询 — 查询端）     │
└────────────────────────────────────────────────────────────────────────┘
```

**设计原则**：
- **简单查询保持直联** — 用户通过飞书发起数据查询（提数工单），飞书机器人直接调用 SuperSonic Service，保持 <5s 响应，不经过 Dify
- **复杂流程可选路由** — 若识别到多步骤意图（如"对账完发通知"），可转发到 Dify Workflow 编排，但这是阶段 3 的能力
- **MCP 工具归平台** — 飞书 API 的 MCP 封装（`sendFeishuMessage`、`createFeishuApproval` 等）由平台级 `mcp/` 模块统一注册，供 Dify Workflow 调用。飞书机器人模块通过 `FeishuMessageSender` 暴露发送能力，MCP 层可复用

### 5.1 Dify 编排对接

```
飞书用户发送消息
  │
  ├─ 简单查询（"查一下昨天的GMV"）
  │   → 直联 SuperSonic Service（阶段 1/2 已实现，<5s 响应）
  │

        ② analyzeDifferences     → 分析差异（MCP Reconciliation Tool）

        ⑤ sendFeishuCardMessage  → 通知运营群（MCP Feishu Tool）
        ⑥ createFeishuApproval   → 如有重大差异，发起审批（MCP Feishu Tool）
      → Dify 返回最终结果 → 飞书机器人回复用户
```

**实现要点**：
- 意图路由：飞书机器人新增 `DifyRoutingHandler`，通过关键词/LLM 判断是否为多步骤意图
- Dify 对接：通过 Dify Chat API（`POST /v1/chat-messages`）发送用户消息，流式接收结果
- MCP Feishu Tools 由平台级 `mcp/` 模块提供（见《智能运营数据中台设计方案》4.4 节），飞书机器人模块不重复实现

## 九、补充设计 — 架构修正与未完成功能

> 以下为阶段 1 实现过程中发现的架构问题和未完成功能的详细设计方案，补充于 2026-02-11。

### 9.0 架构修正 — 去除 chat-server 直接依赖

#### 9.0.1 问题

当前 `feishu-server` 的 pom.xml 直接依赖 `chat-server`，导致传递依赖链：

```
feishu-server → chat-server → headless-server → headless-core → headless-chat → ...
```

飞书模块本质上是**协议适配层**（飞书协议 ↔ SuperSonic API），与 Web 前端的角色相同，不应直接耦合后端 Service 层。直接依赖带来的问题：

- **编译耦合**：chat-server 的任何内部变更都可能导致 feishu-server 需要重新编译
- **无法独立部署**：feishu 模块必须和完整后端在同一 JVM 中
- **职责混淆**：绕过了 REST API 的认证/租户/审计拦截链路

#### 9.0.2 修正方案

`feishu-server` 改为通过 HTTP 调用 SuperSonic 已有的 REST API（与前端走同一链路），Maven 依赖缩减为：

```xml
<!-- 修正后的 feishu-server 依赖（仅保留轻量模块） -->
<dependencies>
    <dependency>feishu-api</dependency>     <!-- 自身 API POJOs -->
    <dependency>common</dependency>          <!-- User, TenantContext 等共享类型 -->
    <dependency>auth-api</dependency>        <!-- @AuthenticationIgnore 注解 -->
    <dependency>caffeine</dependency>        <!-- 缓存 -->
    <dependency>spring-boot-starter-data-redis (optional)</dependency>
</dependencies>

<!-- 删除 -->
<!-- <dependency>chat-server</dependency> -->
```

#### 9.0.4 内部 API 调用方式

飞书模块通过 `RestTemplate` 调用本机 REST API，需携带映射用户的认证信息：

```java
@Service
public class SuperSonicApiClient {

    private final RestTemplate restTemplate = new RestTemplate();
    private final InternalTokenGenerator tokenGenerator;  // auth-api 接口

    @Value("${server.port:9080}")
    private int serverPort;

    private String baseUrl() {
        return "http://localhost:" + serverPort;
    }

    private HttpHeaders buildHeaders(User user) {
        HttpHeaders headers = new HttpHeaders();
        headers.setContentType(MediaType.APPLICATION_JSON);
        String token = tokenGenerator.generateInternalToken(user, 60_000L);
        headers.setBearerAuth(token);
        if (user.getTenantId() != null) {
            headers.set("X-Tenant-Id", String.valueOf(user.getTenantId()));
        }
        return headers;
    }
}
```

> **为什么不直接传递 User 对象**：REST API 端点通过 `UserHolder.findUser(request)` 从 JWT Token 中提取用户身份。飞书模块以映射用户的身份生成 Token，确保请求经过完整的认证拦截器链路（`TenantInterceptor` → `AuthenticationInterceptor`），权限和租户隔离与 Web 端一致。

#### 9.0.5 SuperSonicApiClient 核心方法

> **重要实现细节**（联调中发现）：
> 1. **ResponseAdvice 包装**：SuperSonic 的 `ResponseAdvice` 将所有 REST 响应包装为 `ResultData{code, data}`，内部 HTTP 调用必须反序列化为 `ResultData<T>` 再提取 `.getData()`
> 2. **Accept 头**：classpath 存在 `jackson-dataformat-xml`，不指定 `Accept: application/json` 时服务端可能返回 XML
> 3. **Chat 会话隔离**：飞书查询需要 `chatId`，但不应复用 Web 端创建的会话。通过 `chatName=飞书助手` 做会话隔离，按 `chatName` 过滤查找已有会话
> 4. **ChatController.getAll 扩展**：新增可选参数 `chatName`，SQL 层加 `<if test='chatName != null'>AND chat_name = #{chatName}</if>`

```java
@Component
@Slf4j
public class SuperSonicApiClient {

    private static final String FEISHU_CHAT_NAME = "飞书助手";

    private final InternalTokenGenerator tokenGenerator;
    private final FeishuProperties properties;
    private final RestTemplate restTemplate;
    /** Cache: "chatId:userId:agentId" → chatId，已改用 FeishuCacheService 抽象层（30min TTL + maxSize 限制） */
    private final FeishuCacheService cacheService;  // ✅ 已修复：替代原 ConcurrentHashMap 无界缓存

    /**
     * NL2SQL 查询（解析 + 执行）。携带 chatId 以支持多轮对话上下文。
     * 对应 POST /api/chat/query/
     */
    public QueryResult query(String queryText, int agentId, User user) {
        Integer chatId = getOrCreateChat(agentId, user);
        ChatParseReq req = ChatParseReq.builder()
                .queryText(queryText).agentId(agentId).chatId(chatId).build();
        ResponseEntity<ResultData<QueryResult>> response = restTemplate.exchange(
                url, HttpMethod.POST, new HttpEntity<>(req, buildHeaders(user)),
                new ParameterizedTypeReference<ResultData<QueryResult>>() {});
        return extractData(response);
    }

    /**
     * 查找或创建飞书专属 Chat 会话。
     * 1. 先通过 getAll?chatName=飞书助手 查找已有会话（不与 Web 会话混用）
     * 2. 找不到则创建新会话
     * 3. 结果缓存在内存中，应用重启后首次访问重新查找
     */
    private Integer findOrCreateChat(int agentId, User user) {
        // 1. 按 chatName 过滤查找
        String getAllUrl = baseUrl + "/api/chat/manage/getAll?agentId=" + agentId
                + "&chatName=" + FEISHU_CHAT_NAME;
        List<Map<String, Object>> chats = extractData(restTemplate.exchange(...));
        if (chats != null && !chats.isEmpty()) {
            return ((Number) chats.getFirst().get("chatId")).intValue();  // 复用已有
        }
        // 2. 创建新会话
        String saveUrl = baseUrl + "/api/chat/manage/save?chatName=" + FEISHU_CHAT_NAME + "&agentId=" + agentId;
        Long chatId = extractData(restTemplate.exchange(...));
        return chatId.intValue();
    }

    /** 解包 ResultData 响应 */
    private <T> T extractData(ResponseEntity<ResultData<T>> response) {
        ResultData<T> body = response.getBody();
        if (body == null) return null;
        if (body.getCode() != 200) throw new RuntimeException("API error: " + body.getMsg());
        return body.getData();
    }

    private HttpHeaders buildHeaders(User user) {
        HttpHeaders headers = new HttpHeaders();
        headers.setContentType(MediaType.APPLICATION_JSON);
        headers.setAccept(List.of(MediaType.APPLICATION_JSON));  // 防止返回 XML
        headers.set("Authorization", "Bearer " + tokenGenerator.generateToken(user));
        if (user.getTenantId() != null) {
            headers.set("X-Tenant-Id", String.valueOf(user.getTenantId()));
        }
        return headers;
    }
}
```

当前 `FeishuUserMappingService` 通过 `JdbcTemplate` 直接查 `s2_user` 表做自动匹配。改为调用 REST API：

```java
// 改造前
```

#### 9.0.9 返回值类型处理

改为 HTTP 调用后，`QueryResult`、`ChatParseResp` 等响应类仍在 `chat-api`、`headless-api` 中定义。有两种方案：

| 方案 | 做法 | 优缺点 |
|------|------|--------|
| **A. 依赖 API 模块** | `feishu-server` 依赖 `chat-api`、`headless-api`（仅 POJO，不含业务逻辑） | 类型安全，IDE 友好；但增加了编译依赖 |
| **B. 用 Map/JsonNode** | 不依赖任何 API 模块，用 `Map<String, Object>` 或 `JsonNode` 接收响应 | 零耦合；但丢失类型信息，代码可读性下降 |

#### 9.1.1 实际实现方案

> **设计变更说明**：原设计方案通过重新执行 NL2SQL 查询获取全量数据。实际实现改为**直接用存储的 SQL 查询语义层**，原因：

**实际依赖链**：`ExportMessageHandler` → `SuperSonicApiClient.queryBySql()` → `POST /api/semantic/query/sql`

### 9.3 SQL 预览 — PreviewMessageHandler（✅ 已实现）

#### 9.3.1 设计思路

用户发送 `/sql <查询文本>` 时，仅执行 NL2SQL 解析生成 SQL，不实际执行查询。用于确认 SQL 正确性后再执行。

> **实现状态**：已完成。实际实现与设计基本一致，关键差异：使用 `msg.getAgentId()`（用户级 Agent 路由）替代 `properties.getDefaultAgentId()`；`FeishuCardRenderer.renderSqlPreviewCard()` 中 SQL 优先级为 `correctedS2SQL > parsedS2SQL > querySQL`。

#### 9.3.2 流程

```
用户发送 "/sql 查昨天北京的GMV"
  │
  ▼
提取查询文本（去除 "/sql " 前缀）
  │
  ▼

  │

  │
  └─ state=COMPLETED
      │
      ▼
    取 selectedParses[0]（最高置信度）
      │
      ▼
    渲染 SQL 预览卡片：
      Header: "🔍 SQL 预览"
      Body:
        - 查询意图: parse.textInfo
        - 数据集: parse.dataSet.name
        - 生成 SQL（代码块格式）: parse.sqlInfo.sql
        - 置信度: parse.score
      Footer: "直接发送查询文本可执行此 SQL"
```

#### 9.3.3 实现要点

```java
@Component
@Slf4j
@RequiredArgsConstructor
public class PreviewMessageHandler implements MessageHandler {

    private final SuperSonicApiClient apiClient;
    private final FeishuCardRenderer cardRenderer;
    private final FeishuMessageSender messageSender;
    private final FeishuProperties properties;

    @Override
    public void handle(FeishuMessage msg, User user) {
        // 去除 "/sql " 前缀
        String queryText = msg.getContent().replaceFirst("(?i)/sql\\s+", "");
        if (queryText.isBlank()) {
            messageSender.replyText(msg.getMessageId(), "请输入查询内容，例如: /sql 查昨天北京的GMV");
            return;
        }

        // 通过 HTTP API 仅解析不执行
        ChatParseResp resp = apiClient.parse(queryText, properties.getDefaultAgentId(), user);

        if (resp.getState() == ParseState.FAILED || resp.getSelectedParses().isEmpty()) {
            messageSender.replyCard(msg.getMessageId(),
                    cardRenderer.renderErrorCard("无法解析查询: " + resp.getErrorMsg()));
            return;
        }

        SemanticParseInfo topParse = resp.getSelectedParses().get(0);
        Map<String, Object> card = cardRenderer.renderSqlPreviewCard(queryText, topParse);
        messageSender.replyCard(msg.getMessageId(), card);
    }
}
```

### 9.4 频率限制（✅ 已实现）

#### 9.4.1 设计思路

防止单用户刷屏，限制每用户每分钟最多 20 次查询请求。在 `FeishuCacheService` 接口层实现，Caffeine 和 Redis 各自提供实现。

> **实现状态**：已完成。实际实现使用**固定窗口计数器**（`incrementCounter()` 返回递增值）替代设计中的滑动窗口，接口更简洁。Caffeine 用 `counterCache.asMap().merge(key, 1L, Long::sum)`（60s TTL），Redis 用 `INCR` + 条件 `EXPIRE 60s`。限流配置通过 `FeishuProperties.RateLimitConfig`（enabled + maxRequests）支持运行时调整。

#### 9.4.2 接口扩展

```java

public interface FeishuCacheService {

    /**
     * 检查用户是否被限流。
     * 如果未超限，增加计数并返回 false；如果已超限，返回 true。
     * @param openId 飞书用户 ID
     * @param maxPerMinute 每分钟最大请求数
     */
    boolean isRateLimited(String openId, int maxPerMinute);
```

#### 9.4.3 Caffeine 实现

```java
// CaffeineFeishuCacheService 新增

// 滑动窗口计数器：key=openId, value=最近 1 分钟的请求时间戳列表
private final Cache<String, Deque<Long>> rateLimitCache = Caffeine.newBuilder()
        .expireAfterAccess(Duration.ofMinutes(2))
        .maximumSize(10000)
        .build();

@Override
public boolean isRateLimited(String openId, int maxPerMinute) {
    long now = System.currentTimeMillis();
    long windowStart = now - 60_000;

    Deque<Long> timestamps = rateLimitCache.get(openId, k -> new ConcurrentLinkedDeque<>());
    // 清理过期时间戳
    while (!timestamps.isEmpty() && timestamps.peekFirst() < windowStart) {
        timestamps.pollFirst();
    }
    if (timestamps.size() >= maxPerMinute) {
        return true;  // 超限
    }
    timestamps.addLast(now);
    return false;
}
```

#### 9.4.4 Redis 实现

```java
// RedisFeishuCacheService 新增

private static final String RATE_KEY_PREFIX = "feishu:rate:";

@Override
public boolean isRateLimited(String openId, int maxPerMinute) {
    String key = RATE_KEY_PREFIX + openId;
    // INCR + EXPIRE 原子操作
    Long count = redisTemplate.opsForValue().increment(key);
    if (count != null && count == 1) {
        // 首次请求，设置 60s 过期
        redisTemplate.expire(key, 60, TimeUnit.SECONDS);
    }
    return count != null && count > maxPerMinute;
}
```

> **Redis 方案说明**：使用简单计数器 + TTL，比滑动窗口简单但存在边界突发问题（窗口交界处可能短时间达到 2x 上限）。对于飞书机器人场景（20 次/分）可以接受。如需精确滑动窗口可改用 Redis Sorted Set。

### 9.5 HMAC-SHA256 Webhook 签名验证

#### 9.5.1 设计思路

当前 `FeishuEventController` 仅验证 `verification-token`（事件头中的 `token` 字段等于配置值）。飞书开放平台还支持更安全的 **Encrypt Key 签名验证**：飞书在推送事件时，使用 `encrypt_key` 对请求体计算 HMAC-SHA256 签名，放在 HTTP Header `X-Lark-Signature` 中。

#### 9.5.2 签名算法

飞书的签名计算方式：

```
签名输入 = timestamp + nonce + encrypt_key + request_body
签名结果 = SHA256(签名输入)

其中:
  timestamp = HTTP Header "X-Lark-Request-Timestamp"
  nonce     = HTTP Header "X-Lark-Request-Nonce"
  encrypt_key = 飞书应用配置中的 Encrypt Key（对应 s2.feishu.encrypt-key）
  request_body = 原始 HTTP 请求体字符串
```

> **注意**：此签名算法与 `FeishuDeliveryChannel.generateSign()` 中的**出站 Webhook 签名**不同。出站签名用于飞书自定义机器人的 Webhook URL，算法为 `HMAC-SHA256(timestamp + "\n" + secret, secret)`。入站事件验签算法为 `SHA256(timestamp + nonce + encrypt_key + body)`，是纯 SHA256 摘要而非 HMAC。

#### 9.5.3 实现方案

在 `FeishuEventController` 中增加签名校验逻辑：

```java
@PostMapping
@AuthenticationIgnore
public Object handleEvent(@RequestBody String rawBody,
                           @RequestHeader Map<String, String> headers) {

    // 1. URL 验证挑战（保持不变）
    // ...

    // 2. 签名验证（新增）
    String encryptKey = properties.getEncryptKey();
    if (StringUtils.isNotBlank(encryptKey)) {
        String timestamp = headers.get("x-lark-request-timestamp");
        String nonce = headers.get("x-lark-request-nonce");
        String signature = headers.get("x-lark-signature");

        if (signature == null || !verifySignature(timestamp, nonce, encryptKey, rawBody, signature)) {
            log.warn("HMAC-SHA256 signature verification failed");
            return Map.of("code", 0);
        }
    }

    // 3. 后续逻辑（解析 JSON、token 验证、去重、分发）
    Map<String, Object> body = objectMapper.readValue(rawBody, Map.class);
    // ...
}

/**
 * 验证飞书事件推送的签名。
 * 算法: SHA256(timestamp + nonce + encryptKey + body)
 */
private boolean verifySignature(String timestamp, String nonce,
                                 String encryptKey, String body, String expected) {
    try {
        String content = timestamp + nonce + encryptKey + body;
        MessageDigest digest = MessageDigest.getInstance("SHA-256");
        byte[] hash = digest.digest(content.getBytes(StandardCharsets.UTF_8));
        String calculated = bytesToHex(hash);
        return calculated.equals(expected);
    } catch (Exception e) {
        log.error("Signature verification error", e);
        return false;
    }
}

private static String bytesToHex(byte[] bytes) {
    StringBuilder sb = new StringBuilder(bytes.length * 2);
    for (byte b : bytes) {
        sb.append(String.format("%02x", b));
    }
    return sb.toString();
}
```

#### 9.5.4 Controller 方法签名变更

当前 `handleEvent` 接收 `@RequestBody Map<String, Object> body`（Spring 自动反序列化）。启用签名验证后需要访问**原始请求体字符串**，因此方法签名需改为接收 `String rawBody`，然后手动反序列化：

```java

// 变更后

                           @RequestHeader Map<String, String> headers)
```

- `encrypt-key` 和 `verification-token` 可同时启用（双重验证）
- 建议生产环境同时配置两项以获得最高安全性

### 9.7 OAuth 自助绑定 — 飞书用户自主关联 SuperSonic 账号

> 补充于 2026-02-28。阶段 2 功能，目标：自动匹配失败的用户无需管理员介入，通过飞书内点击卡片按钮 → H5 页面输入 SuperSonic 账号密码 → 后端验证通过 → 自动完成映射绑定。

#### 9.7.1 问题与动机

当前阶段 1 的身份映射策略为 **B（共享字段自动匹配）+ A（管理员手动兜底）**：

1. 用户首次发消息 → `FeishuUserMappingService.resolveMapping()` 尝试按 `employee_id` / `email` / `mobile` 自动匹配
2. 匹配失败 → 创建 PENDING 记录（`status=0, s2_user_id=NULL`）
3. 回复纯文本："未找到您的账号映射，已自动提交映射申请，请联系管理员..."
4. 管理员在 Web 管理页手动审核 → 选择 `s2_user_id` → 启用映射

**痛点**：
- 新用户无法自助完成绑定，必须等待管理员操作，延迟可达数小时至数天
- 管理员需要逐条手动匹配 SuperSonic 用户，当 PENDING 积压时工作量大
- 用户体验差：收到纯文本提示后无明确行动路径

**目标**：补充 **方案 C（OAuth 自助绑定）**，自动匹配失败时引导用户通过飞书卡片按钮打开 H5 绑定页，用户输入 SuperSonic 账号密码验证通过后自动完成映射，全程无需管理员介入。

#### 9.7.2 方案选择

飞书场景下的"OAuth 自助绑定"与标准 OAuth 登录有本质区别：

| 维度 | 标准 OAuth 登录（OAuthController 已实现） | 飞书自助绑定（本节设计） |
|------|----------------------------------------|------------------------|
| 目的 | 用户通过第三方 IdP 登录 SuperSonic Web | 已知飞书身份的用户关联 SuperSonic 账号 |
| 发起方 | 用户在 Web 登录页点击"使用 Google 登录" | 飞书机器人发送绑定卡片，用户点击按钮 |
| 身份验证方向 | 第三方 IdP → 验证用户身份 → 创建/匹配 SuperSonic 用户 | 用户提供 SuperSonic 账号密码 → 验证通过 → 关联飞书 open_id |
| 运行环境 | 浏览器，支持 cookie/redirect | 飞书内置浏览器（H5），不依赖 cookie |
| 复用 OAuth 框架 | ✅ 使用 `OAuthService` + `OAuthProvider` + `OAuthController` | ❌ 不适用（不是第三方 IdP 登录，是反向绑定） |

**结论**：飞书自助绑定是**反向身份关联**（已知飞书身份 → 验证 SuperSonic 身份 → 建立映射），不走 `OAuthService` / `OAuthController` 的标准 OAuth 流程。实现为 `feishu-server` 模块内的独立绑定端点 + 轻量 H5 页面。

**两种子方案对比**：

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **C1：H5 页面 + 账号密码验证** | 卡片按钮打开 H5 页面，用户输入 SuperSonic 账号密码 | 实现简单，1-2 天；无需飞书 OAuth 应用额外配置；覆盖所有有 SuperSonic 账号的用户 | 用户需要记住 SuperSonic 密码 |
| **C2：飞书 OAuth 网页授权** | 卡片按钮触发飞书网页 OAuth → 获取飞书用户信息 → 按邮箱/工号二次匹配 | 用户无需输入密码，体验更好 | 本质上是自动匹配的重试（飞书 OAuth 获取的信息与通讯录 API 相同），如果自动匹配已失败说明无共享字段，OAuth 也匹配不上；需额外飞书 OAuth 配置（redirect_uri 白名单） |

**推荐策略**：优先实现 **C1（H5 页面 + 账号密码验证）**。C2 仅在"通讯录 API 权限不足但 OAuth 可获取邮箱"的特殊场景下有增量价值，作为后续可选扩展。

#### 9.7.4 bindToken 设计

绑定流程需要一个安全令牌将飞书卡片按钮与 H5 绑定请求关联，防止伪造绑定：

```
bindToken = base64url(JSON{
    openId: "ou_xxx",           // 飞书 open_id
    mappingId: 123,             // s2_feishu_user_mapping.id（PENDING 记录）
    tenantId: 1,                // 租户 ID
    exp: 1709200000             // 过期时间（30 分钟）
}) + "." + HMAC-SHA256(payload, appSecret)
```

| 属性 | 说明 |
|------|------|
| 生成时机 | `createPendingMapping()` 成功后，由 `FeishuBotService` 调用 `FeishuBindTokenService.generateToken()` |
| 有效期 | 30 分钟（`s2.feishu.oauth.bind-token-ttl-minutes=30`），过期后需用户重新发消息触发 |
| 使用次数 | 一次性，绑定成功后标记已使用（通过 FeishuCacheService 缓存 `bind:used:{tokenHash}`） |
| 签名密钥 | 复用 `FeishuProperties.appSecret`（已有，无需新增密钥） |
| 防重放 | tokenHash 缓存去重（Caffeine / Redis） |

**为什么不用 UUID 随机令牌**：UUID 方案需要服务端存储 token → openId 映射（额外一张表或缓存），HMAC 签名方案是无状态的，服务端通过验签 + 过期时间即可确认合法性，无需存储。

#### 9.7.6 API 设计

##### GET /api/feishu/bindPage

H5 绑定页面入口。`@AuthenticationIgnore`（无需 SuperSonic 登录态）。

| 参数 | 类型 | 说明 |
|------|------|------|
| `token` | query | bindToken（必选） |

**行为**：
1. 验证 token 签名和有效期
2. 有效 → 返回 `feishu-bind.html` 静态页面（页面内 JS 解析 token 中的 `feishuUserName` 展示）
3. 无效/过期 → 返回错误页面（提示"链接已过期，请在飞书对话中重新发送任意消息获取新链接"）

> **实现方式**：Spring Boot 静态资源 + `@Controller` 转发。H5 页面为 `resources/static/feishu/bind.html`，通过 `GET /api/feishu/bindPage?token=xxx` 重定向到 `/feishu/bind.html?token=xxx`（静态资源路径）。或直接用 `@GetMapping` 返回 `ModelAndView` / `ResponseEntity<Resource>`。

##### POST /api/feishu/bind

绑定验证端点。`@AuthenticationIgnore`。请求体 `application/json`。

**请求**：

```json
{
    "bindToken": "eyJ...",
    "username": "zhangsan",
    "password": "***"
}
```

**响应**：

```json
// 成功
{
    "code": 200,
    "data": {
        "success": true,
        "s2UserName": "张三",
        "message": "绑定成功"
    }
}

// 失败 — 账号密码错误
{
    "code": 200,
    "data": {
        "success": false,
        "message": "账号或密码错误，请重试"
    }
}

// 失败 — token 过期
{
    "code": 400,
    "data": {
        "success": false,
        "message": "绑定链接已过期，请在飞书对话中重新发送任意消息获取新链接"
    }
}
```

**处理流程**：

```java
@PostMapping("/api/feishu/bind")
@AuthenticationIgnore
public ResultData<FeishuBindResult> bind(@RequestBody FeishuBindRequest request) {
    // 1. 验证 bindToken（签名 + 过期 + 防重放）
    BindTokenPayload payload = bindTokenService.validateToken(request.getBindToken());

    // 2. 验证 SuperSonic 账号密码
    //    通过 HTTP 调用 POST /api/auth/user/login（复用已有登录接口）
    User user = apiClient.login(request.getUsername(), request.getPassword(), payload.getTenantId());
    if (user == null) {
        return ResultData.success(FeishuBindResult.fail("账号或密码错误，请重试"));
    }

    // 3. 检查该 SuperSonic 用户是否已被其他飞书账号绑定
    FeishuUserMappingDO existing = mappingService.findByS2UserId(user.getId());
    if (existing != null && !existing.getFeishuOpenId().equals(payload.getOpenId())) {
        return ResultData.success(FeishuBindResult.fail(
            "该平台账号已被其他飞书用户绑定，请联系管理员处理"));
    }

    // 4. 完成绑定（更新 PENDING 映射）
    mappingService.completeBinding(payload.getMappingId(), user.getId());

    // 5. 标记 token 已使用
    bindTokenService.markUsed(request.getBindToken());

    // 6. 通过飞书消息通知用户绑定成功
    messageSender.sendText(payload.getOpenId(),
        "✅ 绑定成功！您的飞书账号已关联平台用户「" + user.getDisplayName() + "」，现在可以开始查询了。");

    return ResultData.success(FeishuBindResult.ok(user.getDisplayName()));
}
```

#### 9.7.10 安全性分析

| 威胁 | 风险 | 缓解措施 |
|------|------|---------|
| bindToken 被转发给他人 | 他人用被绑定者的飞书身份关联自己的 SuperSonic 账号 | token 有效期 30 分钟；绑定成功后 token 失效（一次性）；绑定时需要提供 SuperSonic 密码（攻击者不知道受害者密码） |
| bindToken 被暴力猜测 | 伪造 token 绕过 HMAC 签名 | HMAC-SHA256 签名 + appSecret（256 bit 密钥），暴力破解不可行 |
| 重放攻击 | 已使用的 token 被重复提交 | `bindTokenService.markUsed()` 缓存 tokenHash，重复使用返回"链接已过期" |
| 密码暴力破解 | 通过绑定接口尝试枚举密码 | 同一 token 最多尝试 5 次（超出后 token 作废）；同一 IP 30 分钟内最多 20 次绑定请求 |
| CSRF | 第三方页面伪造绑定请求 | bindToken 绑定到特定 openId，且需提供正确密码 |
| XSS | H5 页面注入恶意脚本 | 飞书用户名通过 `textContent` 渲染（非 `innerHTML`）；token payload 中的字段经过白名单校验 |

#### 9.7.11 已绑定用户重新发消息时的边界处理

| 场景 | 行为 |
|------|------|
| PENDING 用户再次发消息 | 检测到已有 PENDING 记录 → 不重复创建 → 生成新 bindToken → 发送绑定卡片（新链接） |
| 已绑定用户发消息 | `resolveMapping()` 正常返回 → 走查询流程，不触发绑定 |
| 同一飞书用户绑定到已被占用的 SuperSonic 账号 | 绑定接口返回"该平台账号已被其他飞书用户绑定" |
| 管理员在绑定前已手动完成映射 | 用户发消息时 `resolveMapping()` 命中活跃映射 → 正常使用，bindToken 自然过期 |
| 绑定后管理员禁用映射 | `resolveMapping()` 返回 null → 重新走绑定流程 |

#### 9.7.13 与阶段 2 其他功能的关系

- **企业通讯录同步（方案 D）**：独立于 OAuth 绑定。通讯录同步是批量自动化方案（全员覆盖），OAuth 绑定是个体自助方案（匹配失败兜底）。两者可并存：通讯录同步覆盖大部分用户，OAuth 绑定兜底剩余未匹配用户。

- **定时报表推送**：已实现，不依赖 OAuth 绑定。

## 十、改进规划

基于阶段 1 实现的代码 review 和联调过程中发现的问题，以下是后续可改进方向，按类别分组并标注优先级。

### 10.1 可靠性 & 健壮性

#### 10.1.1 chatIdCache 无界增长（✅ 已完成）

**问题**：`SuperSonicApiClient.chatIdCache` 使用无界 `ConcurrentHashMap`，随着用户数增长会持续消耗内存，且无 TTL 机制。

**已实现方案**：复用 `FeishuCacheService` 抽象层（方案二），统一缓存策略。`ConcurrentHashMap` 已替换为 `cacheService.get(key)` / `cacheService.put(key, value)` 调用。底层通过 `generalCache`（Caffeine: 30min TTL + maxSize=5000 / Redis: `feishu:cache:` 前缀 + 30min TTL）实现有界缓存。

#### 10.1.2 queryTimeoutMs 未实际生效（✅ 已完成）

**问题**：`FeishuProperties.queryTimeoutMs=30000` 配置存在但 `SuperSonicApiClient` 的 `RestTemplate` 未设置 `connectTimeout` / `readTimeout`，查询可能无限阻塞。

**已实现方案**：在 `SuperSonicApiClient` 构造函数中通过 `SimpleClientHttpRequestFactory` 设置 `connectTimeout` 和 `readTimeout`，均使用 `properties.getQueryTimeoutMs()`（默认 30s）。

#### 10.1.3 SuperSonicApiClient 缺少重试机制（✅ 已完成）

**问题**：`FeishuTokenManager` 有 3 次指数退避重试，但 `SuperSonicApiClient.query()` 失败直接抛异常。瞬时网络抖动或 GC 暂停可能导致不必要的查询失败。

**已实现方案**：新增 `exchangeWithRetry()` 方法，仅对 `ResourceAccessException`（连接超时 / 读超时 / 连接拒绝）重试 1 次（共 2 次尝试），间隔 1s。业务错误（`extractData()` 抛出的非 200 响应）不重试。已应用于 `doQuery()`、`doParse()`、`queryBySql()` 三个核心路径。与 chatId 重试（10.1.4）分层：网络重试为内层，chatId 重试为外层。

#### 10.1.4 chatId 缓存失效场景（✅ 已完成）

**问题**：管理员在 Web 端删除 chat session 后，`chatIdCache` 仍持有旧 chatId，后续查询返回 404。

**已实现方案**：
- `FeishuCacheService` 新增 `remove(key)` 方法（Caffeine: `invalidate()` / Redis: `delete()`）
- `SuperSonicApiClient` 新增 `evictAndRefreshChat(agentId, user)` 方法：清除缓存 → 重新 `getOrCreateChat()`
- `query()` 和 `parse()` 首次调用失败时，evict 缓存并用新 chatId 重试一次（仅当新旧 chatId 不同时重试）
- 结合 10.1.1 的 TTL 机制（30min），过期缓存自动清理

### 10.2 功能增强

#### 10.2.1 导出功能完整实现（✅ 已完成）

#### 10.2.3 查询上下文 / 追问能力（✅ 已完成）

**已实现方案**：利用 SuperSonic 已有的多轮对话能力，飞书端通过三层机制支持追问：

1. **chatId 共享**（已有）：同一 user+agent 复用 chatId，`NL2SQLParser.rewriteMultiTurn()` 自动从 chatId 历史加载上一轮查询上下文（需 Agent 启用 `REWRITE_MULTI_TURN`）
2. **dataSetId 传递**（新增）：`QueryMessageHandler` 缓存上一次成功查询的 `dataSetId`（via `FeishuCacheService`），下次查询时通过 `ChatParseReq.dataSetId` 传入，帮助 parser 定位到同一数据集
3. **动态追问提示**（新增）：成功查询的结果卡片底部显示基于查询语义上下文动态生成的追问引导

**追问提示生成 — `FollowUpHintGenerator`**

抽取为 `headless-api` 模块的公共工具类 `FollowUpHintGenerator`，其他模块（如 Web Chat UI）可直接复用。

```java
// 任意模块调用
List<String> hints = FollowUpHintGenerator.generate(parseInfo, rowCount, maxDisplayRows);
// 便捷重载（不需要行数信息时）
List<String> hints = FollowUpHintGenerator.generate(parseInfo);
```

基于 `SemanticParseInfo` 的 5 类规则，按优先级生成最多 3 条口语化提示：

| 优先级 | 类别 | 数据来源 | 示例 |
|--------|------|----------|------|
| 1 | 维度 drill-down | `dimensions` + `elementMatches` | 纯聚合→`各城市的呢`；单维度→`加上部门看看`；多维度→`只按城市看` |
| 2 | 筛选变更 | `dimensionFilters` | 有筛选值→`换成全部城市`；长值→`换个城市看看` |
| 3 | 指标切换 | `metrics`（多指标时） | `只看访问量` |
| 4 | 日期变更 | `dateInfo`（mode/unit/period） | 1天→`看看最近7天的`；7天→`看看最近30天的` |
| 5 | 结果精炼 | 结果行数 + `limit` | 超表格上限→`导出`；无 limit→`看前10名` |

维度 drill-down 优先从 `elementMatches` 中查找同数据集的未使用非时间维度（真实维度名），找不到时退化为基于指标名的通用引导。

- 示例交互：
  ```
  用户: 昨天的销售额
  Bot:  [查询结果卡片]
        ───────────
        可继续追问，如「各城市的呢」、「看看最近7天的」

  用户: 各城市的呢
  Bot:  [按城市的查询结果卡片]
        ───────────
        可继续追问，如「加上部门看看」、「换成全部城市」、「看看最近7天的」
  ```

**帮助卡片动态化**

`renderHelpCard(agentName, examples)` 从当前 Agent 的 `examples` 字段动态读取示例问题，不再硬编码。`HelpMessageHandler` 通过 `apiClient.getAgentList()` 获取当前 Agent 信息传入。

### 10.4 安全性

#### 10.4.1 飞书 API 调用频率控制

**问题**：飞书开放平台对各 API 有 QPS 限制（如 IM 消息发送 50 QPS），当前无客户端限流。

**改进方案**：
- 在 `FeishuMessageSender` / `FeishuTokenManager` 添加基于令牌桶的本地限流
- 可选方案：Guava `RateLimiter` 或 Resilience4j `RateLimiter` + `CircuitBreaker`
- 配置化：`s2.feishu.rate-limit.message-qps=40`（留 20% 余量）

#### 10.4.2 敏感数据脱敏

**问题**：查询结果可能包含敏感信息（手机号、身份证、金额等），飞书消息一旦发出不可撤回。

**改进方案**：
- 在 `FeishuCardRenderer` 渲染前检查 `QueryColumn` 的脱敏标记（SuperSonic 已有字段级脱敏能力）
- 对超过 N 行且包含敏感列的数据，自动隐藏具体值并引导走导出流程
- 群聊场景下，含敏感字段的结果仅发送单聊

#### 10.4.3 PENDING 映射自动过期

**问题**：PENDING 状态的映射记录如果长期未处理，飞书用户每次发消息都会检查（虽然有去重逻辑，不会创建重复 PENDING）。

**改进方案**：
- PENDING 记录超过 30 天自动标记为 EXPIRED（定时任务清理）
- 管理后台增加 PENDING 数量告警（> 10 条时通知管理员）
- EXPIRED 用户再次发消息时重新创建 PENDING

### 10.5 架构优化

#### 10.5.1 消息处理异步队列化 ✅ 已实现

**改进内容**：

1. **专用线程池**：新增 `FeishuAsyncConfig` 配置类，注册 `feishuExecutor` Bean（`ThreadPoolTaskExecutor`），通过 `FeishuProperties.AsyncConfig` 可配置：
   - `corePoolSize`（默认 4）、`maxPoolSize`（默认 8）、`queueCapacity`（默认 100）
   - 拒绝策略：`AbortPolicy`（抛出 `RejectedExecutionException`）

2. **手动提交代替 `@Async`**：`FeishuBotService` 不再使用 `@Async` 注解，改为手动调用 `feishuExecutor.execute()`，这样可以在提交时捕获 `RejectedExecutionException`，直接向用户回复"系统繁忙"错误卡片。

3. **优雅拒绝**：队列满时，`replyBusy()` 方法从事件中提取 `messageId`（兼容 v1.0/v2.0），通过 `FeishuCardRenderer.renderErrorCard()` 发送友好提示。

**涉及文件**：
| 文件 | 变更 |
|------|------|
| `feishu/api/.../FeishuProperties.java` | 新增 `AsyncConfig` 内部类 |
| `feishu/server/.../config/FeishuAsyncConfig.java` | 新建，注册 `feishuExecutor` Bean |
| `feishu/server/.../service/FeishuBotService.java` | 移除 `@Async`，改用手动提交 + 拒绝处理 |

**配置示例**（`application.yaml`）：
```yaml
s2:
  feishu:
    async:
      core-pool-size: 4
      max-pool-size: 8
      queue-capacity: 100
```

**后续可选扩展**：引入 Redis Stream 做持久化缓冲（支持消费者组，多实例水平扩展）

#### 10.5.2 SuperSonicApiClient 双模式改造 ✅ 已完成

**已实现方案**：`SuperSonicApiClient` 提取为接口，提供两种实现通过 SPI 自动切换（无需配置项）：
- `HttpSuperSonicApiClient`（`feishu-server`）— HTTP 回环调用，`@ConditionalOnMissingBean` 兜底，微服务部署时生效
- `DirectSuperSonicApiClient`（`launchers/standalone`）— 直接调用 Service 层，通过 `spring.factories` 注册为 auto-configuration，standalone 部署时优先生效

**原设计方案**（保留供参考）：抽取统一接口，提供两种实现，通过配置项切换：

```java
// 统一接口
public interface SuperSonicApiClient {
    QueryResult query(String queryText, int agentId, User user);
    Integer getOrCreateChat(int agentId, User user);
    List<Map<String, Object>> getAgentList();
}

// 实现 1：Standalone 同 JVM 直联（默认）
@Component
@ConditionalOnProperty(name = "feishu.api-mode", havingValue = "direct", matchIfMissing = true)
public class DirectServiceClient implements SuperSonicApiClient {
    private final ChatQueryService chatQueryService;
    private final ChatManageService chatManageService;
    // 直接调用 Service，无序列化开销
}

// 实现 2：微服务独立部署 HTTP 调用
@Component
@ConditionalOnProperty(name = "feishu.api-mode", havingValue = "http")
public class HttpApiClient implements SuperSonicApiClient {
    private final RestTemplate restTemplate;
    // 保留现有 HTTP 调用逻辑
}
```

**配置**（`application.yaml`）：

```yaml
feishu:
  api-mode: direct   # direct = Standalone 同 JVM | http = 微服务独立部署
```

**直联模式收益**（Standalone 部署）：
- 消除 HTTP 序列化/反序列化开销
- 无需处理 `ResponseAdvice` 的 `ResultData` 包装
- 无需 `Accept: application/json` header（避免 XML 响应问题）
- 无需 `InternalTokenGenerator` JWT 认证

**HTTP 模式保留**（微服务部署）：
- 第二阶段 feishu 模块独立部署时，切换 `feishu.api-mode=http` 即可
- 现有 HTTP 调用逻辑迁移至 `HttpApiClient`，修复已知的 `ResponseAdvice` 解包装问题

#### 10.5.3 飞书卡片样式外部化

**现状**：`FeishuCardTemplate` 通过 Java HashMap 手动拼装 JSON，修改卡片样式需改 Java 代码并重新编译。

**约束分析**：查询结果卡片（表格卡片）的结构本身是动态的——列数、列名、行数完全取决于 NL2SQL 返回的 `QueryResult`，无法用静态 JSON 模板表达。飞书卡片的嵌套结构（`column_set` → `column` → `markdown`）深度随列数变化，用模板引擎生成比代码更难维护。

**改进方案（分类处理）**：
- **固定结构卡片**（帮助、错误、引导绑定、单值指标）→ 存为 JSON 模板文件（`resources/feishu/templates/*.json`），通过 Mustache 填充变量，非开发人员可通过飞书[消息卡片搭建工具](https://open.feishu.cn/cardkit)设计后导入
- **动态表格卡片** → 保持 `FeishuCardRenderer` 代码生成，将样式参数（颜色、字号、间距、截断行数等）提取到 `FeishuProperties` 配置项，修改样式无需改代码
- 固定结构模板支持热更新：模板文件变更后无需重启
