## 0. 一页摘要

### 0.1 文档目标

本文是模板报表子系统的**子系统内核设计文档**，重点回答：

- 模板报表的边界是什么
- 查询、导出、调度、分发如何统一收口
- 权限、审计和治理如何落在执行链路上
- 哪些能力属于本子系统核心，哪些只是依赖的平台能力

### 0.2 当前结论

- **执行主链路**：统一由 `ReportExecutionOrchestrator` 收口
- **导出主链路**：统一由 `ExportTaskService` 承接
- **调度主链路**：统一由 Quartz + 调度配置表驱动
- **平台边界**：数据同步、连接器、投递通道是共享能力，模板报表只复用，不单独重新定义平台总架构

### 0.3 与其他文档的关系

- 平台主架构和 MVP 边界详见 [智能运营数据中台设计方案](智能运营数据中台设计方案.md)
- 飞书入口如何调用模板报表能力详见 [飞书机器人用户交互层设计方案](飞书机器人用户交互层设计方案.md)
- 数据同步与连接器平台依赖详见 [模板报表-数据同步与连接器依赖说明](details/模板报表-数据同步与连接器依赖说明.md)

- **已上线**：主干已具备且可直接用于生产/测试环境
- **已在主干**：代码已存在于当前主干，但仍需接入、联调或补充外围治理

- **待开发**：当前工程中不存在对应实现

## 1. 设计目标

模板报表子系统旨在将**高频、标准化的数据查询与报表需求产品化**，通过"模板 + 参数 + 权限 + 调度 + 多渠道输出"的方式，实现：

- 固定报表无需重复开发
- 业务可自助、安全地获取数据
- 报表口径统一、可治理、可审计
- 支撑 Web、API、定时任务、协作工具等多种触达方式

该子系统是数仓查询能力向"数据产品能力"升级的关键基础设施。

### 1.1 文档边界

| 范围 | 是否属于本文核心 |
|------|----------------|
| 模板生命周期、参数、执行编排 | **是** |
| 导出、调度、执行审计 | **是** |
| 报表结果分发（站内/邮件/IM） | **是** |
| Dify / MCP 总体集成策略 | **否**，以整体方案为准 |
| 通用连接器平台 | **否**，本文只说明报表如何复用 |
| 数据同步平台 / Connection 平台 | **否**，本文只说明报表查询隔离依赖 |

## 2. 系统整体架构

### 2.1 架构分层

```
用户 / Web / API / Agent / 定时任务 / 外部 Workflow
        │
        ▼
模板报表服务（Template Report Service）
├── 模板管理模块         → 已上线：SemanticTemplate + DataSet
├── 参数与校验模块       → 已上线：ConfigParam + QueryStructReq
├── 执行编排模块         → 已上线：ReportExecutionOrchestrator + ReportExecutionContext
├── SQL 渲染与执行模块   → 已上线：SemanticLayerService + SqlTemplateEngine
├── 权限注入模块         → 已上线：S2DataPermissionAspect
├── 输出与导出模块       → 已上线：同步导出 + 异步导出 + 分发通道
├── 调度与触发模块       → 已上线：Quartz Scheduler + Web 管理能力
├── 查询资源隔离         → 已上线：SQL 超时 + 连接池隔离 + 扫描行数估算
└── 治理与审计模块       → 已上线：执行日志 + 影响分析 + 下线保护
        │
        ▼
多数据源（ClickHouse / MySQL / PostgreSQL / H2 / DuckDB）
```

### 2.2 与现有工程的关系

模板报表子系统**不是独立新建的模块**，而是在现有语义模板（Semantic Template）和数据集（DataSet）体系上的**功能增强**：

- **语义模板**解决"数据是什么"——定义 Domain、Model、Metric、Dimension、DataSet
- **模板报表**解决"数据怎么查、怎么推"——在 DataSet 之上定义参数化查询、调度、输出

两者共享同一套查询执行引擎（`SemanticLayerService`）、权限体系（`S2DataPermissionAspect`）和多租户隔离（`TenantContext`）。

## 3. 核心对象模型

### 3.1 核心对象关系

```
SemanticTemplate ──(部署)──→ SemanticDeployment
      │                              │
      │ 1:N                    冻结配置快照（不可变）
      │
      ├──→ DataSet ──→ ReportSchedule ──→ ReportExecutionContext
      │                     │                     │
      │                     │ 触发                 │ 产生
      │                     ▼                     ▼
      │               ReportExecution      ExportTask
      │                     │
      │                     ▼
      └──→ Agent        ReportDeliveryRecord（推送记录）
```

**核心对象说明**：

| 对象 | 职责 |
|------|------|
| `SemanticTemplate` | 模板定义，含版本号（`currentVersion`），状态 Draft/Deployed/Offline |
| `SemanticDeployment` | 部署记录，含配置快照（`templateConfigSnapshot`，不可变） |
| `ReportSchedule` | 调度配置，含 Cron 表达式、绑定的 `templateVersion`、推送渠道 |
| `ReportExecutionContext` | 不可变执行上下文，统一所有入口（Web/Agent/Schedule/API）的执行参数 |
| `ReportExecution` | 执行记录，含完整的 `execution_snapshot`，用于历史复现和审计 |
| `ExportTask` | 异步导出任务，状态 PENDING/RUNNING/SUCCESS/FAILED/EXPIRED |
| `ReportDeliveryConfig` | 推送配置，支持 EMAIL/WEBHOOK/FEISHU/DINGTALK/WECHAT_WORK |
| `ReportDeliveryRecord` | 推送记录，含幂等键（`delivery_key`）防重复推送 |

### 3.2 执行流程概览

```
所有入口（Web / Agent / Schedule / API）
        │
        ▼
ReportExecutionOrchestrator.execute(ReportExecutionContext)
 ├── 1. 校验模板状态（必须 Deployed）
 ├── 2. 校验运行时参数（类型 + 必填）
 ├── 3. 注入数据权限（行级 + 字段级）
 ├── 4. 渲染 SQL（结构化查询 / SQL 模板三路解析）
 ├── 5. 执行查询（QueryExecutor 多数据源）
 ├── 6. 持久化执行记录（s2_report_execution）
 ├── 7. 分流：EXPLAIN 预估行数 ≤ 阈值 → 同步导出；> 阈值 → 异步 ExportTask
 ├── 8. 格式化输出（Excel / CSV / JSON）
 └── 9. 投递（返回 / 推送到渠道）
```

→ 详见 [执行引擎](details/report/02-execution-engine.md)

## 4. 与 Agent / 外部 Workflow 的协作

模板报表能力优先通过 SuperSonic 内部 Agent / API 能力暴露。

语义模板部署时通过 `TemplateDeployedEvent` 自动创建 Agent 和 Plugin：

```
SemanticTemplate.deploy()
    → 创建 Domain / Model / Metric / DataSet
    → 发布 TemplateDeployedEvent
        → 创建 Agent（关联 DataSet，配置 ChatApp）
        → 创建 Plugin（WEB_PAGE / WEB_SERVICE 类型）
        → 创建 REPORT_SCHEDULE Plugin 并绑定到 Agent
```

已支持通过 `REPORT_SCHEDULE` 类型 Plugin 实现自然语言创建定时报表。

**外部 Workflow 接入原则**：Workflow 通过 HTTP API 或 MCP Tool 调用接口，不持有模板报表内部状态，不替代 `ReportExecutionOrchestrator`。

| 工具 | 现有实现 | 状态 |
|------|---------|------|
| listQueryTemplates | `SemanticTemplateController.getTemplateList()` | ✅ 已上线 |
| executeQueryTemplate | `ChatQueryApiController` + `SqlQueryApiController` | ✅ 已上线 |
| exportQueryResult | `DownloadServiceImpl.downloadByStruct()` | ✅ 已上线 |
| scheduleQuery | `ReportScheduleController` + Quartz Scheduler | ✅ 已上线 |

## 5. 非功能性设计

### 5.1 性能

| 能力 | 实现 | 状态 |
|------|------|------|
| 查询结果缓存 | `CaffeineCacheManager`，可配置 TTL | ✅ 已上线 |
| 大报表异步导出 | Export Worker 线程池 + JDBC fetchSize 流式读取 + EasyExcel 流式写入 | ✅ 已上线 |
| 分页查询 | `PageHelper` + `limit` | ✅ 已上线 |
| SQL 级超时 | `JdbcTemplate.setQueryTimeout()`，`s2.source.query-timeout`（默认 30 秒） | ✅ 已上线 |
| 连接池隔离 | `JdbcDataSource` 四种池类型（INTERACTIVE/REPORT/EXPORT/SYNC） | ✅ 已上线 |
| EXPLAIN 扫描估算 | `RowCountEstimator`，`supersonic.export.async-threshold`（默认 10000 行） | ✅ 已上线 |

### 5.2 可扩展性

| 能力 | 实现 | 状态 |
|------|------|------|
| 新数据源 | `QueryExecutor` + `DbAdaptor` 接口 | ✅ MySQL/ClickHouse/PostgreSQL/DuckDB/H2 |
| 输出渠道插件化 | `ReportDeliveryChannel` 策略模式，新增渠道只需实现接口 | ✅ 已上线 |
| 自定义模板 | `SemanticTemplateConfig` | ✅ 已上线 |

### 5.3 多租户

| 能力 | 实现 |
|------|------|
| 数据隔离 | `TenantSqlInterceptor` 自动注入 `WHERE tenant_id = ?` |
| 模板隔离 | `SemanticTemplateDO.tenantId`，内置模板全局可见 |
| 配额控制 | `SubscriptionInfoProvider` 通过订阅计划限制 DataSet/Model/Agent 数量 |

### 5.4 上线验收清单

- [ ] 功能验收：模板执行、调度、导出、推送、同步全链路通过
- [ ] 性能验收：高频模板与大导出场景压测达标
- [ ] 稳定性验收：重试、熔断、降级策略验证通过
- [ ] 审计验收：执行日志、推送记录、版本追踪可回放
- [ ] 指标验收：`/actuator/prometheus` 可查询以下指标（tags 完整）：`supersonic.report.schedule.dispatch.total`、`supersonic.report.execution.total`、`supersonic.report.execution.duration`、`supersonic.report.export.total`、`supersonic.report.export.pending`、`supersonic.report.delivery.total`
- [ ] 告警验收：执行成功率 < 95%、P95 时延 > 10s、导出 pending 高、推送失败率 > 5% 等关键告警在测试环境触发并有处置记录

告警规则参考：`docker/prometheus/rules/report-slo-alert-rules.yml`
