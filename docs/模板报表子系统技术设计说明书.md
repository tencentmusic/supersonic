# 模板报表子系统技术设计说明书

> 适用范围：智能运营数据中台（SuperSonic）

---

## 1. 设计目标

模板报表子系统旨在将**高频、标准化的数据查询与报表需求产品化**，通过"模板 + 参数 + 权限 + 调度 + 多渠道输出"的方式，实现：

- 固定报表无需重复开发
- 业务可自助、安全地获取数据
- 报表口径统一、可治理、可审计
- 支撑 Web、API、定时任务、协作工具等多种触达方式

该子系统是数仓查询能力向"数据产品能力"升级的关键基础设施。

---

## 2. 系统整体架构

### 2.1 架构分层

模板报表子系统位于数据中台的**数据服务层**，对上承接 Agent / Workflow，对下对接数仓与存储系统。

```
用户 / Agent / Workflow / 定时任务
        │
        ▼
模板报表服务（Template Report Service）
├── 模板管理模块         → 已实现：SemanticTemplate + DataSet
├── 参数与校验模块       → 已实现：ConfigParam + QueryStructReq
├── 执行编排模块         → 已实现：ReportExecutionOrchestrator + ReportExecutionContext
├── SQL 渲染与执行模块   → 已实现：SemanticLayerService
├── 权限注入模块         → 已实现：S2DataPermissionAspect
├── 输出与导出模块       → 已实现：Excel/CSV 同步导出 + 异步导出任务中心 + 多渠道推送
├── 调度与触发模块       → 已实现：Quartz Scheduler 内嵌 + Web 管理界面
├── 数据同步模块         → 已实现：内置 DataX 模式增量同步 + Web 管理界面
├── 查询资源隔离         → 已实现：SQL 超时 + 连接池隔离（四种池类型）+ 扫描行数估算
└── 治理与审计模块       → 已实现：执行日志 + 影响分析（DataSet 删除级联校验 + 模板下线保护）
        │
        ▼
多数据源（ClickHouse / MySQL / PostgreSQL / H2 / DuckDB）
```

### 2.2 与现有工程的关系

模板报表子系统**不是独立新建的模块**，而是在现有语义模板（Semantic Template）和数据集（DataSet）体系上的**功能增强**。核心思路：

- **语义模板**解决"数据是什么"——定义 Domain、Model、Metric、Dimension、DataSet
- **模板报表**解决"数据怎么查、怎么推"——在 DataSet 之上定义参数化查询、调度、输出

两者共享同一套查询执行引擎（`SemanticLayerService`）、权限体系（`S2DataPermissionAspect`）和多租户隔离（`TenantContext`）。

## 3. 功能模块设计

### 3.1 执行上下文模型

为统一手动执行、调度执行、Agent 调用等多种场景，引入不可变执行上下文对象。


**设计收益**：

- 执行过程可追溯、可回放

- 为失败重跑、历史重算提供基础能力

**复杂度**：中（全栈，~12 个文件）| **优先级**：P1

##### 设计方案

采用 **"主表版本号 + 部署快照"** 模式，避免独立版本表带来的额外 JOIN 复杂度。

**核心原则**：
- 模板每次编辑 → `current_version++`（版本号单调递增）
- 部署时将当前版本号和配置快照写入 `s2_semantic_deployment`（不可变）
- 调度任务绑定的 `templateVersion` 指向创建/更新时刻的版本号
- 只允许最新版本部署（不支持老版本重新部署）

##### 数据模型变更

```sql
-- V19__template_version.sql (MySQL)

-- 1. 模板主表增加版本号
ALTER TABLE s2_semantic_template
  ADD COLUMN `current_version` bigint NOT NULL DEFAULT 1
    COMMENT '当前版本号，每次编辑+1';

-- 2. 部署记录表增加版本快照
ALTER TABLE s2_semantic_deployment
  ADD COLUMN `template_version` bigint DEFAULT NULL
    COMMENT '部署时的模板版本号',
  ADD COLUMN `template_config_snapshot` longtext DEFAULT NULL
    COMMENT 'JSON: 部署时刻的模板配置快照（不可变，用于回溯）';

-- PostgreSQL 版本同步创建
```

### 3.3 模板元数据模型

#### 3.3.1 报表模板（映射到语义模板 + 数据集）

报表模板的元数据分布在两层：

**语义模板层**（`s2_semantic_template`）：

| 字段 | 对应现有字段 | 说明 |
|------|-----------|------|
| template_code | `biz_name` | 模板唯一编码 |
| template_name | `name` | 模板名称 |
| business_domain | `template_config → DomainConfig.name` | 业务域 |
| datasource_type | `template_config → ConfigParam[type=DATABASE]` | 数据源由部署参数指定 |
| sql_template | 不直接对应 | 见下方说明 |
| status | `status` | 0=Draft, 1=Deployed, 2=Offline |
| creator | `created_by` | 创建人 |
| current_version | `current_version`（已实现，见 3.2.1 版本管理） | 当前生效版本号，模板每次编辑+1 |

**数据集层**（`s2_data_set`）— 模板部署后生成：

| 字段 | 说明 |
|------|------|
| `data_set_detail` | JSON：关联的 Model、Metric、Dimension 集合 |
| `query_config` | JSON：默认查询配置（时间范围、聚合方式） |

> **说明**：现有体系不使用"参数化 SQL 模板"，而是通过**结构化查询请求**（`QueryStructReq`）组合 Dimension + Metric + Filter + DateConf，由 `SemanticLayerService` 自动生成 SQL。这种方式比手写 SQL 模板更安全（天然防注入），且自动适配不同数据源方言。

#### 3.3.2 参数定义（映射到 ConfigParam + QueryStructReq）

**部署时参数**（`ConfigParam`）：

| 字段 | 对应现有字段 | 说明 |
|------|-----------|------|
| param_key | `ConfigParam.key` | 参数名 |
| param_type | `ConfigParam.type` | DATABASE / TABLE / FIELD / TEXT |
| required | 隐式（模板引用即必填） | `${key}` 占位符在部署时解析 |
| default_value | `ConfigParam.defaultValue` | 默认值 |

**运行时查询参数**（`QueryStructReq`）：

| 设计概念 | 现有实现 | 说明 |
|---------|---------|------|
| 日期参数 | `DateConf`（RECENT / LAST / BETWEEN） | 支持相对日期和绝对日期范围 |
| 维度过滤 | `List<Filter> dimensionFilters` | 支持 =, IN, LIKE, >, < 等操作符 |
| 指标过滤 | `List<Filter> metricFilters` | HAVING 条件 |
| 分组维度 | `List<String> groups` | GROUP BY 字段 |
| 聚合方式 | `List<Aggregator> aggregators` | SUM / COUNT / AVG / MAX / MIN |
| 排序 | `List<Order> orders` | ORDER BY |
| 分页 | `Long limit` | 行数限制 |

> **Gap**: 运行时参数缺少显式的参数声明和校验机制。建议后续在 `QueryConfig` 中增加参数模板定义，包含类型校验、默认值和枚举约束。

#### 3.3.3 参数模板定义

参数采用**显式 Schema 定义**，运行期强校验。

```json
{
"startDate": { "type": "date", "required": true, "maxRangeDays": 31 },
"city": { "type": "enum", "multi": true, "optionsFrom": "dimension:city" },
"limit": { "type": "number", "max": 50000 }
}
```

校验维度：

- 类型合法性
- 是否必填
- 枚举与范围限制
- 时间跨度与资源保护

### 3.4 SQL 模板规范

#### 3.4.1 现有实现：结构化查询而非 SQL 模板

现有体系**不采用手写 SQL 模板**，而是通过结构化请求自动生成 SQL：

```
用户选择 DataSet + Metric + Dimension + Filter
        │
        ▼
QueryStructReq.convert() → QuerySqlReq
        │
        ▼
SemanticLayerService → SQL 生成 → 数据源执行
```

**优势**：
- 天然防 SQL 注入（参数不直接拼入 SQL）
- 自动适配数据源方言（MySQL / ClickHouse / PostgreSQL）
- 权限注入在 SQL 生成阶段自动完成

**对于需要自定义 SQL 的场景**，现有 `QuerySqlReq` 支持直接传入 SQL：

```java
public class QuerySqlReq extends SemanticQueryReq {
    private String sql;           // 自定义 SQL
    private List<Long> modelIds;  // 关联模型
}
```

#### 3.4.2 安全限制

| 安全措施 | 现有实现 | 说明 |
|---------|---------|------|
| 禁止 DDL/DML | `SqlEditEnum` 校验 | 查询请求仅允许 SELECT |
| 禁止多语句 | SQL 解析层拦截 | JSqlParser 解析确保单语句 |
| 只读执行 | 数据源配置 | 查询使用只读连接池（Druid） |
| 行数限制 | `Constants.DEFAULT_DOWNLOAD_LIMIT` | 防止大结果集 |

```java
private SemanticQueryReq parseQueryConfig(ReportExecutionContext ctx) {

    // 路径 1：尝试 QueryStructReq（现有）
    // 路径 2：尝试 QuerySqlReq（现有）

    // 路径 3：尝试 SqlTemplateConfig（新增）

        sqlReq.setSql(renderedSql);
        sqlReq.setDataSetId(ctx.getDatasetId());
        return sqlReq;
    }

    throw new IllegalArgumentException("Unable to parse queryConfig");
}
```

### 3.5 模板执行流程

执行编排器统一处理所有执行入口（Web / Agent / 定时任务），每次执行首先构建 `ReportExecutionContext`（见 3.1 执行上下文模型），后续步骤基于该上下文对象进行：

所有执行入口（Web / Agent / Schedule / API）统一委托给 `ReportExecutionOrchestrator`，Job 和 Controller 不直接编排业务逻辑：

```
ReportExecutionOrchestrator.execute(ReportExecutionContext ctx)
 │
 ├── 1. buildContext        → 解析入口参数，构建 ReportExecutionContext（见 3.1）
 ├── 2. validateTemplate    → 校验模板状态（必须 Online / Deployed）
 ├── 3. validateParams      → 运行时参数强校验（见 3.3.3 参数模板 Schema）
 ├── 4. injectPermission    → 注入数据权限（行级 / 字段级）
 ├── 5. renderSQL           → 渲染 SQL（结构化生成或模板渲染）
 ├── 6. executeQuery        → 执行查询
 ├── 7. persistExecution    → 写入 s2_report_execution（含 execution_snapshot）
 ├── 8. routeExport         → 判断同步/异步导出（EXPLAIN 预估行数 vs 阈值，见 3.7.1）
 ├── 9. formatOutput        → 输出格式化（Excel / CSV / JSON）
 └── 10. deliverOutput      → 返回 / 推送 / 导出（异步时写入 s2_export_task）
```

**各入口调用方式**：

```java
// Web / API 入口
@PostMapping("/api/v1/report/execute")
public SemanticQueryResp execute(@RequestBody ReportExecuteReq req) {
    ReportExecutionContext ctx = contextBuilder.build(req, currentUser);
    return orchestrator.execute(ctx);
}

// Quartz Job 入口（轻量分派，不含业务逻辑）
public class ReportScheduleJob implements Job {
    public void execute(JobExecutionContext jctx) {
        Long scheduleId = jctx.getMergedJobDataMap().getLong("scheduleId");
        dispatcher.dispatch(scheduleId);  // → 构建 context → orchestrator.execute(ctx)
    }
}

// Agent 入口
agentTool.invoke(params) → contextBuilder.buildFromAgent(params) → orchestrator.execute(ctx)
```

#### 3.7.1 异步导出机制

**问题背景**：多个用户同时发起大数据量导出时，同步处理会导致：

| 风险 | 原因 | 后果 |
|------|------|------|
| 堆内存溢出（OOM） | ResultSet 全量加载 + Excel 对象模型 | JVM 崩溃 |
| 数据库连接耗尽 | 长查询持有 JDBC 连接 | 业务请求阻塞 |
| Web 线程饿死 | 同步导出占用 Tomcat 线程 | 页面无响应 |

**设计方案**：引入同步/异步分流 + 独立 Export Worker 线程池。

**分流策略**：

```
用户发起导出请求
    │
    ├── EXPLAIN 预估行数 ≤ 阈值（默认 5000）
    │       → 同步下载（当前模式，体验好）
    │
    └── EXPLAIN 预估行数 > 阈值
            → 异步任务（返回任务 ID，后台执行）
            → 用户通过"任务中心"查看进度并下载
```

**异步导出流程**：

```
1. 用户提交导出申请 → 后端创建 s2_export_task（status=PENDING），返回任务 ID
2. Export Worker 线程池拉取任务
   → 设置 TenantContext + 权限上下文
   → JDBC fetchSize=1000 流式读取
   → EasyExcel streaming write 逐行写入临时文件
   → 单任务内存占用固定 ~50MB，与数据量无关
3. 写入完成 → 文件存至导出目录（本地 / OSS）→ status=SUCCESS
4. 用户通过"任务中心"下载文件
5. 定时清理过期文件（默认 7 天）
```

**数据模型**：

```
s2_export_task（导出任务）
├── id
├── task_name             → 任务名称（自动生成："GMV日报导出_20260203"）
├── user_id               → 发起用户
├── dataset_id            → 关联数据集
├── query_config          → JSON：完整的 QueryStructReq
├── output_format         → EXCEL / CSV
├── status                → PENDING / RUNNING / SUCCESS / FAILED / EXPIRED
├── file_location         → 结果文件路径（本地目录或 OSS URL）
├── file_size             → 文件大小（字节）
├── row_count             → 导出行数
├── error_message
├── created_at
├── expire_time           → 文件过期时间（默认 created_at + 7 天）
├── tenant_id
```

**Export Worker 线程池**（独立于 Web 线程池和 Quartz 调度线程池）：

```yaml
supersonic:
  export:
    worker-threads: 3                     # Export Worker 线程数
    max-queue-size: 50                    # 等待队列上限
    sync-threshold: 5000                  # 同步/异步分流阈值（行数）
    file-expire-days: 7                   # 导出文件保留天数
    storage-type: local                   # local / oss
    local-dir: ${java.io.tmpdir}/supersonic-export
```

**内存控制要点**：

| 环节 | 措施 | 说明 |
|------|------|------|

| 列 | 说明 |

### 3.8 调度与自动化

**设计目标**：
- 支持 Cron 表达式，用户可自定义调度时间
- 定时任务与模板/数据集绑定
- 支持失败重试（指数退避）与告警
- 集群部署时任务不重复执行
- 提供 Web 管理界面（任务列表、执行记录、手动触发/暂停）
- 调度任务绑定 template_version
- 模板升级不影响已有调度

**典型场景**：
- GMV 日报（每日 9:00 推送）
- 对账日报（每日 10:00 生成 Excel 发送邮件）
- 风控周报（每周一 8:00 推送飞书群）

#### 3.8.1 技术选型：Quartz Scheduler（内嵌）

**选型对比**：

| 方案 | 动态 Cron | 持久化 | 集群防重复 | 重试/Misfire | 管理界面 | 额外基础设施 | Spring Boot 3 |
|------|----------|--------|-----------|-------------|---------|-------------|--------------|
| Spring `@Scheduled` + DB 轮询 | 自行实现 | 自行实现 | 需加 ShedLock | 自行实现 | 自行实现 | 无 | 原生 |
| **Quartz（内嵌）** | **内置** | **内置（JDBC JobStore）** | **内置（DB 行锁）** | **内置 Misfire 策略** | 自行实现 | **无** | **原生支持** |
| XXL-JOB（内嵌） | 内置 | 内置 | 内置 | 内置 | 内置 | 无 | **不兼容**（javax） |

**选择 Quartz 的理由**：

1. Spring Boot 3.x 原生支持 `spring-boot-starter-quartz`，零额外基础设施
2. JDBC JobStore 将任务持久化到数据库，重启不丢失
3. 内置集群模式（DB 行锁协调），多实例部署时自动防重复执行
4. 内置 Misfire 策略（停机恢复后可自动补跑错过的任务）
5. 内嵌在 SuperSonic 进程中，部署形态不变（单个 `java -jar`）

**依赖引入**：

```xml
<!-- launchers/standalone/pom.xml -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-quartz</artifactId>
</dependency>
```

**配置**（`application.yaml`）：

```yaml
spring:
  quartz:
    job-store-type: jdbc                          # 持久化到数据库
    jdbc:
      initialize-schema: always                   # 自动建表（首次启动）
    properties:
      org.quartz.scheduler.instanceName: SuperSonicScheduler
      org.quartz.scheduler.instanceId: AUTO       # 集群中每个节点自动分配 ID
      org.quartz.jobStore.isClustered: true       # 启用集群模式
      org.quartz.jobStore.clusterCheckinInterval: 15000
      org.quartz.jobStore.misfireThreshold: 60000 # Misfire 阈值 60 秒
      org.quartz.threadPool.threadCount: 5        # 调度线程数
```

> Quartz JDBC JobStore 会自动创建 11 张 `QRTZ_*` 表。`initialize-schema: always` 在首次启动时自动建表，后续启动会跳过已存在的表。

```
├── execution_snapshot   → JSON：完整的 ReportExecutionContext 快照（用于历史复现）
├── template_version     → 执行时的模板版本号
├── engine_version       → 执行时的系统版本号
├── scan_rows            → 预估/实际扫描行数（来自 EXPLAIN 或执行统计）
├── execution_time_ms    → 查询执行耗时（毫秒）
├── io_bytes             → IO 读取字节数（仅 ClickHouse 等支持的数据源可填充，来自 system.query_log）

```

> `s2_report_schedule` 是业务配置，`QRTZ_*` 表是调度引擎状态。两者通过 `quartz_job_key` 关联。创建/修改/删除调度任务时，同步操作 Quartz 的 Job 和 Trigger。

上述 `execution_snapshot` + `template_version` + `scan_rows` + `execution_time_ms` 字段的设计目标：

- **历史复现**：通过 `execution_snapshot` 保存完整执行上下文，可在任意时间点回放某次执行的参数、权限和查询条件
- **成本分析**：通过 `scan_rows`、`execution_time_ms` 和 `io_bytes` 统计报表查询的资源消耗，可构建报表成本排行榜、识别高成本报表并优化
- **风险审计**：通过 `template_version` + `engine_version` 追踪执行时的系统状态，结合 `sql_hash` 审计查询变更历史

#### 3.8.3 核心实现

**设计原则**：Quartz Job 保持轻量，只负责分派；业务逻辑统一由 `ReportExecutionOrchestrator`（见 3.5）处理。

**Quartz Job 实现（轻量分派）**：

```java
/**
 * Quartz Job：轻量分派器。
 * 只负责设置租户上下文 + 调用 dispatcher，不包含任何业务逻辑。
 */
@Slf4j
public class ReportScheduleJob implements Job {

    @Override
    public void execute(JobExecutionContext context) throws JobExecutionException {
        Long scheduleId = context.getMergedJobDataMap().getLong("scheduleId");
        Long tenantId = context.getMergedJobDataMap().getLong("tenantId");

        TenantContext.setTenantId(tenantId);
        try {
            ReportScheduleDispatcher dispatcher =
                    ContextUtils.getBean(ReportScheduleDispatcher.class);
            dispatcher.dispatch(scheduleId);
        } finally {
            TenantContext.clear();
        }
    }
}
```

**分派器 + 重试机制**：

```java
/**
 * 调度分派器：从配置构建 Context，委托 Orchestrator 执行，处理重试。
 * 与 Quartz Job 解耦，可被 API / Agent 入口复用重试逻辑。
 */
@Service
@RequiredArgsConstructor
public class ReportScheduleDispatcher {

    private final ReportScheduleMapper scheduleMapper;
    private final ReportExecutionOrchestrator orchestrator;
    private final ReportExecutionContextBuilder contextBuilder;

    public void dispatch(Long scheduleId) {
        ReportSchedule schedule = scheduleMapper.selectById(scheduleId);
        executeWithRetry(schedule);
    }

    /**
     * 指数退避重试。retryInterval 为基数（秒），实际间隔 = base * 2^(attempt-1)。
     * 例如 retryInterval=30: 首次重试等 30s，第二次 60s，第三次 120s。
     */
    private void executeWithRetry(ReportSchedule schedule) {
        int maxAttempts = schedule.getRetryCount() + 1;
        for (int attempt = 1; attempt <= maxAttempts; attempt++) {
            try {
                ReportExecutionContext ctx = contextBuilder.buildFromSchedule(schedule, attempt);
                orchestrator.execute(ctx);  // 统一编排入口
                break;
            } catch (Exception e) {
                if (attempt < maxAttempts) {
                    long delay = schedule.getRetryInterval() * (1L << (attempt - 1));
                    log.warn("Schedule {} attempt {}/{} failed, retry in {}s",
                            schedule.getId(), attempt, maxAttempts, delay);
                    Thread.sleep(delay * 1000);
                } else {
                    log.error("Schedule {} all {} attempts exhausted", schedule.getId(), maxAttempts);
                }
            }
        }
    }
}
```

```java
public class ReportScheduleServiceImpl implements ReportScheduleService {

    private final Scheduler scheduler;              // Quartz Scheduler（Spring 自动注入）
    private final ReportScheduleMapper scheduleMapper;
    private final ReportExecutionMapper executionMapper;

    /** 创建调度任务：业务表 + Quartz Job/Trigger */
    public void createSchedule(ReportSchedule schedule) {
        // 1. 写入 s2_report_schedule
        scheduleMapper.insert(schedule);

        // 2. 创建 Quartz Job
        String jobKey = "report_" + schedule.getId();
        JobDetail job = JobBuilder.newJob(ReportScheduleJob.class)
                .withIdentity(jobKey, "REPORT")
                .usingJobData("scheduleId", schedule.getId())
                .usingJobData("tenantId", schedule.getTenantId())
                .storeDurably()
                .build();

        // 3. 创建 Cron Trigger
        CronTrigger trigger = TriggerBuilder.newTrigger()
                .withIdentity(jobKey, "REPORT")
                .withSchedule(CronScheduleBuilder
                        .cronSchedule(schedule.getCronExpression())
                        .withMisfireHandlingInstructionFireAndProceed()) // 错过后立即补执行一次
                .build();

        scheduler.scheduleJob(job, trigger);
        schedule.setQuartzJobKey("REPORT." + jobKey);
        scheduleMapper.updateById(schedule);
    }

    public void pauseSchedule(Long id) {
        ReportSchedule schedule = scheduleMapper.selectById(id);
        scheduler.pauseJob(JobKey.jobKey(schedule.getQuartzJobKey()));
        schedule.setEnabled(false);
        scheduleMapper.updateById(schedule);
    }
```

#### 3.8.5 与现有 @Scheduled 任务的共存

Quartz 仅用于**用户自定义的报表调度**。现有系统级 `@Scheduled` 任务（字典刷新、嵌入同步、部署超时清理等）保持不变，两者互不干扰：

| 调度类型 | 框架 | 管理方式 |
|---------|------|---------|
| 系统内部任务（固定周期） | Spring `@Scheduled` | 硬编码在代码中，运维无需管理 |
| 用户报表调度（动态 Cron） | Quartz Scheduler | 数据库持久化 + Web 管理界面 |

Spring Boot 的 `spring-boot-starter-quartz` 与 `@Scheduled` 使用独立的线程池，不会互相影响。

### 3.10 数据同步与查询隔离

#### 3.10.1 问题背景

报表类查询（全表聚合、大范围扫描）即使使用只读连接池和行数限制，仍会对业务库造成 CPU/IO/连接池争用。行数限制只控制**结果集大小**，不控制**扫描量**——`SELECT COUNT(*) FROM 10亿行表` 返回 1 行但扫描全表。

**目标**：报表查询完全不接触业务库，通过将数据同步到分析库（ClickHouse / DuckDB）实现查询隔离。

#### 3.10.2 技术选型

参考业界主流数据同步方案（DataX / Addax / SeaTunnel / Airbyte），采用**内置轻量同步**模式，复用现有基础设施：

| 业界方案 | 架构 | SuperSonic 对应能力 |
|---------|------|-------------------|
| DataX/Addax Reader Plugin | JDBC 读取源库 | `JdbcExecutor` + `DbAdaptor`（已支持 6 种 DB） |
| DataX/Addax Writer Plugin | JDBC 写入目标库 | 同上 |
| DataX Channel | 并发传输通道 | `ThreadPoolConfig`（已有多个线程池） |
| DataX/Addax JSON 任务配置 | 声明式同步规则 | `s2_data_sync_config`（JSON 存储） |
| SeaTunnel/Airbyte 调度 | 独立调度引擎 | Quartz Scheduler（已实现） |
| SeaTunnel/Airbyte 数据源管理 | 连接器配置 | `s2_database`（已有完整数据源管理） |
| SeaTunnel/Airbyte Web UI | 管理界面 | Ant Design Pro（已有前端框架） |

不引入外部同步工具，在现有架构上实现 DataX 模式的 **Framework + Plugin** 同步能力。

#### 3.10.3 增量同步模式

采用业界通用的**水位线增量同步**模式：

| 模式 | 原理 | 适用场景 | 能否检测删除 |
|------|------|---------|------------|
| **时间戳水位线** | `WHERE updated_at > last_sync_time` | 有 updated_at 字段的表（最常用） | 否 |
| **自增 ID 水位线** | `WHERE id > last_sync_id` | 只追加不更新的流水表（订单、日志） | 否 |
| **分区覆盖** | `DELETE WHERE dt = ? + INSERT` | 按日期分区的事实表 | 是（分区级） |
| **全量快照** | 每次 TRUNCATE + INSERT 全量 | 小表、维度表 | 是 |

> 如需检测硬删除，可定期（如每周）执行一次全量快照同步作为补偿。

#### 3.10.4 数据模型

```
s2_data_sync_config（同步配置 — 业务元数据）
├── id
├── name                     → 任务名称（如"订单表同步至 ClickHouse"）
├── source_database_id       → 源数据源（关联 s2_database，业务库）
├── target_database_id       → 目标数据源（关联 s2_database，分析库）
├── sync_config              → JSON：同步规则
│   ├── tables[]             → 同步表列表
│   │   ├── source_table     → 源表名
│   │   ├── target_table     → 目标表名（可不同）
│   │   ├── columns          → 字段映射（"*" 或显式列表）
│   │   ├── sync_mode        → FULL / INCREMENTAL / INCREMENTAL_DEDUP / PARTITION_OVERWRITE
```

#### 3.10.5 核心实现

**Quartz Job**（复用报表调度的 Quartz 引擎）：

```java
@Slf4j
public class DataSyncJob implements Job {

    @Override
    public void execute(JobExecutionContext context) throws JobExecutionException {
        Long configId = context.getMergedJobDataMap().getLong("syncConfigId");
        Long tenantId = context.getMergedJobDataMap().getLong("tenantId");

        TenantContext.setTenantId(tenantId);
        try {
            DataSyncService service = ContextUtils.getBean(DataSyncService.class);
            service.executeSync(configId);
        } finally {
            TenantContext.clear();
        }
    }
}
```

**同步执行核心逻辑**：

```java
public void executeSyncTable(TableSyncRule table, JdbcExecutor source, JdbcExecutor target,
                             DataSyncExecution execution) {
    // 1. 获取上次水位线
    String lastWatermark = getLastWatermark(table);

    // 2. 构造读取 SQL
    String readSql;
    switch (table.getSyncMode()) {
        case INCREMENTAL:
            readSql = String.format("SELECT %s FROM %s WHERE %s > '%s' ORDER BY %s",
                    table.getColumns(), table.getSourceTable(),
                    table.getWatermarkColumn(), lastWatermark,
                    table.getWatermarkColumn());
            break;
        case PARTITION_OVERWRITE:
            // 先在目标库删除对应分区
            target.execute(table.getPreSql());
            readSql = String.format("SELECT %s FROM %s WHERE %s",
                    table.getColumns(), table.getSourceTable(), table.getPartitionFilter());
            break;
        case FULL:
        default:
            target.execute("TRUNCATE TABLE " + table.getTargetTable());
            readSql = String.format("SELECT %s FROM %s", table.getColumns(), table.getSourceTable());
    }

    // 3. 流式读取 + 批量写入（防止 OOM）
    long rowsWritten = 0;
    try (ResultSet rs = source.executeQuery(readSql)) {
        List<Object[]> batch = new ArrayList<>(table.getBatchSize());
        while (rs.next()) {
            batch.add(extractRow(rs));
            if (batch.size() >= table.getBatchSize()) {
                target.batchInsert(table.getTargetTable(), batch);
                rowsWritten += batch.size();
                batch.clear();
            }
        }
        if (!batch.isEmpty()) {
            target.batchInsert(table.getTargetTable(), batch);
            rowsWritten += batch.size();
        }
    }

    // 4. 更新水位线
    execution.setRowsWritten(rowsWritten);
    execution.setWatermarkValue(getCurrentMaxWatermark(source, table));
}
```

#### 3.10.7 与报表调度的协作

数据同步和报表调度共享 Quartz 引擎，通过 Job Group + 独立线程池实现资源硬隔离：

```
Quartz Scheduler（统一调度引擎）
├── Group: REPORT     → ReportScheduleJob（报表定时执行）  threadCount=5
├── Group: DATA_SYNC  → DataSyncJob（数据同步任务）        threadCount=2
└── Group: ...        → 未来可扩展其他调度类型
```

**资源隔离配置**（`application.yaml`）：

```yaml
spring:
  quartz:
    properties:
      # 报表调度线程池（默认线程池）
      org.quartz.threadPool.threadCount: 5
      # 数据同步使用独立线程池（通过 Job 级别 @DisallowConcurrentExecution 控制）

# 应用层执行配额（Semaphore 限流）
supersonic:
  report:
    max-concurrent-per-tenant: 3          # 同一租户同时最多 3 个报表执行
  data-sync:
    max-concurrent-per-tenant: 2          # 同一租户同时最多 2 个同步任务
    max-full-sync-per-dataset: 1          # 同一 dataset 同时最多 1 个全量同步
```

**执行配额说明**：

| 配额项 | 限制 | 实现方式 | 目的 |
|--------|------|---------|------|
| 同一租户报表并发 | 最多 3 个 | `ConcurrentHashMap<tenantId, Semaphore>` | 防止单租户占满线程池 |
| 同一租户同步并发 | 最多 2 个 | 同上 | 避免同步任务互相阻塞 |
| 同一 dataset 全量同步 | 最多 1 个 | `@DisallowConcurrentExecution` + JobDataMap | 防止全量同步重复执行 |

> **关键点**：报表调度和数据同步使用不同线程池，数据同步慢不会饿死报表执行。配额通过应用层 Semaphore 控制（而非 Quartz 线程池），粒度更细。

**典型使用流程**：

```
1. 管理员在"数据库管理"中添加业务库（MySQL）和分析库（ClickHouse）连接
2. 在业务库连接的操作栏点击"配置同步"→ 向导 Step1 选分析库 → Step2 勾选表并配置同步模式 → Step3 设置 Cron
3. 在"语义模板"中创建模板，DataSet 指向分析库的 dwd_order
4. 在"报表调度"中创建定时报表：每天 9:00 查询分析库并推送
   → 报表查询走分析库，业务库零压力
```

> **关键交互优化**：数据同步配置入口在数据库管理页面，用户无需在两个独立页面间跳转。源数据库在上下文中已确定，通过向导引导完成目标库选择、Schema 发现、流级配置和调度设置。

#### 3.10.9 升级路径：Airbyte Connection 模型（Phase 3）

当前阶段（Phase 2）同步配置作为数据库连接的附属能力，满足基本的数据同步需求。后续计划升级为 **Airbyte Connection 模型**，将同步提升为独立的一等公民概念。

##### 当前模型 vs Connection 模型

| 维度 | 当前模型（Phase 2） | Connection 模型（Phase 3） |
|------|-------------------|--------------------------|
| 概念定位 | 数据库连接的子配置 | 独立实体（Connection = Source + Destination + Sync 规则） |
| 入口 | 数据库管理页的"配置同步"按钮 | 独立的 **Connections** 页面 + 侧边栏菜单项 |
| 生命周期 | 依附于数据库连接 | 独立 CRUD，Source/Destination 可被多个 Connection 共享 |
| 状态管理 | 简单 enabled/disabled | 完整状态机：Active / Paused / Broken / Deprecated |
| Schema 变更 | 手动重新配置 | 自动检测 Schema 变更并通知用户（Breaking / Non-breaking） |
| 可观测性 | 执行记录列表 | Connection Timeline（事件流：sync / schema change / error） |
| 状态恢复 | 无 | Checkpointing + State 管理（可手动重置同步位点） |

##### 迁移策略

Phase 2 → Phase 3 的升级路径：

1. **数据迁移**：`s2_data_sync_config` 记录自动转换为 `s2_connection` 记录
2. **Source/Destination 抽象**：`s2_database` 同时作为 Source 和 Destination（通过 role 字段区分），无需新建实体
3. **API 兼容**：Phase 2 的 `/api/v1/dataSyncConfigs` 标记为 Deprecated，保留 2 个版本周期
4. **前端迁移**：数据库管理页的"配置同步"入口保留（快捷方式），新增独立 Connections 页面
5. **执行记录**：`s2_data_sync_execution` 表结构兼容，增加 `connection_id` 字段

> **关键原则**：Phase 2 的实现应预留扩展点（如 `sync_config` JSON 结构对齐 ConfiguredCatalog），降低 Phase 3 升级的改造量。

**设计决策**：`s2_database` 同时可作为 Source 和 Destination（一个数据库可同时被读和被写），因此 `roles` 采用**多值标签**而非单值枚举。

**数据模型变更**：

```sql
-- V19 或 V20（与版本管理共用或独立迁移）
ALTER TABLE s2_database
  ADD COLUMN `roles` varchar(50) DEFAULT NULL
    COMMENT '角色标签: SOURCE,DESTINATION (逗号分隔，NULL表示纯查询用途)';
```

**虚拟 Source / Destination API**：

不创建新的 `s2_source` / `s2_destination` 表，通过视图 API 封装 `s2_database`：

```

GET    /api/v1/sources
POST   /api/v1/sources                        → 创建 s2_database + 设置 roles='SOURCE'
GET    /api/v1/sources/{sourceId}
PATCH  /api/v1/sources/{sourceId}
DELETE /api/v1/sources/{sourceId}
POST   /api/v1/sources/{sourceId}:discoverSchema   → 复用现有 schema discovery
POST   /api/v1/sources/{sourceId}:checkConnection  → 复用现有 testConnect

# Destination 视图 API（底层查询 s2_database WHERE roles LIKE '%DESTINATION%'）
GET    /api/v1/destinations
POST   /api/v1/destinations                   → 创建 s2_database + 设置 roles='DESTINATION'
GET    /api/v1/destinations/{destinationId}
PATCH  /api/v1/destinations/{destinationId}
DELETE /api/v1/destinations/{destinationId}
POST   /api/v1/destinations/{destinationId}:checkConnection
```

后端实现为 `SourceController` / `DestinationController`，内部委托 `DatabaseService`，仅增加 role 过滤与标记逻辑。

**Connection 创建时自动标记 role**：

```java

    addRole(sourceDb, "SOURCE");   // 逗号分隔追加
```

#### 3.10.8 查询资源隔离

即使没有独立分析库，也应在查询层面做资源隔离防护：

| 防护层 | 措施 | 实现方式 |
|--------|------|---------|
| **SQL 级超时** | 查询执行超过阈值自动终止 | MySQL: `SET max_execution_time = 30000`; ClickHouse: `max_execution_time` setting |
| **连接池隔离** | 报表查询使用独立连接池，不与业务请求争用 | 为 `s2_database` 增加 `pool_type` 字段（BUSINESS / REPORT），不同 pool 独立管理 |
| **并发限流** | 限制同时执行的报表查询数 | Quartz `threadPool.threadCount` 控制 + Semaphore 限流 |
| **扫描行数估算** | 执行前 EXPLAIN 估算扫描行数，超阈值拒绝执行 | `EXPLAIN SELECT ...` 解析 rows 估算值 |

#### 3.11.2 连接器配置模型（ConnectorConfig）

```java
@Data
public class ConnectorConfig {
    // ==================== 基础配置 ====================
    private String name;                    // 连接器名称
    private String description;             // 连接器描述
    private ConnectorType type;             // HTTP_REQUEST / GRAPHQL / WEBHOOK / INTERNAL

    // ==================== 请求配置 ====================
    private String url;                     // 请求 URL，支持占位符 {{param_name}}
    private String method = "POST";         // HTTP 方法：GET / POST / PUT / DELETE
    private String contentType;             // application/json / x-www-form-urlencoded
    private Map<String, String> headers;    // 自定义请求头
    private List<ParamOption> queryParams;  // URL 查询参数
    private List<ParamOption> bodyParams;   // 请求体参数
    private String bodyTemplate;            // 请求体模板（JSON 字符串，支持 {{placeholder}}）

    // ==================== 认证配置 ====================
    private AuthConfig auth;

    // ==================== 响应处理 ====================
    private ResponseConfig response;

    // ==================== 高级配置 ====================
    private Integer timeoutMs = 30000;      // 超时时间（毫秒）
    private Integer retryCount = 2;         // 重试次数
    private Integer retryDelayMs = 1000;    // 重试间隔（毫秒）
    private Boolean followRedirects = true; // 是否跟随重定向
    private Boolean verifySsl = true;       // SSL 证书验证
}
```

## 7. 设计总结

模板报表子系统通过标准化与治理化设计，将数据查询从"技术能力"升级为"平台级产品能力"，是智能运营数据中台规模化落地的关键支撑模块。

SuperSonic 工程已完成 Phase 1（基础能力）和 Phase 2（报表调度 + 查询隔离 + 数据同步 + 多渠道推送）的全部建设。

- Phase 2 的核心架构决策是引入 `ReportExecutionOrchestrator` 作为统一执行编排入口，Web / Agent / 定时调度三个入口均委托 Orchestrator 处理，配合不可变的 `ReportExecutionContext` 实现执行可追溯、可回放。在此基础上：

​	（1）通过 Quartz Scheduler（Spring Boot 原生 Starter，内嵌部署无额外基础设施）实现用户自定义报表调度，Quartz Job 仅做轻量分派，配套 Web 管理界面；

​	（2）通过异步导出机制（独立 Export Worker 线程池 + JDBC 流式读取 + EasyExcel 流式写入 + 任务中心）解决大数据量导出的 OOM 和线程饿死问题；

​	（3）通过 SQL 超时、连接池隔离（INTERACTIVE/REPORT/EXPORT/SYNC 四种池类型）和扫描行数估算（`RowCountEstimator` + Caffeine 缓存）实现查询资源隔离，报表调度与数据同步线程池硬隔离 + 租户级执行配额限流；

​	（4）借鉴 Airbyte 的 Configure sync 页面设计，以数据库管理页为入口，通过多步向导（选择目标库 → Schema 自动发现 + 流级粒度配置 → 调度设置）完成数据同步配置，支持四种同步模式（FULL/INCREMENTAL/INCREMENTAL_DEDUP/PARTITION_OVERWRITE）和数据库级 UPSERT 防重复，API 遵循 Google RESTful API 设计规范（资源导向 URL、标准方法 + 自定义方法 `:verb`、`pageToken` 分页、子资源模式）；

​	（5）通过 `ReportDeliveryChannel` 策略模式实现多渠道报表推送（EMAIL/WEBHOOK/FEISHU/DINGTALK/WECHAT_WORK），支持指数退避自动重试、渠道级速率限制、连续失败自动禁用，配套推送统计仪表盘和管理界面。

- Phase 3 已完成核心功能建设。Airbyte Connection 模型核心功能已落地（独立 Connections 管理页面、完整生命周期状态机、Schema 变更检测、Checkpointing、Connection Timeline、数据迁移脚本），Agent 自然语言创建定时报表已通过 `ReportScheduleExecutor` + REPORT_SCHEDULE Plugin 实现。

  - P0 和 P1 优先级的三项功能已全部实现：

    - （P0）影响分析与下线保护 — `DataSetServiceImpl.delete()` 新增级联校验（删除前检查活跃调度任务）+ 模板 Offline(2) 状态 + `SemanticTemplateServiceImpl.offlineTemplate()` 方法（下线前检查调度依赖）+ `POST /templates/{id}:offline` API 端点（见 3.9.2）；（P1）SQL 模板引擎 — 新建 `SqlTemplateConfig` POJO（`templateSql` + `variables`）+ `QueryConfig.sqlTemplateConfig` 扩展字段 + `SqlTemplateEngine` 组件（ST4 渲染 + `checkSensitiveSql` 安全校验）+ `ReportExecutionOrchestrator.parseQueryConfig()` 三路解析（SqlTemplateConfig → QueryStructReq → QuerySqlReq）（见 3.4.3）；

      （P1）模板版本管理 — `SemanticTemplateDO.currentVersion` 字段（每次编辑自增）+ `SemanticDeploymentDO.templateVersion`/`templateConfigSnapshot` 字段（部署时冻结）+ V19 迁移脚本（MySQL + PostgreSQL）（见 3.2.1）。

      ​	剩余（P2）Source/Destination 抽象（`s2_database.roles` 多值标签 + 虚拟视图 API + Connection 创建时自动标记）建议观察 Connection 模型实际使用情况后再决定实施时机（见 3.10.9）。所有新增模块均通过现有的 Spring Event、策略模式和 MyBatis-Plus 体系接入，无需架构层面的重大调整。
